# ADR-003: Idempotency-Key stored in dedicated table

## Status

Accepted

## Date

2026-04-24

## Context

Feature ADR-002 requires `POST /api/library/books` to accept an `Idempotency-Key` header and replay the cached response when the same key + body pair is seen within a 24-hour window. Three storage options exist:

- **Dedicated `idempotency_keys` table** in Postgres, cleaned up hourly by a pg-boss job.
- **Column on `books` row** — annotate the created row with the key; check for existing key on POST.
- **In-memory LRU cache** — keyed Map inside `apps/api` with 24-hour TTL.

Features 002-004 will add additional mutating POST endpoints (ingestion enqueue, agent thread create) that also benefit from idempotent semantics. The chosen storage must generalize without refactoring.

## Decision

Store Idempotency-Key state in a **dedicated `idempotency_keys` table**. Schema:

```
idempotency_keys
  key TEXT PRIMARY KEY
  request_hash TEXT NOT NULL      -- SHA-256 of canonical request body
  response_status INT NOT NULL
  response_body JSONB NOT NULL
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()

CREATE INDEX idempotency_keys_created_at_idx ON idempotency_keys(created_at);
```

**Middleware behavior** (Hono, applied to POST endpoints that declare idempotency):

1. No `Idempotency-Key` header → proceed normally.
2. `Idempotency-Key` present → compute `request_hash = sha256(canonical(body))`; SELECT row by `key`.
   - If row exists and `request_hash` matches → return cached `response_status` + `response_body`.
   - If row exists and `request_hash` differs → 422 Problem Details `idempotency-key-conflict`.
   - If row absent → execute handler; on success (2xx), INSERT `(key, request_hash, response_status, response_body)` before returning.

**Cleanup job** `catalog.cleanup-idempotency-keys` runs hourly via pg-boss `scheduleJob`:

```sql
DELETE FROM idempotency_keys WHERE created_at < now() - interval '24 hours';
```

**Applicability**: this ADR establishes the store. Feature 002's `POST /api/library/books/:id/ingest` and Feature 003's `POST /api/qa/threads` reuse the same middleware + table by opting in.

## Alternatives Considered

### Alternative 1: Column on `books` row

- **Description**: add `idempotency_key TEXT UNIQUE NULL` to the `books` table; POST checks for existing key on books.
- **Pros**: no new table; simplest schema change.
- **Cons**: couples idempotency to books exclusively; Feature 002+ would need to add the column to its own tables (ingestion jobs, threads), duplicating the pattern and introducing cross-table search for key matches.
- **Why rejected**: a cross-cutting concern belongs in a cross-cutting table.

### Alternative 2: In-memory LRU cache

- **Description**: `lru-cache` Map<key, {request_hash, response}> with 24-hour TTL, living in the `apps/api` process.
- **Pros**: zero DB cost; fastest middleware path.
- **Cons**: restart wipes the cache; same Idempotency-Key retried after restart produces a fresh 201 (breaking idempotency contract); in a future multi-instance deploy, each instance has its own cache (broken).
- **Why rejected**: violates the contract on the first `apps/api` restart during dogfooding.

## Consequences

### Positive

- Single table reusable by every idempotent POST across features 001-004.
- Survives `apps/api` restart — idempotency contract holds over restarts.
- `request_hash` stored alongside `key` allows the 422 conflict path without re-running the handler on mismatches.
- Cleanup is cheap (partial index on `created_at`) and automatic via pg-boss.

### Negative

- Every idempotent POST adds two DB round-trips (SELECT + INSERT). For single-user local scale, negligible.
- Cleanup job depends on pg-boss being healthy; a pg-boss outage lets the table grow. Mitigation: cleanup is idempotent, so catch-up on recovery is automatic.
- `response_body` stored as JSONB: binary response types (if future features add them) need explicit serialization.

### Risks

- **Risk**: `request_hash` canonicalization disagrees between client and server (key order in JSON, whitespace).
  **Mitigation**: canonicalization function `canonicalizeBody(body)` is defined once in `@dialogus/shared/http` and used on both sides if a client SDK is ever built. For dogfooding, curl-style clients hash the exact body we store.
- **Risk**: cleanup job lag during heavy-usage spikes.
  **Mitigation**: partial index on `created_at` makes DELETE fast; job runs hourly but is harmless to invoke more often manually if table grows.

## Implementation Notes

- Middleware file: `apps/api/src/infrastructure/http/middleware/idempotency.ts`.
- The middleware is opt-in per route; registered via `app.post('/api/library/books', idempotency(), handler)`.
- Hash format: `sha256(JSON.stringify(sortedKeysCanonicalForm(body)))`, stored as hex.
- When hash matches, response headers include `X-Idempotency-Replay: true` to help debugging (not strictly required by spec).
