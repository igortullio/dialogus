# ADR-001: Two-namespace API shape (catalog + library)

## Status

Accepted

## Date

2026-04-24

## Context

Catalog is the first dIAlogus feature that exposes domain-level HTTP endpoints. Its API shape sets the convention that Features 002-004 inherit. Three URL-organization approaches were considered:

- **Two namespaces** (`/api/catalog/*` for Gutendex gateway, `/api/library/*` for local CRUD) — cleanly separates external-source operations from local-state operations.
- **Single namespace** (`/api/books/*` with `?scope=remote|local` query parameter) — fewer URLs, more coupling between very different operations.
- **Three namespaces** (`/api/gutendex/*` as explicit adapter + `/api/library/*`) — more granular, explicit naming of the external dependency in URLs.

The product TechSpec already enumerated endpoints under `/api/catalog/*` and `/api/library/*` without settling the rationale. This ADR makes that decision explicit so Features 002-004 inherit it predictably.

## Decision

Use **two namespaces**:

- **`/api/catalog/*`** — Gutendex gateway. Reads from an external source with a 60s LRU cache layer. Currently: `GET /api/catalog/search`, `GET /api/catalog/books/:gutendex_id`. Feature 002+ may extend with other remote operations.
- **`/api/library/*`** — local CRUD over the personal acervo. Currently: `POST /api/library/books`, `GET /api/library/books`, `GET /api/library/books/:id`, `DELETE /api/library/books/:id`, `POST /api/library/books/:id/restore`. Feature 002 extends with `POST /api/library/books/:id/ingest`, `GET /api/library/books/:id/ingestion`, `POST /api/library/books/:id/ingest/retry`, `GET /api/library/chunks/:id`.

Tests, documentation, and future routes reinforce this split.

## Alternatives Considered

### Alternative 1: Single namespace `/api/books/*` with scope parameter

- **Description**: `GET /api/books?scope=remote&q=...` searches Gutendex; `GET /api/books?scope=local` lists library; `POST /api/books` promotes from remote to local.
- **Pros**: Fewer top-level URLs; surface looks simpler at first read.
- **Cons**: Conflates very different semantics (external cache-bounded read vs. local transactional CRUD). Errors, caching, and idempotency behave differently per scope, forcing route handlers into `if (scope === 'remote') ...` branches that erode clarity. Reviewers reading the API surface can't tell at a glance which ops hit external infrastructure and which hit the DB.
- **Why rejected**: scope-as-parameter hides exactly the distinction that matters most for reasoning about failure modes.

### Alternative 2: Three namespaces (`/api/gutendex/*` + `/api/library/*`)

- **Description**: Name Gutendex explicitly in the URL; treat it as a named adapter. `/api/gutendex/search` maps 1:1 to Gutendex operations; `/api/library/*` stays for local CRUD.
- **Pros**: Maximum transparency — a reviewer sees "gutendex" and knows this hits an external service; if we add another source later (e.g., Archive.org), it gets its own namespace.
- **Cons**: Encodes an implementation detail (which external source) into URLs that clients (the web app, future API consumers) then depend on. Changing the source requires rewriting URLs. "Catalog" is a product concept; "Gutendex" is an implementation choice.
- **Why rejected**: couples URL shape to the source of truth; the product PRD already forecloses alternative sources for V1 and "catalog" is the stable abstraction.

## Consequences

### Positive

- Clear reading: `/api/catalog/*` touches external cache; `/api/library/*` touches the DB.
- Feature 002-004 can predictably extend: ingestion operations go under `/api/library/*` because they operate on local entities; the agent endpoints (Feature 003) go under `/api/qa/*` (a third product namespace).
- Error handling per namespace can differ where appropriate (catalog endpoints may return `503` for Gutendex outage; library endpoints will not).

### Negative

- Two namespaces to version and document separately (moderate).
- Clients must distinguish "search" from "list library" at the URL level even when they display them in a unified view (handled client-side in Feature 004).

### Risks

- **Risk**: future route for "search within library" is ambiguous — library or catalog?
  **Mitigation**: local search stays under `/api/library/*` since it operates on local entities; the `catalog/` namespace is reserved for external gateway reads.

## Implementation Notes

- Hono route modules mirror the namespaces: `apps/api/src/infrastructure/http/routes/catalog.ts` and `apps/api/src/infrastructure/http/routes/library.ts`.
- Use-case modules in `apps/api/src/application/catalog/` similarly split: `search-gutendex.ts` lives alongside catalog; `add-book-to-library.ts`, `list-library.ts`, etc. live as library concerns but still within `application/catalog/` (single bounded context — the package is `@dialogus/catalog`).
- OpenAPI spec (Phase 2) uses `tags: [catalog]` and `tags: [library]` to group endpoints.
