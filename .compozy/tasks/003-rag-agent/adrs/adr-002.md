# ADR-002: Agent responds in the user's message language (auto-detect per turn)

## Status

Accepted

## Date

2026-04-24

## Context

The product PRD commits UI strings and user-facing copy to Portuguese. Source code, logs, and technical documentation stay in English. The agent itself — the prose text the LLM produces in reply to a question — is not addressed by any product-level artifact.

The owner reads classics in both English and Portuguese and will frequently mix languages: asking in EN about *Dom Casmurro*, asking in PT about *Moby Dick*. A single fixed output language creates friction in either direction.

## Decision

**The agent responds in the language of the user's most recent message.** Detection happens per turn; the agent's output language can shift mid-thread if the user does. Book quotes retrieved by `semantic_search` and excerpts attached to citations remain in the book's original language regardless of the response language.

- The system prompt includes an explicit instruction: "Respond in the same language as the user's latest message. When quoting book passages, preserve the original language of the source."
- Detection is implicit to the LLM (no explicit language-detection tool or preprocessing step). Claude 4.5/4.6 handles EN/PT disambiguation natively; a simple prompt instruction is sufficient.
- Short or ambiguous messages (one-word follow-ups, "ok", "continue") inherit the language of the preceding agent turn, which is already correct by construction.

## Alternatives Considered

### Alternative: Always respond in Portuguese

- **Description**: The agent replies in pt-BR regardless of the user's input language. Quotes remain in book language.
- **Pros**: matches the product's Brazilian user framing; one consistent voice; simpler system prompt; no detection edge cases.
- **Cons**: forces EN-speaking users (future portfolio reviewers, secondary persona) into PT-only output; awkward when asking complex questions in EN about an EN book and receiving PT commentary about it.
- **Why rejected**: the owner's own usage is bilingual; a PT-only agent trains the owner to artificially phrase questions in PT even when thinking in EN.

### Alternative: Match the primary book in thread scope

- **Description**: If the thread is scoped to EN books, respond in EN; PT books, respond in PT; mixed, default to PT.
- **Pros**: ties language to content context; somewhat intuitive.
- **Cons**: unnatural when the user is thinking in one language and the book happens to be in another; mixed threads need an arbitrary tie-breaker; spoilt by edge cases.
- **Why rejected**: overspecified; no clear user benefit over simpler rules.

### Alternative: Always respond in English

- **Description**: EN-only agent voice.
- **Pros**: simplest prompt; consistent engineering demo.
- **Cons**: contradicts the product's UI-in-PT positioning; PT users must read mixed EN/PT screens.
- **Why rejected**: direct conflict with product positioning.

## Consequences

### Positive

- Bilingual fluency — the owner types in whichever language they are currently thinking in.
- Implicit onboarding: a reviewer testing the product in English gets an English response without configuration.
- No language-switcher UI control required.

### Negative

- System prompt gains a language-matching instruction; small prompt-complexity tax.
- Edge cases: extremely short messages ("qual?", "why?") can be ambiguous; mitigated by inheriting the prior turn's language.
- Mixed-language citations: an EN-speaking user asking about *Dom Casmurro* will see PT excerpts inline with EN prose. This is correct behavior (quotes stay in source language) but may look odd at first glance.

### Risks

- **Risk**: Haiku or Sonnet misdetects language on very short messages and flips mid-thread.
  **Mitigation**: the system prompt's instruction is deterministic; modern Claude models handle EN↔PT with very high accuracy on messages ≥ 5 tokens. The dogfooding validation round explicitly includes one "short follow-up" journey.
- **Risk**: users of future Phase 2 public-deploy expect a language toggle and complain when it is absent.
  **Mitigation**: V1 is local-first; Phase 2 can add a toggle if telemetry shows demand.

## Implementation Notes

- Single instruction in the system prompt, Markdown asset in `@dialogus/rag`, caching-eligible via Anthropic prompt caching.
- No language-detection library or preprocessing middleware — defer everything to the model.
- The refusal-with-reformulation UX (ADR-003) inherits the same language-match rule: the refusal message appears in the user's message language.
