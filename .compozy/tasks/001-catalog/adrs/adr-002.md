# ADR-002: Portfolio-grade 2026 API contract

## Status

Accepted

## Date

2026-04-24

## Context

Catalog is the first dIAlogus API surface that a portfolio reviewer reads in detail. Three levels of contract polish were on the table:

- **Portfolio-grade 2026**: cursor pagination, RFC 9457 Problem Details, envelope `{ data, meta, links }`, Idempotency-Key header on mutating POST, soft-delete via `DELETE 204` plus `POST /:id/restore`.
- **MVP speed**: offset pagination, plain JSON errors, arrays returned directly, no idempotency plumbing, hard-delete.
- **Hybrid**: some portfolio signals (Problem Details + envelope + soft-delete) but keep offset pagination and skip idempotency.

The product PRD names "portfolio reviewer" as a secondary persona; the catalog API is the first artifact that persona encounters. Market research on 2026 senior-engineering API conventions consistently flags cursor pagination, Problem Details, envelope shape, and idempotency as table-stakes signals; offset pagination over a mutable catalog and ad-hoc JSON errors are commonly cited as junior tells.

## Decision

Adopt **portfolio-grade 2026 conventions** across every catalog endpoint. Specifics:

- **Cursor pagination**: list endpoints return `{ data: [...], meta: { count?, limit }, links: { next?: "...?cursor=..." , prev?: "..." , self: "..." } }`. Cursors are opaque base64-encoded `{ created_at, id }` tuples (library) or pass-through Gutendex page numbers (catalog search). Clients treat cursors as opaque strings.
- **RFC 9457 Problem Details**: every error response has `Content-Type: application/problem+json` with `{ type, title, status, detail, instance }`. Validation errors include an `errors` extension (array of field-level issues). `type` URIs use a stable `urn:dialogus:problems:<slug>` namespace documented in README.
- **Envelope shape**: `{ data, meta, links }` for list responses. Single-resource responses use `{ data }` (meta and links omitted). POST success returns `{ data }`. DELETE returns empty 204. Consistency is mandatory; partial envelope usage is forbidden.
- **Idempotency-Key header**: `POST /api/library/books` accepts an optional `Idempotency-Key` header. Same key + same request body within a 24-hour window returns the cached response (same status, same body). Different bodies with same key produce 422 Problem Details. Keys are stored in a dedicated `idempotency_keys` table (TechSpec details).
- **Soft-delete + restore**: `DELETE /api/library/books/:id` returns 204 and sets `deleted_at = now()`. `POST /api/library/books/:id/restore` returns 200 with envelope and clears `deleted_at`. Hard-delete is not exposed in V1.

Cross-cutting: this contract is mandatory for Features 002-004 unless the feature PRD explicitly overrides.

## Alternatives Considered

### Alternative 1: MVP speed — offset pagination, plain JSON errors, hard-delete

- **Description**: `?page=N&limit=N` offset pagination; errors as `{ error: 'Not found', code: 'NOT_FOUND' }`; `DELETE` hard-removes the row; no idempotency plumbing; no envelope.
- **Pros**: Simplest possible implementation; fastest dogfooding path.
- **Cons**: Offset pagination over a mutable library shifts rows when new books are added; plain JSON errors can't convey validation issues cleanly; hard-delete loses the metadata snapshot that future features (002-004) depend on for referential integrity (chunks pointing at a deleted book would become orphan); no envelope means clients parse different shapes per endpoint.
- **Why rejected**: every "nice" in the MVP column becomes a "not-nice" in the durability column within two features.

### Alternative 2: Hybrid (Problem Details + envelope + soft-delete, keep offset + skip idempotency)

- **Description**: Adopt the visible polish items (error format, envelope, soft-delete) but skip the less-visible items (cursor pagination, idempotency).
- **Pros**: Saves ~1 day of implementation relative to full portfolio-grade.
- **Cons**: Introduces inconsistency — Feature 002 or 003 may adopt cursor pagination for ingestion status lists, creating a mid-project convention shift. Idempotency-Key is specifically called out in research as a 2026 portfolio signal; skipping it while adopting Problem Details sends a mixed message.
- **Why rejected**: the cost of consistency is small; the cost of inconsistency compounds across four features.

## Consequences

### Positive

- Single consistent contract across all current and future endpoints — reviewers see a coherent story.
- Cursor pagination prevents the "added book shifts page 2 results" bug that shows up in any dogfooding session longer than a week.
- Problem Details makes errors self-describing and localizable per client (future Portuguese validation messages can be structured).
- Soft-delete with restore keeps referential integrity simple: `chunks.book_id` foreign key remains valid even when the book is "gone" from the user's view.
- Idempotency-Key makes API retries safe — especially valuable if/when CLI tools, scripts, or webhooks eventually call the API.

### Negative

- Higher upfront implementation cost (~2 days over MVP-speed): cursor encoding/decoding, Problem Details middleware, envelope wrapper utilities, Idempotency-Key middleware + storage table.
- Additional test surface — every contract item (cursor edge cases, Problem Details shape, idempotency key conflicts) adds test cases.
- Clients must be taught to treat cursors as opaque; a client that parses and mutates them breaks on a future encoding change.

### Risks

- **Risk**: Idempotency-Key storage grows unbounded if no retention policy.
  **Mitigation**: cleanup job deletes keys older than 24h; implemented as a pg-boss job registered during Feature 002 or Foundation-level scheduled-job infra. For V1, a simple `DELETE WHERE created_at < now() - interval '24 hours'` on each insert suffices.
- **Risk**: cursor tokens leak internal sort keys and an observer reverse-engineers them.
  **Mitigation**: cursors are base64 JSON, not encrypted. For single-user local V1 this is fine; Phase 3 (public deploy) revisits (add HMAC signing).
- **Risk**: future features break the envelope convention by accident.
  **Mitigation**: a shared response helper (`withEnvelope`, `problemDetails`) lives in `@dialogus/shared/http` and PR review enforces usage.

## Implementation Notes

- Shared utilities live in `@dialogus/shared/http`: `envelope()`, `problemDetails()`, `encodeCursor()`, `decodeCursor()`. Added as part of Feature 001 TechSpec, reused by every later feature.
- Hono middleware for Idempotency-Key stores `(key, request_hash, response)` in `idempotency_keys` table; responds from cache on key-match-with-same-hash, 422 on key-match-with-different-hash.
- Error `type` URIs use format `urn:dialogus:problems:<slug>` — example slugs: `duplicate-gutendex-id`, `validation-failed`, `gutendex-upstream-error`. README "API Problems" section enumerates all slugs.
