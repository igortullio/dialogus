# ADR-003: Resume from failed stage via SHA-256 checkpoint

## Status

Accepted

## Date

2026-04-24

## Context

Any ingestion stage can fail: Gutendex 503 during download, OpenAI 429 during embed, malformed EPUB during parse. `POST /api/library/books/:id/ingest/retry` must react to such failures. Three semantics were considered:

- **Resume from failed stage** — re-run only the failed stage (and downstream); preserve outputs of successful upstream stages.
- **Restart fresh** — re-run from stage 1 (download), overwriting previous outputs.
- **Per-failure user choice** — API accepts `mode=resume|fresh`.

Key constraints:

- Download stage is rate-limited and constrained by Gutenberg politeness; re-hitting Gutenberg is costly.
- Embed stage incurs OpenAI API charges; re-embedding already-embedded chunks burns money.
- Product PRD explicitly mandates "pipeline is resumable".
- `books.raw_hash` (SHA-256) is already reserved in the schema for download-stage idempotency.

## Decision

Retry semantics are **resume from failed stage**:

- `POST /api/library/books/:id/ingest/retry` re-enqueues ONLY the stage recorded as failed in `books.ingestion_status`.
- Successful upstream artifacts are reused:
  - Download → raw file at `./storage/raw/<gutendex_id>.<ext>` verified against `books.raw_hash` (SHA-256); matching hash skips re-download.
  - Clean → cleaned-text file at `./storage/clean/<gutendex_id>.txt` (overwritten fresh per run; idempotent).
  - Parse → existing `chapters` rows for the book are kept; stage checks for presence before re-parsing.
  - Chunk → existing `chunks` rows (with `embedding = NULL`) are kept; re-chunk detects presence and skips.
  - Embed → chunks where `embedding IS NOT NULL` are skipped; only missing embeddings are generated.
  - Index → always re-runs (cheap; ensures fresh HNSW state).
- If the failed stage is the LAST one that ran, retry runs THAT stage plus all downstream.
- If the owner manually calls `retry` on a book in `ready` state, API returns 409 Problem Details `book-already-ready`.
- If the owner calls `retry` on a book in `discovered` state (never ingested), API returns 409 `book-not-in-retryable-state` — use `/ingest` instead.

## Alternatives Considered

### Alternative 1: Restart fresh

- **Description**: retry re-runs from stage 1; deletes prior artifacts (raw file, cleaned text, chapters, chunks, embeddings) and redoes everything.
- **Pros**: single, uniform retry semantics; no per-stage "already done?" check logic.
- **Cons**: violates Gutenberg politeness on every retry; re-bills OpenAI for embeddings that already succeeded; extends retry wall-clock from seconds to minutes.
- **Why rejected**: Gutenberg politeness is a hard constraint (ADR-002 handles it operationally); embedding cost, while small per book, scales with retry frequency in unpredictable ways.

### Alternative 2: Per-failure user choice

- **Description**: retry endpoint accepts `{ mode: 'resume' | 'fresh' }`.
- **Pros**: flexible.
- **Cons**: the owner rarely has the context to make this choice correctly. "Fresh" is the right answer only in weird edge cases (e.g., suspected corruption of the raw file). Those cases are better served by `db:reset` or manual `rm ./storage/raw/...`.
- **Why rejected**: unhelpful flexibility.

## Consequences

### Positive

- Retries are fast: typical embed-stage failure retry finishes in under 2 minutes (vs. 5+ for fresh).
- Gutenberg is hit at most once per book, regardless of retry count.
- OpenAI costs are bounded: each chunk embeds exactly once per successful run.
- SHA-256 checkpoint provides integrity guarantee — if raw file was corrupted on disk, hash mismatch triggers re-download automatically.
- `ingestion_status = 'failed'` on a book acts as a resumable state marker; the retry endpoint doesn't need a separate "last successful stage" column.

### Negative

- Per-stage handlers include "is upstream output already present?" logic — slight code complexity.
- If retry is called after a partial embed run (say, 60 % of chunks embedded before crash), the 40 % remaining run MUST have a query plan that targets only `embedding IS NULL`; a naive "embed all chunks" would double-bill.
- Developer must be disciplined: every stage handler either produces its output idempotently or checks for prior success before starting.

### Risks

- **Risk**: a partial output from a crashed stage is corrupt but not detected by the upstream-check.
  **Mitigation**: each stage checkpoint has a defined invariant (chunk rows have `embedding IS NOT NULL` on success; chapter rows have `plain_text` NOT NULL on success; raw file has matching SHA-256). A violation of any invariant triggers re-running that stage.
- **Risk**: the owner runs `retry` many times on a permanently broken book and never realizes it.
  **Mitigation**: each retry logs the stage + failure reason; a book that fails the same stage 3 times consecutively surfaces a `book-permanently-failed` state (Phase 2; V1 just keeps `failed` status).

## Implementation Notes

- `/ingest/retry` handler logic:
  1. Read `books.ingestion_status`. If `ready` → 409. If `discovered` → 409.
  2. Otherwise, enqueue `ingestion.<current_status>` for the book.
- Each stage handler's "already done?" check:
  - Download: stat `./storage/raw/<gutendex_id>.<ext>`; if present and hash matches stored `raw_hash`, skip to enqueue-next.
  - Clean: stat `./storage/clean/<gutendex_id>.txt`; if present, skip.
  - Parse: `SELECT count(*) FROM chapters WHERE book_id = $1`; if > 0, skip.
  - Chunk: `SELECT count(*) FROM chunks WHERE book_id = $1`; if > 0, skip.
  - Embed: operates on `chunks WHERE book_id = $1 AND embedding IS NULL`; if none remain, skip.
  - Index: always re-runs (cheap).
- Deletion via `db:reset` removes `./storage/raw/*` too (operators should expect this).
