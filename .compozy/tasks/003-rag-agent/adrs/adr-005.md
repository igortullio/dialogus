# ADR-005: Chapter summaries live in a dedicated `chapter_summaries` table (1:1 with chapters)

## Status

Accepted

## Date

2026-04-24

## Context

ADR-001 moved chapter-summary generation into Feature 002 (Ingestion). The shape of the storage was left to the 003 TechSpec. Two clean options exist: (A) a new `chapter_summaries` table with a foreign key + unique constraint on `chapter_id`; (B) adding columns (`summary`, `summary_token_count`, `summary_generated_at`, `summary_model`) directly on the existing `chapters` table.

The choice affects both 002's new ingestion stage and 003's `get_chapter_summary` tool â€” but the decision belongs to 003 because 003 is the consumer and owns the repository port shape.

## Decision

**Create a dedicated `chapter_summaries` table with a 1:1 relationship to `chapters`.**

Drizzle table shape:

```typescript
export const chapterSummaries = pgTable('chapter_summaries', {
  id: uuid('id').primaryKey().defaultFn(uuidV4),
  chapterId: uuid('chapter_id')
    .notNull()
    .unique()
    .references(() => chapters.id, { onDelete: 'cascade' }),
  bookId: uuid('book_id')
    .notNull()
    .references(() => books.id, { onDelete: 'cascade' }),
  summary: text('summary').notNull(),
  tokenCount: integer('token_count').notNull(),
  model: text('model').notNull(),
  generatedAt: timestamp('generated_at', { withTimezone: true })
    .notNull()
    .defaultNow(),
})
```

- `chapter_id` is unique â†’ enforces 1:1.
- `book_id` is denormalized to enable fast book-scoped queries (summary listings for a whole book) without joining through `chapters`.
- `on delete cascade` on both FKs keeps the table clean when a book or chapter is removed.
- `model` records the LLM that produced the summary (e.g., `claude-haiku-4-5-2026-04`) so future re-summarization can tell old from new.

Owning repository: `DrizzleChapterSummaryRepository` lives in `@dialogus/ingestion/src/infrastructure/persistence/` (per ADR-006 of this feature, `@dialogus/rag` consumes via dep).

## Alternatives Considered

### Alternative B: Add summary columns to `chapters`

- **Description**: `chapters` gains `summary text`, `summary_token_count int`, `summary_generated_at timestamptz`, `summary_model text`.
- **Pros**: no join at read time; one SQL migration; one repository (`DrizzleChapterRepository`) handles both reads.
- **Cons**: conflates two lifecycle artifacts in one row. `plain_text` is produced by the parse stage and immutable; `summary` is produced later by a new stage and may be regenerated (e.g., prompt tweaks). Regenerating a summary now rewrites a row in the hottest table in the system. Index bloat from updates on a large-column-heavy row is a known Postgres pain.
- **Why rejected**: mixing immutable and regenerable data on the same row invites pain later; the 1:1 join in the dedicated-table shape is nearly free (FK-indexed) and the separation is cheap insurance.

### Alternative C: Single `summaries jsonb` column on `chapters`

- **Description**: one jsonb column holding `{ summary, model, token_count, generated_at, version }`.
- **Pros**: flexible for future versioning (multiple summaries per chapter).
- **Cons**: jsonb for a single current-value object is over-engineered; loses type safety; harder to query/filter in SQL; Drizzle jsonb typing is less ergonomic than native columns.
- **Why rejected**: solves a problem we don't have yet (multi-version summaries) at the cost of ergonomics we'd feel every day.

## Consequences

### Positive

- Clean lifecycle separation: `chapters.plain_text` is parse-immutable; `chapter_summaries.summary` is regenerable.
- Index on `chapter_id` is the only access path the agent needs â€” `findByChapterId` is a single-row lookup by unique FK.
- Future multi-version summaries (Phase 2) add a `version` column + drop the unique constraint without touching `chapters`.
- Summary regeneration is an `UPSERT chapter_summaries` â€” no impact on `chapters` row churn.

### Negative

- One extra join when a query needs chapter body + summary together (rare in the agent's hot path; `get_chapter_summary` returns only the summary).
- One extra table to migrate and test.
- Dual delete cascade (book + chapter) means care around test fixtures; `onDelete: cascade` on `chapter_id` is sufficient by itself, but `book_id` cascade is defensive for the edge case of direct book removal.

### Risks

- **Risk**: summary regeneration during dogfooding requires re-running Feature 002's new `summarize` stage against a ready book; without a dedicated command, the only path is re-ingestion.
  **Mitigation**: Feature 002 amendment adds a `pnpm books:resummarize <book_id>` CLI script (or a POST endpoint) that re-runs only the summarize stage against existing chapters. Detail deferred to Feature 002 amendment TechSpec.
- **Risk**: if the summarize stage fails mid-book, some chapters have summaries and others don't; the agent's `get_chapter_summary` tool can hit a missing row.
  **Mitigation**: `ingestion_status = 'ready'` is only set after the summarize stage completes for all chapters. Partial failures keep the book in `summarizing` or `failed`, never `ready`.

## Implementation Notes

- Migration `drizzle/0004_chapter_summaries.sql` (or appended to Feature 002's migration if the feature ships together) is generated by `drizzle-kit` â€” no hand-editing needed.
- Feature 002 amendment appends a `summarize` pipeline stage between `chunk` and `embed` (or alongside `index`); stage writes to `chapter_summaries`.
- Tool `get_chapter_summary` signature returns `{ summary, chapter_id, chapter_ordinal, chapter_title, book_id }` â€” chapter_title is joined from `chapters` at read time inside the repository.
