# ADR-009: Library page uses RSC for initial fetch + TanStack Query on the client

## Status

Accepted

## Date

2026-04-24

## Context

`/library` displays the user's book collection with covers, statuses, ingestion progress, and inline mutations (add, retry, remove). Next 16 App Router supports four data-fetching shapes: (1) RSC initial render + TanStack Query client hydration; (2) full client-side fetching; (3) full RSC with Server Actions for mutations; (4) streaming RSC with `<Suspense>` boundaries.

The page has both static-feeling content (book metadata, cover URLs — rarely changes) and dynamic content (ingestion progress — polls every 2 seconds for `downloading` / `parsing` / etc. books). It also has multiple mutations the user triggers (add, retry, soft-delete).

## Decision

**The `/library` page is a Server Component that fetches the initial book list, then hydrates `<LibraryGrid>` (a client component) via TanStack Query.**

- `apps/web/src/app/library/page.tsx` — Server Component. Calls `fetch('/api/library/books')` server-side; passes initial data via `<HydrationBoundary>` from `@tanstack/react-query` to the client tree.
- `apps/web/src/app/library/LibraryGrid.tsx` — Client Component (`'use client'`). Uses `useQuery({ queryKey: ['library'], ... })` with the server-prefetched data as `initialData`. Mutations (add, retry, soft-delete) use `useMutation` with optimistic updates and `invalidateQueries(['library'])` on settle.
- Polling for in-progress ingestions: a per-book `useQuery({ queryKey: ['ingestion', bookId], refetchInterval: 2000 })` enabled only when `book.ingestion_status` is one of `downloading | parsing | chunking | summarizing | embedding`. Stops polling automatically when status flips to `ready` or `failed`.
- The "Adicionar do Gutendex" drawer (per ADR-010) opens on top of `<LibraryGrid>`; its own search query lives in another `useQuery` keyed on the search input.

## Alternatives Considered

### Alternative 2: Full client-side with TanStack Query

- **Description**: `/library/page.tsx` is a Client Component; first render shows skeleton; TanStack Query fetches on mount.
- **Pros**: simplest mental model; no hydration boundary.
- **Cons**: TTFB shows skeleton (loading flash); doesn't take advantage of Next 16 RSC streaming; bundle includes TanStack Query on initial JS.
- **Why rejected**: the loading flash is visible on every navigation to `/library` — a real UX cost on a page the owner visits multiple times per session.

### Alternative 3: Full RSC with Server Actions for mutations

- **Description**: All data via Server Components; mutations via Server Actions; `revalidatePath('/library')` after each mutation; polling via `setInterval` + `router.refresh()`.
- **Pros**: minimal client JS; "fully server-rendered" purity.
- **Cons**: polling for ingestion progress via `router.refresh()` re-renders the whole route tree every 2 seconds — heavy and feels janky; mutations can't show optimistic UI without splitting into client islands anyway; loading skeletons require explicit `<Suspense>` boundaries on top of an already-streaming response.
- **Why rejected**: progress polling is the dominant interaction pattern on this page; full-RSC handles it poorly.

### Alternative 4: Streaming RSC with `<Suspense>` per section

- **Description**: Granular `<Suspense>` around each section (book list, search box, etc.); each streams independently.
- **Pros**: progressive rendering for slow data; modern Next 16 idiom.
- **Cons**: book list and search box are not slow enough to need streaming separately; granular suspense adds boilerplate without measurable benefit; doesn't help with polling.
- **Why rejected**: solving for a perf problem we don't have.

## Consequences

### Positive

- Fast TTFB on `/library` — initial render has data; no skeleton flash.
- Mutations are optimistic; UI feels instant.
- Polling is bounded to in-progress books only; idle library pages don't poll.
- The hydration pattern is the most common Next 16 + TanStack Query pattern in 2026; well-documented and easy to debug.
- Future SEO (Phase 3 public deploy) gets server-rendered initial content for free.

### Negative

- Hydration boundary requires careful prop typing (server-fetched data must serialize cleanly through React's RSC payload).
- Two data-flow patterns coexist: server-fetched initial + client-fetched mutations/polling; debuggers see queries from both directions.
- A `dehydrate` / `hydrate` step on every render of the page; small overhead but real.

### Risks

- **Risk**: server-fetched data and client refetch race — the initial RSC data is stale by milliseconds when the client hydrates.
  **Mitigation**: TanStack Query's `staleTime: 0` + `initialData` correctly handles this; the client refetch is the source of truth after hydration. No flicker because both renders show the same data.
- **Risk**: polling continues after a user navigates away if not properly cleaned up.
  **Mitigation**: `useQuery` lifecycle handles unmount cleanup automatically; the `enabled` flag based on book status auto-disables polling for completed books.
- **Risk**: Server Component cannot pass non-serializable data (e.g., functions, Date objects).
  **Mitigation**: types in the API envelope use plain primitives; ISO date strings; no functions. Already enforced by Zod schemas in `@dialogus/shared`.

## Implementation Notes

- `apps/web/src/app/library/page.tsx` (RSC):
  - Calls `getQueryClient()` (a server-side helper); calls `queryClient.prefetchQuery(['library'], fetchLibrary)`; passes `dehydrate(queryClient)` to `<HydrationBoundary>`.
- `apps/web/src/app/library/LibraryGrid.tsx` (Client):
  - `useQuery({ queryKey: ['library'], queryFn: fetchLibrary })` reads from hydrated cache.
  - `<BookCard>` per book; per-card `useQuery({ queryKey: ['ingestion', book.id], enabled: isInProgress(book.ingestion_status), refetchInterval: 2000 })` for live progress.
- `apps/web/src/lib/api/library.ts` — fetcher + mutation functions (typed against `@dialogus/shared/schemas/library`).
- `apps/web/src/lib/query-client.tsx` — `QueryClientProvider` wrapping the entire app; default `staleTime: 30_000` for non-polling queries.
