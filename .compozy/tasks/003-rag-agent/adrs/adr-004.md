# ADR-004: No reranking in V1; pure HNSW semantic retrieval

## Status

Accepted

## Date

2026-04-24

## Context

Retrieval-augmented generation pipelines commonly pair a dense-vector first pass with a second-stage reranker — a cross-encoder (BGE, Cohere Rerank, FlashRank), an LLM-as-judge pass, or an algorithmic reordering (MMR for diversity). Reranking typically pays off on large, noisy corpora where the embedding model's top-k is approximately-but-not-quite right.

The Feature 002 TechSpec commits to HNSW cosine similarity over `text-embedding-3-small` (1536d) with paragraph-aligned 768-token chunks and 10–15% overlap. A V1 library of 5–10 books produces roughly 5k–20k chunks. Queries are scoped to a thread's 1–3 selected books, so the actual search space per query is 500–6000 chunks — two to three orders of magnitude smaller than the typical reranker-justifying corpus.

Market research (2025–2026) consistently reports: on corpora under ~100k chunks with high-quality modern embeddings, reranking rarely moves user-perceived quality enough to justify added latency and cost.

## Decision

**Feature 003 V1 ships with pure HNSW semantic retrieval. No reranker, no LLM-as-judge pass, no MMR. Top-k is returned to the agent as-is, sorted by cosine similarity.**

- `semantic_search` tool returns the top-k chunks (k configurable, default resolved in 003 TechSpec) with scores, metadata, and excerpts.
- The agent treats the returned chunks as authoritative; it does not re-evaluate or reorder them before citing.
- Reranking is explicitly deferred to Phase 2, conditional on dogfooding evidence of precision problems.

## Alternatives Considered

### Alternative: Cohere Rerank or `@mastra/rerank` as a second stage

- **Description**: `semantic_search` pulls top-30; a reranker (Cohere's managed API or Mastra's built-in utility) trims to top-5 before returning to the agent.
- **Pros**: industry-default pattern; tangible precision gains on noisy retrieval; `@mastra/rerank` is idiomatic inside the Mastra ecosystem.
- **Cons**: 200–500 ms latency per tool call (noticeable in the 3-second first-token target); per-query cost (Cohere is paid, Mastra is self-hosted but still an LLM call); adds an external dependency or extra model surface; small-corpus gains are marginal per market research.
- **Why rejected**: cost and latency are certain; quality gain is speculative on this corpus size.

### Alternative: LLM-as-reranker with Haiku

- **Description**: `semantic_search` returns top-20; Haiku 4.5 scores each candidate 0–10 for relevance to the query; the tool returns the top-5 sorted by score.
- **Pros**: cheapest LLM-based reranking option; single-vendor (Anthropic only); one more prompt-cache opportunity.
- **Cons**: adds ~1–2 seconds to every retrieval call; doubles tool-call token count; quality improvement unproven on small corpora; forces every thread to make one extra Haiku call per turn even when retrieval was already good.
- **Why rejected**: same cost/latency concern as Cohere, compounded by the fact that the same Haiku is also the dev-tier agent model — diminishes observability by adding a second LLM step inside retrieval.

### Alternative: MMR diversity reordering (no extra API call)

- **Description**: Take top-30 from HNSW, reorder with Maximal Marginal Relevance to reduce redundancy. Algorithmic; no extra LLM or API.
- **Pros**: cheap (pure math); addresses duplicate-paragraph problems in multi-book threads; one-line dependency addition.
- **Cons**: solves a different problem (redundancy, not relevance); may surface less-relevant-but-more-diverse chunks first; rarely necessary with paragraph-aligned chunking that already limits intra-book duplication.
- **Why rejected**: not solving a demonstrated problem; defer until dogfooding surfaces redundancy complaints.

## Consequences

### Positive

- Simplest possible retrieval path — one HNSW query, one tool return.
- Lower latency — no second-stage hop; helps hit the 3-second first-token target.
- Zero additional cost per query beyond the agent's main LLM turn.
- Smaller tool surface to test; clearer attribution when the agent cites poorly ("the top-k is the top-k; the embedding model's output is the failure").

### Negative

- On queries where the embedding model's top-1 is off by one or two positions, the agent may cite a less-relevant chunk first. Minor UX issue, not a grounding failure.
- If dogfooding reveals systematic top-k imprecision, Phase 2 must add reranking — a non-trivial code + test change.
- A public-deploy Phase 2 with heavier, noisier corpora (per-user libraries in the hundreds) likely cannot skip reranking.

### Risks

- **Risk**: multi-book threads mix books with differing query affinities; HNSW may over-represent the book closest to the query language/style, starving the others.
  **Mitigation**: `semantic_search` tool takes a per-book quota (e.g., top-k per book) — a TechSpec-level design choice; avoids one book dominating top-k. Decision deferred to 003 TechSpec.
- **Risk**: Phase 2 reranker integration is delayed by the tool signature being rigid.
  **Mitigation**: return `score` and `chunk_id` alongside every chunk so a future reranker stage can slot in behind the tool without changing the agent's contract.

## Implementation Notes

- `semantic_search` tool signature (shape frozen now, internals open):
  ```typescript
  {
    query: string
    book_ids: string[]
    spoiler_caps?: Record<string, number>  // book_id -> max chapter ordinal
    k?: number                              // default from 003 TechSpec
  } -> {
    chunks: Array<{
      chunk_id, book_id, chapter_id, chapter_ordinal,
      chapter_title, text, score, excerpt_preview
    }>
  }
  ```
- `score` is kept in the output even in V1 so the agent (or future reranker) can use it.
- The HNSW query uses cosine distance with the params set by Feature 002 (`m=16, ef_construction=64`). Runtime `ef_search` is a TechSpec decision.
- A Phase 2 reranker slots in as a post-processing step inside the tool implementation; the tool's external contract (inputs + outputs) does not change.
