# ADR-002: Spoiler cap persists in browser localStorage only

## Status

Accepted

## Date

2026-04-24

## Context

The spoiler boundary is dIAlogus's signature UX innovation (product PRD Core Feature #4). The agent enforces the cap server-side per request (Feature 003 ADR-006: per-request tool param), but the user's chosen cap value must persist somewhere on the client side so that re-opening a thread restores the cap they set. The product TechSpec listed three options: (A) localStorage only; (B) localStorage + optional `thread_book_preferences` table sync; (C) backend-only persistence; or (D) per-book global cap (not per-thread).

V1 is single-user and local-first per product TechSpec. The owner runs dIAlogus on their own machine, primarily on a single browser. Cross-device sync has no V1 user; offline degradation has no V1 cost.

## Decision

**Spoiler cap persists in browser localStorage only.** No backend table, no cross-device sync, no Mastra Memory write.

- Storage key shape: `dialogus:spoiler_cap:<thread_id>:<book_id>` → integer chapter ordinal (or `null`/missing for "no cap").
- On thread open, the UI hydrates the spoiler slider from localStorage entries matching `dialogus:spoiler_cap:<thread_id>:*`.
- On every message send, the UI reads the current state for each book in scope and includes it in the request body's `spoiler_caps: Record<book_id, max_chapter_ordinal>` field.
- On thread delete, the UI removes all `dialogus:spoiler_cap:<thread_id>:*` entries (cleanup).
- If the user opens dIAlogus in a different browser or device, caps reset to "no cap" for all threads. This is a known and accepted V1 limitation.

## Alternatives Considered

### Alternative B: localStorage + optional `thread_book_preferences` table

- **Description**: localStorage is the source of truth on the client. A backend table mirrors per-thread-per-book caps; on thread open, the UI fetches `GET /api/library/threads/:id/preferences`, merges with localStorage (server wins on conflict), and persists changes via PUT.
- **Pros**: cross-device sync as a bonus; survives clearing browser storage; future Phase 2 audit / analytics path.
- **Cons**: requires a new table in `@dialogus/db`, two new API endpoints (`GET` + `PUT`), conflict-resolution logic, and the cognitive load of "is this in localStorage or DB?" on every change. All this for a feature with no V1 user.
- **Why rejected**: V1 is single-user single-device; the cost is concrete and the benefit is hypothetical.

### Alternative C: Backend-only persistence (no localStorage)

- **Description**: Every cap change writes to the DB synchronously. UI reads from DB on every thread open.
- **Pros**: simplest mental model (one source of truth).
- **Cons**: requires the backend to be reachable for every cap change; offline (rare, but possible) breaks the feature; UI must handle network failure on a UX-critical input; same backend cost as B without any of B's benefits.
- **Why rejected**: localStorage is correct for client-owned UI state; backend-only loses offline tolerance for no V1 gain.

### Alternative D: Per-book global cap (no `thread_id` in the key)

- **Description**: One cap per book, shared across all threads referencing that book. Mental model: "current reading progress in book X."
- **Pros**: arguably more natural — a reader's progress in a book is a property of the book, not of a chat session.
- **Cons**: rules out the legitimate flow "let me discuss chapters 1-5 in one thread and 10-15 in another" — common dogfooding pattern when the owner re-reads a section. Loses flexibility.
- **Why rejected**: per-thread granularity is more useful for reflective re-reading; the global flow is an edge case Phase 2 can revisit if dogfooding reveals the need.

## Consequences

### Positive

- Zero backend code for spoiler cap state.
- localStorage operations are synchronous and effectively instant.
- The `spoiler_caps` request body shape (Feature 003 ADR-006) maps one-to-one to localStorage state — no conversion layer.
- Browser DevTools' Application → Local Storage view becomes a debugging surface for free.
- Future Phase 2 sync-to-DB upgrade is additive: localStorage stays, a new sync hook fires on change.

### Negative

- Cap resets on different browsers/devices/profiles. Documented in the README's "Known V1 limitations" section.
- localStorage clearing (rare but possible — e.g., browser cleanup, dev profile reset) wipes caps. The user reapplies them; not data loss, just UX friction.
- The UI must handle the "stale localStorage entry for a deleted thread" case — cleanup on thread delete is documented in Implementation Notes.
- The `spoiler_caps` request body always reflects "what the UI thinks is current" — there's no server-side check that a cap was actually set; the agent trusts whatever arrives.

### Risks

- **Risk**: localStorage quota exhaustion on a power user with hundreds of threads.
  **Mitigation**: each cap entry is < 100 bytes; 1000 threads × 5 books = ~500 KB, well under the typical 5-10 MB quota. Document a Phase 2 "thread archive" feature if the limit becomes real.
- **Risk**: localStorage available in dev (Chrome/Safari) but Server Components on Next 16 cannot read it during render.
  **Mitigation**: spoiler-cap state is read client-side only via a `'use client'` component; SSR renders the slider in its default state and hydrates from localStorage on mount. Brief flash is acceptable.
- **Risk**: race condition where the user changes the cap mid-message-send.
  **Mitigation**: read localStorage at the moment `useChat.sendMessage` fires; ignore changes mid-flight. The next message picks up the new value.

## Implementation Notes

- A small wrapper hook `useSpoilerCap(threadId, bookId)` encapsulates: hydrate, set, delete-on-thread-delete cleanup, and a hook return of `{ cap, setCap, isLoaded }`.
- The hook must NOT block rendering — `isLoaded` is `false` until `useEffect` reads localStorage.
- Cleanup on thread delete: enumerate localStorage keys with the thread_id prefix and remove. Easy via `Object.keys(localStorage).filter(...)`.
- Phase 2 sync path: a separate hook `useThreadBookPreferencesSync` (Phase 2 task) listens to localStorage changes and PUTs to a new backend endpoint; localStorage stays the source of truth on read; backend wins on conflict only when localStorage is empty for that key.
