# ADR-003: pg-boss init folded into `db:migrate`

## Status

Accepted

## Date

2026-04-24

## Context

pg-boss is the job queue for dIAlogus (product TechSpec). Its `start()` function is idempotent and handles its own schema migration across pg-boss versions — creating the `pgboss` schema, tables, indexes, and functions on first run, and evolving them on subsequent runs.

Three patterns were considered for where/when `pgboss.start()` runs in the dIAlogus lifecycle:

- **In `db:migrate` script**: after Drizzle migrations complete, the same script instantiates pg-boss and calls `start()` + `stop()`.
- **Separate `db:init-pgboss` script**: a dedicated script that developers run alongside `db:migrate`.
- **At app startup**: `apps/api` (or a dedicated init module) calls `pgboss.start()` on boot; idempotent, so safe to re-run.

Foundation does not yet have a worker process or any code that `send()`s jobs, but it must leave the database in a state where Feature 002 can start enqueuing without a schema migration step of its own.

## Decision

**pg-boss init is folded into the `db:migrate` script.** `pnpm db:migrate` executes in two stages inside `packages/db/src/migrate.ts`:

1. Apply pending Drizzle SQL migrations (`drizzle-kit migrate` or equivalent programmatic call).
2. Instantiate pg-boss with the same `DATABASE_URL`, call `await boss.start()`, then `await boss.stop()`.

The migration script logs an explicit stage marker between the two phases so a partial failure is diagnosable. Applications (`apps/api`, future `apps/worker`) never call `pgboss.start()` themselves — they only consume the already-migrated schema via `send()` / `work()`.

## Alternatives Considered

### Alternative 1: Separate `db:init-pgboss` script

- **Description**: a dedicated root script that runs `pgboss.start()` + `stop()` in isolation; developers run both `pnpm db:migrate` and `pnpm db:init-pgboss` during setup.
- **Pros**: explicit separation of concerns; failures are isolable.
- **Cons**: two commands in the README quickstart instead of one; easy to forget the pg-boss step; onboarding friction.
- **Why rejected**: the README quickstart goal is 5 lines; every extra command counts against that budget.

### Alternative 2: `pgboss.start()` at app startup

- **Description**: `apps/api` calls `pgboss.start()` on boot. Future `apps/worker` would do the same.
- **Pros**: no separate migration ceremony; the app "just works".
- **Cons**: couples boot time to schema migration (pg-boss can evolve its schema on version upgrades, adding 1-2 seconds to startup); future multi-process startup (api + worker) creates a race where two processes call `start()` simultaneously — pg-boss handles this but the observability is worse; violates the principle that schema migration is a separate ops step from runtime.
- **Why rejected**: Feature 002 will introduce `apps/worker`, creating exactly the multi-process race problem this alternative can't avoid. The fold-into-migrate pattern is cleaner forever.

## Consequences

### Positive

- Single ceremony for all schema setup: `pnpm db:migrate` handles Drizzle + pg-boss together.
- Applications are simpler — no startup init code, no race conditions between api and worker.
- Observable stage transitions in migration logs ("Drizzle done, starting pg-boss...") pinpoint partial failures.
- Works identically in dev (after `docker compose up -d`) and in CI (against a fresh Testcontainers DB in Feature 002).

### Negative

- `pnpm db:migrate` does two things — slightly violates single-responsibility, but within acceptable bounds for a convenience script.
- pg-boss version upgrades still require a fresh `pnpm db:migrate` run; not auto-handled by CI unless the CI workflow explicitly runs migrations.

### Risks

- **Risk**: `pgboss.start()` fails mid-migration after Drizzle has succeeded, leaving the DB partially migrated.
  **Mitigation**: `runMigrations()` uses try/catch; on pg-boss failure, logs the stage and the original Drizzle success so the developer can retry pg-boss alone (via a future fallback `db:init-pgboss:retry` script, if this proves common).
- **Risk**: pg-boss 12 + Postgres 18 have an unvalidated combination.
  **Mitigation**: Foundation smoke step #4 (`pnpm db:reset && pnpm db:migrate`) explicitly exercises the combo before closing Foundation. If it fails, fall back to PG 17 (per product TechSpec Known Risks) or pin pg-boss 11.

## Implementation Notes

- `packages/db/src/migrate.ts` is the single entry point:
  ```typescript
  import PgBoss from 'pg-boss'
  import { migrate } from 'drizzle-orm/postgres-js/migrator'

  export async function runMigrations(databaseUrl: string): Promise<void> {
    logger.info({ stage: 'drizzle' }, 'applying Drizzle migrations')
    await migrate(createDatabase(databaseUrl), { migrationsFolder: './drizzle' })
    logger.info({ stage: 'pgboss' }, 'starting pg-boss')
    const boss = new PgBoss(databaseUrl)
    await boss.start()
    await boss.stop()
    logger.info({ stage: 'done' }, 'migrations complete')
  }
  ```
- Root `db:migrate` script invokes this programmatically via `tsx packages/db/src/migrate.ts` with `DATABASE_URL` from `@dialogus/shared/config`.
- `db:reset` drops the database (`DROP DATABASE` + `CREATE DATABASE`) and calls `db:migrate`.
