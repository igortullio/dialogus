# ADR-006: `@dialogus/rag` depends on `@dialogus/ingestion` and reuses its Drizzle repositories

## Status

Accepted

## Date

2026-04-24

## Context

The RAG agent's four tools need read access to three tables owned by Feature 002: `chapters`, `chunks`, and (via ADR-005 of this feature) `chapter_summaries`. Feature 002 already placed `DrizzleChapterRepository` and `DrizzleChunkRepository` inside `@dialogus/ingestion/src/infrastructure/persistence/` with their respective ports (`ChapterRepository`, `ChunkRepository`) in `@dialogus/ingestion/src/domain/`.

Feature 003 needs to consume these repositories. Four architectural shapes were considered:

1. Have `@dialogus/rag` depend on `@dialogus/ingestion` and reuse the adapters directly.
2. Move canonical repositories to `@dialogus/db`, so both ingestion and rag depend on it.
3. Declare read-only ports in `@dialogus/rag` and implement adapters inline in `apps/mastra`, accepting duplication.
4. Introduce a new `@dialogus/data-access` package.

## Decision

**`@dialogus/rag` declares `package.json` dependency on `@dialogus/ingestion` and imports `DrizzleChapterRepository`, `DrizzleChunkRepository`, and `DrizzleChapterSummaryRepository` (added in the 002 amendment) directly.**

- No Drizzle adapter duplication.
- No retroactive migration of code between packages.
- `apps/mastra` wires the repositories in `mastra.config.ts` by importing both from `@dialogus/ingestion` and passing them to `createDialogusAgent({ chunkRepo, chapterRepo, chapterSummaryRepo, ... })`.
- `@dialogus/rag` still declares its own domain-level port interfaces (`ChunkReadRepository`, `ChapterReadRepository`, `ChapterSummaryReadRepository`) that the agent tools consume — these are narrower read-only interfaces, and the `@dialogus/ingestion` adapters satisfy them structurally (TypeScript structural typing makes this zero-effort).

## Alternatives Considered

### Alternative 2: Move repositories to `@dialogus/db`

- **Description**: Relocate `DrizzleChapterRepository` / `DrizzleChunkRepository` / `DrizzleChapterSummaryRepository` to `@dialogus/db/src/persistence/`. Both `@dialogus/ingestion` and `@dialogus/rag` import from `@dialogus/db`.
- **Pros**: correct dep direction (data-access in the data package); no cross-feature-package coupling; future packages (analytics in Phase 2) have a natural home.
- **Cons**: retroactive edit of the already-spec-complete 002 TechSpec + tasks; pushes application concerns (write-side invariants, mapper conventions) into a package previously kept intentionally thin (`@dialogus/db` was client + schemas + migrations only); break a design norm for a dep-direction gain that is theoretical in V1.
- **Why rejected**: the change cost is concrete (edit 002), the gain is abstract (cleaner UML graph). Single-user V1, local-first, no public reuse of these packages.

### Alternative 3: Read-only ports in `@dialogus/rag`, adapters inline in `apps/mastra`

- **Description**: `@dialogus/rag` declares ports; `apps/mastra/src/persistence/` implements them with Drizzle. Ingestion keeps its own adapters.
- **Pros**: strongest isolation; `@dialogus/rag` becomes a pure domain/orchestration package with zero database imports.
- **Cons**: two Drizzle adapters implement the same queries against the same tables — one in `@dialogus/ingestion` (write-side), one in `apps/mastra` (read-side). Drift is inevitable; bugs get fixed in one and not the other.
- **Why rejected**: adapter duplication is a concrete, daily cost in exchange for an architectural purity the product doesn't need.

### Alternative 4: New `@dialogus/data-access` package

- **Description**: All Drizzle adapters live in a new dedicated package.
- **Pros**: cleanest architectural layering.
- **Cons**: introduces a 6th package for V1; retroactive migration of 002's adapters; burden on mental model ("where does the Drizzle code live again?") for a single-person project.
- **Why rejected**: over-engineered; 5 packages is already the right count for this size of project.

## Consequences

### Positive

- Zero duplication of Drizzle adapters or mapper code.
- No retroactive change to 002's package layout (002 already lists `DrizzleChunkRepository` + `DrizzleChapterRepository` in its `@dialogus/ingestion/src/infrastructure/persistence/`).
- `@dialogus/rag` stays thin: ports + tools + agent factory + system prompt asset; no persistence code.
- Read-only port interfaces in `@dialogus/rag` protect against accidental write-side coupling (the agent package cannot `save()` or `delete()` chunks because its port interfaces don't declare those methods).

### Negative

- Package dep graph acquires an edge from `@dialogus/rag` → `@dialogus/ingestion`. Inspection of `pnpm why` reveals RAG pulls transitively from ingestion.
- Conceptual imbalance: the "RAG" package depends on an "ingestion" package; most readers would expect the opposite. Mitigated by README's architecture section.
- If Phase 2 introduces a reranker or a cross-book analytics feature inside `@dialogus/rag`, it inherits everything `@dialogus/ingestion` bundles (including `@gxl/epub-parser`, `bottleneck`, etc.) as transitive deps. Tree-shaking mitigates runtime cost; dev install time increases marginally.

### Risks

- **Risk**: `@dialogus/ingestion` evolves its write-side types in a way that breaks read-side consumers.
  **Mitigation**: the narrow read-only ports in `@dialogus/rag` are the contract. Any change to the Drizzle adapters that breaks them surfaces as a TypeScript error at build time. Adding fields is backward-compatible (structural typing); removing or renaming fields requires coordinated PR (documented in the amendment).
- **Risk**: future decision to open-source or publish `@dialogus/rag` independently is complicated by the ingestion dep.
  **Mitigation**: not a V1 concern; if it becomes one, refactor to Alternative 2 (move to `@dialogus/db`) at that time.

## Implementation Notes

- `@dialogus/rag/package.json` gains `"@dialogus/ingestion": "workspace:*"` in `dependencies`.
- `@dialogus/rag/src/domain/ports/ChunkReadRepository.ts`:
  ```typescript
  export interface ChunkReadRepository {
    searchSemantic(params: {
      bookIds: string[]
      queryEmbedding: number[]
      spoilerCaps?: Record<string, number>
      k: number
    }): Promise<ChunkWithContext[]>
    findById(id: string): Promise<ChunkWithContext | null>
    findCharacterMentions(params: {
      bookIds: string[]
      aliases: string[]
    }): Promise<ChunkWithContext[]>
  }
  ```
- Analogous ports for `ChapterReadRepository` and `ChapterSummaryReadRepository`.
- `apps/mastra/mastra.config.ts` imports the three concrete `Drizzle*Repository` classes from `@dialogus/ingestion` and passes them to `createDialogusAgent()`. TypeScript confirms the structural fit.
