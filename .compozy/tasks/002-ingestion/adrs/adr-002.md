# ADR-002: Serial ingestion (one book at a time)

## Status

Accepted

## Date

2026-04-24

## Context

Three concurrency models were considered for the ingestion worker:

- **Serial** — one book processes end-to-end before the next starts.
- **Parallel-limit 2** — up to 2 books process simultaneously across stages.
- **Configurable via env var** — default 1, tunable at deploy.

Project constraints:

- Single-user local dogfooding at ~1-3 books per week typical cadence.
- Project Gutenberg robot policy explicitly asks for serialized, polite downloads.
- OpenAI Tier-1 embedding limits (500 RPM / 1M TPM) are generous but shared across calls.
- Worker complexity has operational cost (schedulers, priority queues, race conditions in shared storage paths).

## Decision

Ingestion is **serial**. pg-boss worker configuration applies `teamConcurrency = 1` (or equivalent) on every `ingestion.*` queue. When multiple books are queued, book N+1 does not begin stage 1 until book N has reached `ready` or `failed`.

The API does not expose a concurrency knob in V1. Adding Phase 2 parallelism will be an explicit upgrade with its own ADR.

## Alternatives Considered

### Alternative 1: Parallel-limit 2

- **Description**: pg-boss `teamConcurrency = 2` on heavy stages (parse / chunk / embed). Download stage stays at concurrency 1 to respect Gutenberg.
- **Pros**: 5 books ingested in ~10 minutes vs. ~20 minutes serial.
- **Cons**: doubles worker operational complexity (shared `./storage/raw/` path has race risks on concurrent writes of different books — solvable, but adds surface); debug sessions with interleaved logs are harder; failure in one book can starve the other via contention on the DB or the embedding provider.
- **Why rejected**: single-user volume (~1-3 books/week) doesn't justify the complexity. A 20-minute walk-away batch is acceptable.

### Alternative 2: Configurable concurrency (env var)

- **Description**: `INGESTION_CONCURRENCY` env var (default 1, max 5). Operators tune per environment.
- **Pros**: flexibility.
- **Cons**: expands config surface for a feature with only one operator (the project owner); "should I raise it?" is a question the owner shouldn't have to ask.
- **Why rejected**: premature flexibility; adding the env var in Phase 2 if the pattern changes is cheap.

## Consequences

### Positive

- Predictable UX: the 5th queued book will not begin until the 4th completes.
- Gutenberg robot policy is satisfied by construction — the download stage can only fire one request at a time.
- Simpler debugging: logs are sequentially interpretable per book.
- pg-boss worker configuration stays trivial.

### Negative

- Adding 10 books at once means ~40+ minutes of queued processing (one of the 10 at a time).
- Worker CPU is underutilized during embed stage's I/O-bound OpenAI waits.

### Risks

- **Risk**: the owner perceives ingestion as "slow" after queuing many books.
  **Mitigation**: `/ingestion` endpoint on each book returns a queue position (derived from pg-boss job state); README documents that multiple-book ingestion is best done overnight.
- **Risk**: worker crashes lose the current book's in-progress stage; next worker boot resumes but no parallelism to fall back on.
  **Mitigation**: ADR-003's resume-from-failed-stage semantics recover gracefully; the next book waits slightly longer but completes.

## Implementation Notes

- pg-boss worker configuration in `apps/worker`: `boss.work('ingestion.download', { teamConcurrency: 1 }, handler)` and similarly for each queue.
- No global mutex or reservation table needed — pg-boss FIFO queue ordering + concurrency=1 suffices.
- If Phase 2 introduces parallelism, the download stage MUST stay at concurrency=1 regardless of other stages' concurrency.
