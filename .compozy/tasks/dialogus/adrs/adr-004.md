# ADR-004: Spoiler boundary as the V1 differentiator

## Status

Accepted

## Date

2026-04-24

## Context

Market research identified four realistic differentiators a solo-portfolio RAG-over-classics product could credibly adopt:

1. UUID-bound citation UX (hover, click-through, side panel).
2. Multi-translation side-by-side view.
3. Spoiler boundary (user sets current chapter; retrieval filters above it).
4. Structure-aware chunking (verse, dialogue, drama-specific handling).

Citation UX is mandatory given the scholarly posture (see ADR-002). The question is which of the remaining three to include in V1 versus defer. A single, well-executed differentiator is more valuable than three half-done ones.

## Decision

**Spoiler boundary** is the V1 differentiator beyond the mandatory citation UX:

- User sets "current chapter" per book within a thread.
- Retrieval filters chunks to `chapter_ordinal <= user_progress` for that book.
- The agent's system prompt explicitly reinforces: "do not reveal any plot, theme, or event from chapters beyond the cap set by the user".
- One-click disable or adjust in the thread header; visible active-cap indicator.

Multi-translation mode and structure-aware chunking are deferred to Phase 2.

## Alternatives Considered

### Alternative 1: Multi-translation mode

- **Description**: Group Gutendex IDs of the same work (Don Quixote, Tolstoy, Dostoevsky all have multiple editions); show passages side-by-side from different translations; agent commentary on translation differences.
- **Pros**: Strongest unique-portfolio pitch — hard to dismiss as "another RAG demo"; leverages Gutendex's breadth.
- **Cons**: Requires a "same work, multiple editions" data-model extension in the catalog; UX is much more sophisticated (side-by-side layouts, aligned paragraphs); chapter alignment across translations is itself a research problem.
- **Why rejected**: ROI per implementation cost is lower than spoiler boundary; high risk of scope blow-up.

### Alternative 2: Structure-aware chunking

- **Description**: Chunker respects verse (keeps stanzas intact), dialogue (keeps speaker with line in drama), paragraph boundaries (never splits mid-paragraph).
- **Pros**: Pitch-worthy engineering story for portfolio (visible-in-code, explainable in interview).
- **Cons**: Invisible in the UX (no screenshot tells this story); doubles ingestion-pipeline complexity; late-bound regressions hard to detect.
- **Why rejected**: portfolio value without UX value is thin; deferred to Phase 2 when evals can measure it.

### Alternative 3: No extra differentiator (only polished citation UX)

- **Description**: V1 ships with grounded citations, hover-preview, side-panel excerpts, and nothing else novel.
- **Pros**: Shortest MVP; maximum focus on citation UX polish.
- **Cons**: Weaker differentiation — "grounded RAG with citations" is widely claimed, rarely delivered well, but not visibly unique on first impression.
- **Why rejected**: spoiler boundary is cheap enough to include and visibly differentiates the product.

## Consequences

### Positive

- Visible product-thinking signal in the UX (a slider, a status indicator, a measurable behavior).
- Low implementation cost — a filter clause on retrieval, a prompt clause, and a UI slider.
- Aligns with the scholarly posture: readers want to study a book without spoilers.
- Measurable success criterion (≥ 95 % respect in owner-posed questions).

### Negative

- Adds a per-thread-per-book state field that has to be persisted and surfaced consistently.
- Chapter detection in the ingestion pipeline must be correct for the spoiler cap to work — shifts pressure onto chapter heuristics.

### Risks

- **Risk**: chapter detection fails for some books (unusual Gutenberg TXT formats), breaking the spoiler cap silently.
  **Mitigation**: ingestion pipeline validates chapter count against a sanity threshold; books below threshold get flagged; catalog shows a warning on such books.
- **Risk**: user forgets to update the cap after reading; agent silently under-answers.
  **Mitigation**: cap is visible in the thread header; nudge-on-question if the thread has been inactive and the user asks about "later" material.

## Implementation Notes

- Chapter detection heuristics in the ingestion pipeline must cover EN + PT patterns (per PRD language scope).
- Retrieval filter is part of the RAG-agent feature spec.
- UI affordance (slider + header indicator) is part of the Chat UI feature spec.
- Multi-translation and structure-aware chunking Phase 2 items will reference this ADR as the "V1 deferral" decision.
