# ADR-001: Six-stage pipeline as chained pg-boss jobs

## Status

Accepted

## Date

2026-04-24

## Context

The ingestion pipeline transforms a book from `discovered` to `ready` through six discrete activities: download the raw file, clean Project Gutenberg boilerplate, parse chapters, chunk text, generate embeddings, and index the vectors. Two shapes were considered:

- **Chained pg-boss jobs** — each stage is its own pg-boss queue (`ingestion.download`, `ingestion.clean`, ..., `ingestion.index`). Stage N completes by enqueuing stage N+1 with a compact payload.
- **Single monolithic job per book** — one pg-boss job runs all six stages inline in its handler; progress is reported via `job.updateProgress(0-100)`.

Feature 002 decisions already require per-stage progress visibility (Q4), resume-from-failed-stage retry (ADR-003), and serial processing (ADR-002). The shape of the pipeline has to support those without workarounds.

## Decision

The ingestion pipeline is **six chained pg-boss jobs**, one per stage:

1. `ingestion.download` → enqueues `ingestion.clean` on success.
2. `ingestion.clean` → enqueues `ingestion.parse` on success.
3. `ingestion.parse` → enqueues `ingestion.chunk` on success.
4. `ingestion.chunk` → enqueues `ingestion.embed` on success.
5. `ingestion.embed` → enqueues `ingestion.index` on success.
6. `ingestion.index` → terminal; sets `books.ingestion_status = 'ready'`.

Each stage handler:

- Reads the book's current state from `books` row (keyed by book_id).
- Performs its work, writing outputs to DB incrementally (streaming-first per ADR-004).
- On success, calls `pgboss.send('ingestion.<next>', { bookId })` and returns.
- On failure, updates `books.ingestion_status = 'failed'` + `ingestion_error = <reason>` and does NOT enqueue downstream. The job itself is marked failed in pg-boss so retry semantics (ADR-003) can re-run just this stage.

Progress reporting is two-layer:

- **Stage column** `books.ingestion_status` reflects the current stage name (`downloading`, `cleaning`, `parsing`, `chunking`, `embedding`, `indexing`, `ready`, `failed`).
- **Progress column** `books.ingestion_progress` (0-100) is updated by each handler as it works (e.g., chunk stage updates per-chapter, embed stage updates per-batch).

## Alternatives Considered

### Alternative 1: Single monolithic job per book

- **Description**: one pg-boss job `ingestion.run` whose handler calls all six stages sequentially inline. Progress via `job.updateProgress(0-100)` mapped across stages (e.g., 0-10 = download, 10-20 = clean, ..., 90-100 = index).
- **Pros**: only one pg-boss queue; simpler operational mental model; single handler function.
- **Cons**:
  - Retry (ADR-003) requires re-running the whole job or implementing custom stage-tracking inside the handler — essentially reinventing the chained-jobs model without pg-boss's native support.
  - A handler crash mid-pipeline (e.g., worker SIGKILL during chunk) loses all in-memory state; pg-boss retry starts from the top.
  - Observability is coarser: pg-boss shows "job failed" but not "which stage"; custom logging has to fill the gap.
  - Timing each stage independently requires hand-rolled timestamps.
- **Why rejected**: contradicts ADR-003's resume-from-failed-stage requirement. Every workaround converges back on the chained-jobs approach.

## Consequences

### Positive

- Retry semantics (ADR-003) are trivial: re-enqueue the failed stage's queue for the book.
- Progress is inherent to the data model — no custom bookkeeping.
- Each stage is independently testable (inject deps, call handler, assert output).
- A crash in one stage leaves the upstream stages' outputs on disk / in DB intact; the next run picks up from where the stage failed.
- pg-boss's native retry + dead-letter handling applies per stage, not per book.

### Negative

- Six pg-boss queues to configure and monitor (vs. one).
- Each stage adds a DB round-trip (enqueue next), which is microseconds — trivial in practice.
- Developers must keep six handler files in sync; a change to payload shape (e.g., adding a new field carried between stages) requires updating the enqueue-and-receive contract across multiple files.

### Risks

- **Risk**: a stage silently fails to enqueue the next and the book is stuck.
  **Mitigation**: every successful stage handler logs `{stage: 'chunk', next: 'embed', enqueued: true}`; a missing `enqueued: true` at a given stage is an alertable condition (manual for V1; automated check in Phase 2).
- **Risk**: payload drift between stages.
  **Mitigation**: a single Zod schema (`ingestionStagePayloadSchema`) validates what each handler receives; drift fails loudly at the first affected stage.

## Implementation Notes

- Stage handlers live in `@dialogus/ingestion/application/stages/<stage>.ts`.
- `apps/worker` registers all six handlers at boot and starts pg-boss worker loops; concurrency setting applies globally (ADR-002 mandates 1).
- Each handler signature: `handle(payload: StagePayload, deps: StageDeps): Promise<void>`. Returning without throwing signals success; throwing signals failure.
