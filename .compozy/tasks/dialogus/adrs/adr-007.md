# ADR-007: Testcontainers for integration tests, CI-only

## Status

Accepted

## Date

2026-04-24

## Context

Integration tests need a real Postgres 18 + pgvector instance to exercise Drizzle migrations, pgvector queries, and pg-boss job flow. Three strategies were considered: (A) **Testcontainers per suite always** — ephemeral container per run in dev and CI; (B) **Template-database cloning** on a shared local Postgres (m5nita pattern); (C) **Hybrid** — template-DB locally, Testcontainers in CI only.

The previous `dialogus-2` attempt ran Testcontainers in the pre-commit hook, which took long enough to become a friction point and was implicated as a likely contributor to the project stalling. The key lesson is **separate daily-velocity hooks from expensive verification**: pre-commit should stay snappy; integration verification happens elsewhere.

## Decision

**Testcontainers spin a fresh Postgres 18 + pgvector container per integration test suite.** The container is created on suite setup and torn down at teardown. Pre-commit does NOT run integration tests.

Configuration:

- Pre-commit hook (`.githooks/pre-commit`): `pnpm lint && pnpm typecheck && pnpm test` — where `pnpm test` runs ONLY unit tests (with mocks).
- Integration tests live in `*.integration.test.ts` files and are selected by a separate Vitest config (`vitest.integration.config.ts`).
- Root scripts:
  - `pnpm test` — unit tests only.
  - `pnpm test:integration` — integration tests (Testcontainers); on demand locally.
  - `pnpm test:all` — both.
- CI runs `pnpm test` in the standard test job and `pnpm test:integration` in a dedicated `integration` job (with Docker-in-Docker available). Both must pass before build.
- Unit tests use MSW 2 for HTTP mocking (Gutendex, OpenAI, Anthropic) and `MockEmbeddingProvider` for deterministic embeddings.

## Alternatives Considered

### Alternative 1: Template-database cloning on a shared local Postgres (m5nita pattern)

- **Description**: Postgres runs via `docker-compose` permanently. A `template_*` DB is migrated once per changeset; each Vitest worker clones the template (~50ms) for its suite.
- **Pros**: Much faster per-suite (~50ms clone vs. 3-5s container boot); single Postgres to manage locally; works well with parallel Vitest workers.
- **Cons**: Requires Postgres always running + template refresh logic + per-worker cleanup; two test runners (one for unit, one for integration with template-clone harness) — net config complexity.
- **Why rejected**: adds mechanism specifically to shave per-suite boot time, but the owner's explicit preference is "don't run integration tests on every commit anyway" — making boot time less important than config simplicity.

### Alternative 2: Testcontainers always, including pre-commit

- **Description**: Same ephemeral container pattern, but also invoked in the pre-commit hook.
- **Pros**: Every commit has been fully verified end-to-end; no "it passed pre-commit but failed CI" surprises.
- **Cons**: Reproduces the exact failure mode of the previous project — a slow pre-commit hook kills daily velocity.
- **Why rejected**: learning from the past attempt; pre-commit budget is strictly lint+typecheck+unit.

### Alternative 3: No integration tests, mock everything

- **Description**: Mock the database too; rely on type safety and manual dogfooding for integration confidence.
- **Pros**: Zero test infra.
- **Cons**: Can't catch Drizzle migration bugs, pgvector index quirks, or pg-boss race conditions — all things the PRD explicitly worries about.
- **Why rejected**: the ingestion pipeline and retrieval tooling are exactly where real DB integration is most valuable.

## Consequences

### Positive

- Pre-commit stays fast (< 30s target on typical changes); developer velocity preserved.
- Integration tests verify the whole stack cleanly; each suite is hermetic (no cross-suite pollution).
- One test-infra mental model (Testcontainers) — no template-DB harness complexity.
- CI failure modes are reproducible locally by running `pnpm test:integration`.

### Negative

- Running all integration suites locally is slow (container boot × N suites). Owner accepts this because they run integration only on demand.
- Docker Desktop is a required local dependency (not purely a CI concern).
- CI integration job runtime can grow materially as the suite grows; may need parallelization later.

### Risks

- **Risk**: Owner forgets to run `pnpm test:integration` locally before pushing, CI fails repeatedly.
  **Mitigation**: optional git `pre-push` hook (not pre-commit) that runs integration tests; or simply live with the CI feedback loop.
- **Risk**: Testcontainers on Apple Silicon has historically been flaky with some Postgres images.
  **Mitigation**: pin to `pgvector/pgvector:pg18` or a known-good multi-arch tag; document local setup in README.
- **Risk**: CI runtime balloons beyond budget (say, > 10 min) as integration suites multiply.
  **Mitigation**: revisit with suite-level parallelization (Vitest supports it); keep only genuinely integration-requiring tests in the integration tier.

## Implementation Notes

- Use `@testcontainers/postgresql` 11+.
- Suite setup: start container, wait for ready, run migrations via Drizzle, return a pooled connection. Teardown: close pool, stop container.
- Per-suite containers (not per-test) — amortizes boot time over many assertions.
- Example `vitest.integration.config.ts`:
  ```typescript
  import { defineConfig } from 'vitest/config'
  export default defineConfig({
    test: {
      include: ['**/*.integration.test.ts'],
      pool: 'forks',           // isolate containers
      testTimeout: 30_000,     // container boot budget
      hookTimeout: 30_000,
    },
  })
  ```
- pg-boss integration: one test suite validates pgboss start → send → work round-trip on a real container.
