# ADR-007: Thread metadata (custom title + pinned) lives in Mastra Memory's metadata field

## Status

Accepted

## Date

2026-04-24

## Context

Feature 004 ADR-004 commits to thread rename + pin in V1. Mastra Memory (per product ADR-006) owns thread/message persistence; whether it exposes a metadata field for arbitrary client-controlled state on a thread depends on the pinned `@mastra/*` version.

Three storage shapes exist: (A) Mastra Memory's `thread.metadata` field if available; (B) a dedicated `thread_metadata` table in `@dialogus/db` with a thread-id foreign key; (C) localStorage, mirroring the spoiler-cap pattern from 004 ADR-002.

## Decision

**Use Mastra Memory's `thread.metadata` field as primary storage. If the pinned Mastra version does not support metadata (or its shape is too constrained), fall back to a `thread_metadata` table in `@dialogus/db`.**

Primary path:

- Schema in metadata: `{ custom_title: string | null, pinned: boolean }`. Fields default to `null` and `false`; the UI treats absence as default.
- Update: client calls Mastra's update-thread API with the merged metadata object.
- Read: Mastra's list-threads and get-thread responses include metadata; UI renders directly.

Fallback path (only if Mastra metadata is unavailable or unsuitable at the pinned version):

- New table: `thread_metadata(thread_id text pk, custom_title text, pinned boolean default false, updated_at timestamptz default now())`. `thread_id` is a soft reference to Mastra's thread id (no FK because the `mastra_*` tables are Mastra-owned).
- Two new endpoints in `apps/api`: `GET /api/library/threads/:id/metadata` returning `{ custom_title, pinned }`; `PUT /api/library/threads/:id/metadata` accepting the same shape.
- Migration `0005_thread_metadata.sql`.

The TechSpec resolves the path during task_01 of Feature 004 by inspecting the pinned `@mastra/core` API surface; the fallback is documented and ready to swap in if the verification fails.

## Alternatives Considered

### Alternative B as primary: `thread_metadata` table from day 1

- **Description**: Skip Mastra metadata entirely; always use the dedicated table.
- **Pros**: independent of Mastra's API churn; a stable table is a stable contract.
- **Cons**: forks the source of truth (Mastra owns thread + messages, dIAlogus owns thread metadata) — a future Phase 2 feature reading "all thread state" must join two systems; runs counter to product ADR-006's spirit (Mastra owns conversation persistence).
- **Why rejected**: forks ownership unnecessarily when Mastra likely supports metadata natively.

### Alternative C: localStorage like spoiler caps

- **Description**: Same pattern as 004 ADR-002; thread title and pin in browser only.
- **Pros**: zero backend code; consistent with spoiler-cap state.
- **Cons**: thread title is shown in the sidebar — the same sidebar a portfolio reviewer sees in the screencast. localStorage-only means a fresh browser session shows generic auto-titles ("primeira pergunta truncada...") for every renamed thread. Spoiler cap is a per-message detail; thread title is a persistent identity.
- **Why rejected**: thread identity should survive a browser change; localStorage is wrong for this category of data.

## Consequences

### Positive

- Single source of truth (Mastra) for the primary path; no synchronization issues.
- Mastra Studio sees the same metadata the UI does — debugging is unified.
- Fallback path is documented and ready; no second decision required if primary fails.
- Phase 2 cross-device behavior is a free side effect (Mastra metadata is server-side).

### Negative

- Verification step required during task_01: read Mastra docs at the pinned version, confirm metadata API exists with the expected fields.
- If fallback is taken, the implementation cost is ~2 tasks (table + endpoints + Drizzle adapter).
- If Mastra metadata schema is more rigid than expected (e.g., key/value strings only), the `pinned: boolean` cast becomes `pinned: 'true' | 'false'` strings — implementation handles this transparently in a small adapter.

### Risks

- **Risk**: Mastra's metadata API exists but rate-limits writes (e.g., "max 100 metadata updates per minute per thread").
  **Mitigation**: rename + pin are infrequent operations (a few per day per user); rate limits not a V1 concern.
- **Risk**: Mastra metadata is overwritten by a Mastra internal update (e.g., a thread-rename API that strips other keys).
  **Mitigation**: write the full metadata object on every update (read-modify-write with optimistic concurrency if Mastra supports `If-Match`).

## Implementation Notes

- A small wrapper hook in `apps/web`: `useThreadMetadata(threadId)` returns `{ customTitle, pinned, isLoading, mutateRename, mutatePin }`. Internally talks to Mastra metadata or falls back to the `thread_metadata` API.
- The hook hides which storage path is active; the UI never branches on it.
- TechSpec's task_01 includes "verify Mastra metadata at pinned version" as an explicit subtask. If verification fails, task_01 also schedules the fallback table migration (adds 2-3 tasks).
- Sidebar sort order: pinned threads first (descending by `updated_at`), then unpinned (descending by last message timestamp). Both keys are available from Mastra natively.
