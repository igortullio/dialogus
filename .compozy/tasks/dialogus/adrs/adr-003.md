# ADR-003: Full 5-feature MVP with lighter guardrails

## Status

Accepted

## Date

2026-04-24

## Context

The old `dialogus-2` project planned 5 features (foundation, book-catalog, book-ingestion, rag-agent, chat-ui) but failed during implementation. Analysis of the surviving specs shows two probable causes: (a) excessive upfront specification of feature 000 (50 tasks, 8-principle Constitution, 6 packages with strict DDD boundaries) relative to the one-person MVP context, and (b) a heavy test harness (Testcontainers per suite) that hung pre-commit hooks and eroded velocity. Features 002 (catalog), 004 (agent), and 005 (chat UI) were never spec'd; feature 003 (ingestion) had a quickstart but no task decomposition.

Four MVP sizing options were considered: CLI-only single book, vertical-slice 1–2 books with web chat, focused 5-feature MVP at reduced scope per feature, and the old plan's full scope with lighter guardrails. The owner chose the full 5-feature scope with lighter guardrails — wanting the complete product surface, but conscious of the old project's over-engineering.

## Decision

V1 covers the **full 5-feature scope** (foundation, catalog, ingestion, RAG agent, chat UI) but with explicitly reduced guardrail weight:

- **Constitution**: 4 principles (TypeScript-first, test-first for `ingestion` and `rag` only, observability by default, single source of truth — Gutendex). Reduced from the old 8.
- **Package count**: 5 packages (`shared` [merged `core` + `config`], `db`, `catalog`, `ingestion`, `rag`) instead of the old 6. Utilities collapsed; only real bounded contexts get their own package.
- **Test harness**: integration tests (with real Postgres via Testcontainers) run only in a dedicated CI job, never in pre-commit. Pre-commit stays at lint + typecheck + unit tests with mocked dependencies.
- **Spec cadence**: all 5 feature-level PRDs + TechSpecs + task lists written upfront, before any implementation begins — this product-level PRD is the first artifact of that session.
- **pg-boss**: fully initialized in the Foundation feature, not left "half-installed" to be picked up by a later feature.

## Alternatives Considered

### Alternative 1: Vertical-slice (1–2 curated books + minimal UI)

- **Description**: 2–3 week MVP seeded with *Don Quixote* + *Moby Dick*; minimal chat UI with citations; ingestion by script, no catalog UI.
- **Pros**: Fastest path to a working core; lowest over-engineering risk; validates RAG + citation UX end-to-end before building shell features.
- **Cons**: No catalog, no library management, no ingestion UI; portfolio screencast looks thin; doesn't match the owner's preference for a full product.
- **Why rejected**: owner explicitly chose full scope after weighing this option.

### Alternative 2: CLI-only single book

- **Description**: 1–2 week spike with one hardcoded book, terminal chat, zero web UI.
- **Pros**: Fastest path to learning Mastra + pgvector + Claude tool use; zero frontend overhead.
- **Cons**: Not shippable as a portfolio piece; no UX story; doesn't exercise the catalog/ingestion surfaces.
- **Why rejected**: owner prioritized portfolio visibility and full user journeys.

### Alternative 3: Old plan as-is (5 features, old Constitution, old test harness)

- **Description**: Replay the old plan exactly.
- **Pros**: Zero re-planning; full guardrails intact.
- **Cons**: Repeats the exact failure modes of the previous attempt (spec drift, heavy harness, over-engineering for a one-person project).
- **Why rejected**: doing the same thing and expecting different results.

## Consequences

### Positive

- Full product surface for portfolio demonstration.
- Reduced Constitution and package count lower daily friction.
- Spec-upfront for all 5 features prevents the drift that killed the previous attempt.
- Test-harness lightening keeps pre-commit fast and unblocks daily flow.

### Negative

- 7–9 weeks of planned effort — longer than the vertical-slice alternative; higher risk of real-life interruptions derailing the schedule.
- Less learning-by-doing: the spec-upfront session requires committing to feature designs before touching code.

### Risks

- **Risk**: the spec-upfront session produces designs that are wrong in ways only implementation would reveal.
  **Mitigation**: TechSpecs explicitly allow revision per feature; the PRD stays stable but tactical design can flex.
- **Risk**: 4 principles still enforce too much for a one-person MVP.
  **Mitigation**: principles are declared guidelines, not CI-blocking gates, except where they are already mechanical (TypeScript strict, lint/typecheck on pre-commit).
- **Risk**: full scope + dogfooding gate pushes the owner to skip the screencast step to claim V1 done.
  **Mitigation**: screencast is an explicit exit criterion in the PRD; treating it as non-negotiable.

## Implementation Notes

- Next artifacts in the same planning session: 5 feature-level PRDs, 5 TechSpecs, and task files per feature.
- Foundation is trimmed to ≤ 25 tasks (target, not mandate).
- Package structure: `apps/api`, `apps/web`, `apps/worker`, `packages/shared`, `packages/db`, `packages/catalog`, `packages/ingestion`, `packages/rag`. (Apps may grow; packages kept at 5.)
