# ADR-001: Chat-first product shape

## Status

Accepted

## Date

2026-04-24

## Context

When designing the primary entry point and navigation shape for dIAlogus V1, three approaches were on the table: (a) library-first — user lands on a book grid and starts chat from a specific book; (b) chat-first — user lands on chat with a sidebar of threads, selects books via a composer-level picker; (c) dashboard-first — user lands on a dashboard with multiple tiles.

The original `dialogus-2` HANDOFF assumed library-first (User stories 1–3 in feature 004 were about the library page as the entry). The market-research scan of "chat with classic books" products showed heavy convergence toward chat-first landing, and the owner's stated personal usage pattern (many short queries across a stable ~5-book set) favors returning to an ongoing conversation over navigating a catalog every visit.

## Decision

The V1 UI uses a **chat-first shape**:

- Landing route is the chat view.
- A sidebar lists previous threads.
- Empty state invites "Nova conversa".
- Within a thread, the composer's book picker selects one or more books from the personal library, with an inline "adicionar do Gutendex" shortcut.
- Library management sits behind a secondary "Gerenciar acervo" link in the sidebar footer.

## Alternatives Considered

### Alternative 1: Library-first

- **Description**: Landing is a book grid; chat opens from a per-book "Estudar" button; multi-book threads are an opt-in from the thread header.
- **Pros**: Matches the "collection anchor" mental model; discoverable for first-time users; zero retrofit cost from the old HANDOFF assumption.
- **Cons**: Slower time-to-value on return visits (requires clicking into a book even when continuing an existing thread); less aligned with the actual single-user usage pattern.
- **Why rejected**: optimizes for discovery at the expense of return-usage ergonomics; the owner expects to be a heavy returning user.

### Alternative 2: Dashboard-first

- **Description**: Landing is a dashboard with three tiles — recent threads, recent books, suggested next book.
- **Pros**: Shows product breadth on first render; visually strong for a portfolio screencast.
- **Cons**: Three competing CTAs slow time-to-first-question; dashboard logic adds surface area without core-use value.
- **Why rejected**: portfolio value does not outweigh everyday-use friction; the screencast can showcase breadth without the dashboard being the landing surface.

## Consequences

### Positive

- Minimum clicks from landing to asking a question for returning users.
- Thread list always visible — mirrors the owner's actual reading cadence (parallel book conversations).
- Clear focus for the Chat UI feature spec (one landing surface to build, not three).

### Negative

- First-time users see an empty state with no visible content; onboarding requires a strong "Primeiros passos" prefill to avoid a cold landing.
- Library is a secondary surface — book-management flows (ingestion retry, status monitoring) are one extra click away.

### Risks

- **Risk**: first-time visitors bounce without understanding the product's scope.
  **Mitigation**: empty state shows 3 recommended titles with one-click ingestion; sidebar always shows the "Gerenciar acervo" affordance.

## Implementation Notes

The Chat UI feature spec is responsible for:

- Defining the empty-state card content and behavior.
- Placing the "Gerenciar acervo" link consistently in the sidebar footer.
- Ensuring the composer's book picker handles both "add existing" and "add from Gutendex" via a single UI affordance.
