# ADR-001: Feature 003 is agent-only; chapter-summary generation moves into Feature 002

## Status

Accepted

## Date

2026-04-24

## Context

Clarifying rounds on the Feature 003 PRD surfaced a concrete need: the `get_chapter_summary` tool should return an LLM-crafted summary rather than raw (or truncated) chapter text. Raw text is cheapest but bleeds through the agent's context window on long chapters (Tolstoy, Hugo, Melville produce chapters easily over 15k tokens). On-demand summarization at tool-call time adds 3–8 seconds of latency per invocation and multiplies LLM spend during dogfooding. The durable solution is to generate a summary once per chapter and cache it.

The question is where the generation lives: inside Feature 002 (ingestion) or inside Feature 003 (RAG agent). Both features were planned against the product TechSpec, which lists `get_chapter_summary` among the agent's four tools but does not fix the production method.

## Decision

**Chapter-summary generation is owned by Feature 002 (Book Ingestion).** Feature 003 (RAG Agent) consumes summaries via a read-only port and never writes them.

- Feature 002 retroactively receives a new `chapter_summaries` table (or columns on `chapters`) and a new ingestion sub-stage (`summarize`) that fires between `chunk` and `embed`, or as part of `index`. Summaries are generated once per chapter with Claude Haiku 4.5 and cached in Postgres.
- Feature 003 ships a `ChapterSummaryRepository` port inside `@dialogus/rag`, implemented by `@dialogus/ingestion` (or the shared `@dialogus/db`). The `get_chapter_summary` tool reads from this repository only.
- Feature 003 does not contain any pg-boss job, background task, or write-side logic. `@dialogus/rag` is query-only against the data layer plus Mastra Memory for conversation state.

## Alternatives Considered

### Alternative B: Feature 003 owns summary generation via a post-ingestion job

- **Description**: Feature 002 stays sealed. Feature 003 adds a `ChapterSummaryJob` triggered on `books.ingestion_status = 'ready'`, plus a backfill command for pre-existing books.
- **Pros**: no retroactive touch to Feature 002's already-closed spec; the argument "summaries are a prompt artifact" has merit; introduces background-job experience inside the RAG package.
- **Cons**: double ownership of post-ingestion side effects — Feature 002 owns the queue and pipeline, Feature 003 subscribes to its terminal event; a book is briefly `ready` but not agent-ready, creating a state the UI must handle or hide; backfill is an extra code path that would not exist otherwise; distorts the package layer boundaries (RAG reaching into data preparation).
- **Why rejected**: the coupling is worse than the retroactive scope bump. "Ingestion produces everything an agent needs" is a stronger architectural statement than "RAG hooks into ingestion's tail."

### Alternative C: Defer summaries to Phase 2; V1 returns truncated raw text

- **Description**: `get_chapter_summary` returns `chapters.plain_text` truncated to a fixed byte budget (e.g., 12k chars). Phase 1.5 — after 2+ weeks of dogfooding — adds LLM-generated summaries.
- **Pros**: smallest V1 surface; zero coupling with Feature 002; fastest to the dogfooding gate; matches the "lighter guardrails" spirit of product ADR-003.
- **Cons**: reverses the explicit product decision to precompute summaries; truncation is not safe on books with long chapters (Les Misérables, Moby Dick, War and Peace); the tool is weaker on day one and the owner may lose trust in it before Phase 1.5 ships; two shipping cycles instead of one.
- **Why rejected**: the V1 tool would be unreliable on exactly the books V1 is meant to dogfood.

## Consequences

### Positive

- Clean package boundaries preserved: `@dialogus/ingestion` owns everything a book needs to become queryable; `@dialogus/rag` is a pure query + prompt layer.
- A book reaching `ingestion_status = 'ready'` is fully agent-ready — no intermediate state to hide or display.
- Summary generation benefits from the ingestion pipeline's existing infrastructure: pg-boss retries, streaming discipline, SHA-256 resumption, OpenAI/Anthropic client abstractions.
- Cost of summary generation is one-shot per chapter, not per user query — predictable dogfooding spend.

### Negative

- Feature 002's already-accepted spec gets a scope bump (est. 4–6 new tasks, one new table or column set, one new pipeline stage).
- The `summarize` stage adds wall-clock time to ingestion (one Haiku call per chapter; mitigated by parallelism within a book or by piggybacking on the existing stage chain).
- Feature 002 now depends on an LLM API key at ingestion time (currently it only needs OpenAI embeddings). Anthropic key expands 002's failure surface; ingestion pipeline must handle LLM rate limits and retries for the summarize stage.

### Risks

- **Risk**: Feature 002 is in `_tasks` status and further tasks may require renumbering or careful insertion.
  **Mitigation**: add the new tasks as 002's highest-numbered files (append rather than insert); update 002's TechSpec with an ADR-level note referencing this 003 ADR.
- **Risk**: summary quality varies across books; a poorly summarized chapter could degrade the agent's answers in subtle ways.
  **Mitigation**: summaries are read alongside retrieved chunks, not replacing them; the agent always has the raw excerpt via `semantic_search`. The dogfooding validation round (≥ 10 owner-posed questions) exercises summary-dependent journeys.
- **Risk**: bilingual corpus — summaries must match chapter language.
  **Mitigation**: the summarize-stage prompt reads `books.languages` and instructs Haiku to summarize in the same language.

## Implementation Notes

- Feature 002 changes (amendment to `_techspec.md` + new tasks):
  - Migration: new table `chapter_summaries(id uuid pk, chapter_id uuid fk unique, book_id uuid fk, summary text, token_count int, model text, generated_at timestamptz)` OR two columns on `chapters` (`summary text`, `summary_generated_at timestamptz`). Final shape resolved in 003 TechSpec.
  - New stage `summarize` in `@dialogus/ingestion/application/stages/summarize.ts`, invoked between `chunk` and `embed`. Reads chapter text, calls Anthropic with cached system prompt, writes result.
  - New port `ChapterSummaryGenerator` abstracting the LLM call; `AnthropicChapterSummaryGenerator` as primary adapter, `MockChapterSummaryGenerator` for tests.
  - `books.ingestion_status` enum gains an optional intermediate `summarizing` value (if we want visible progress) OR the stage reports under `chunking` to avoid enum churn; final call made in 003 TechSpec.
- Feature 003 changes:
  - `@dialogus/rag` declares `ChapterSummaryRepository` port; wiring in `apps/mastra` injects the Drizzle-backed implementation from `@dialogus/ingestion` or a shared package.
  - `get_chapter_summary` tool calls `chapterSummaryRepo.findByChapterId(chapterId)` and returns `{ summary, book_id, chapter_id, chapter_ordinal, chapter_title }`.
  - Tool contract guarantees a summary exists for any `ready` book's chapters. If missing, tool returns an explicit error; UI surfaces it (should never happen in normal ops).
