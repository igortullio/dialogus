# ADR-001: Linear + transparent delivery with E2E wiring and Day-1 polish

## Status

Accepted

## Date

2026-04-24

## Context

Foundation is the scaffolding feature for dIAlogus V1. Two orthogonal decisions shape how it ships:

- **Vertical-slice thickness**: pure scaffold (nothing wires to anything), E2E wiring proof (one request crosses web → api → db), or shell UI preview (layout chassis also included).
- **Developer experience transparency**: README walks each step explicitly, or a single `pnpm bootstrap` command hides the mechanism.

A third dimension — **portfolio polish depth** in the first commit — is also in scope (minimum / Day-1 ready / full 2026 hardening).

The previous `dialogus-2` attempt over-specified Foundation (50 tasks, 8 principles) and still shipped a partially-integrated outcome. Product-level ADR-003 mandates lighter guardrails. This ADR captures the Foundation-specific delivery shape.

## Decision

Foundation ships with **Linear + transparent delivery, E2E wiring proof, and Day-1 ready polish**:

- **Delivery transparency**: README explicitly walks `docker compose up -d` → `pnpm db:migrate` → `pnpm dev` as three visible steps. No `bootstrap` command hides complexity.
- **E2E wiring proof**: `apps/web` landing is a Next 16 Server Component that fetches `/health` from `apps/api` at render time and renders "dIAlogus — api: up / db: up". This single request exercises env loading, fetch between processes, Drizzle connect, and applied migrations.
- **Day-1 polish**: first commit includes README with 5-line quickstart + 3-paragraph architecture summary + "Next steps" pointer, LICENSE (MIT), `.env.example` documenting every planned env var across all features, `.nvmrc` pinning Node 22.13+, `packageManager` field pinning pnpm 9.15+, and Conventional Commits guidance in README.
- **Quality gates**: pre-commit runs `lint + typecheck + unit tests` (NOT integration); CI runs 3 jobs (lint-and-typecheck, test, build).

## Alternatives Considered

### Alternative 1: Magic bootstrap (single `pnpm bootstrap` command)

- **Description**: a root script `pnpm bootstrap` performs docker-compose up + wait-for-db + migrate + seed + dev launch. README collapses to one command.
- **Pros**: fastest time-to-hello-world; impressive in a screencast.
- **Cons**: hides the mechanism from the developer (they learn nothing); bootstrap failures surface as opaque errors; non-idiomatic versus the m5nita reference; contradicts Day-1 ready polish (which favors legibility over brevity).
- **Why rejected**: legibility wins for a portfolio piece; the 3 commands are themselves educational.

### Alternative 2: Pure scaffold without E2E wiring

- **Description**: `apps/web` renders a static "dIAlogus" placeholder; `/health` exists on `apps/api` but no cross-talk; Foundation proves "tools compile", not "stack works".
- **Pros**: fastest to finish; smallest code surface.
- **Cons**: no proof that env, fetch, DB, and migrations all wire correctly — first real test comes in Feature 001 and failures cascade; screencast is less convincing.
- **Why rejected**: E2E wiring is a one-Server-Component addition yielding outsized confidence.

### Alternative 3: Shell UI preview included in Foundation

- **Description**: Foundation ships structural UI (sidebar, main pane, empty state, Tailwind v4 + shadcn installed, branded header).
- **Pros**: Foundation already looks like the real product in screencasts; Feature 004 inherits a UI chassis.
- **Cons**: pulls Tailwind v4 + shadcn decisions before the Chat UI TechSpec; locks in layout choices that should be made with Feature 004 context; risks rework.
- **Why rejected**: YAGNI for Foundation; Feature 004 owns layout decisions.

### Alternative 4: Full 2026 hardening polish

- **Description**: Day-1 items + commitlint hook enforcing Conventional Commits + Renovate / Dependabot + PR template + CODE_OF_CONDUCT.
- **Pros**: maximum signal of "mature open-source project".
- **Cons**: maintenance burden for a single-user personal project; commit message enforcement is friction without a team; Renovate config is noise on a tree that rarely moves.
- **Why rejected**: can be added in Phase 2 if dogfooding extends; Day-1 ready is sufficient signal for portfolio.

## Consequences

### Positive

- Every step of the setup flow is legible; failures are diagnosable at their exact stage.
- First commit is immediately shareable (public-ready) — LICENSE and quickstart in place.
- E2E wiring via Server Component is a one-file demonstration that the stack truly works.
- Matches m5nita's transparent convention, reducing cognitive mismatch for the developer.
- Features 001-004 inherit a stable baseline with known extension points.

### Negative

- Developers must run (or copy-paste) 3 commands instead of 1.
- E2E wiring adds a small amount of code to `apps/web` (a fetch call and status rendering) that will be rewritten in Feature 004.
- Day-1 polish takes ~1–2 hours beyond pure scaffold work.

### Risks

- **Risk**: README quickstart drifts from actual commands over time as features add setup steps.
  **Mitigation**: keep README quickstart stable across Foundation lifetime; each feature updates README if it adds a required step.
- **Risk**: Server Component fetch from `apps/web` to `apps/api` fails due to a Next 16 + Hono CORS surprise.
  **Mitigation**: Server Component fetches are server-side; no CORS preflight. Verify in Foundation implementation.

## Implementation Notes

- `.env.example` includes every env var planned across all features (with comments noting which feature consumes each), not only Foundation's.
- `packageManager` field must match `.nvmrc` guidance on Corepack activation.
- README quickstart must be tested verbatim by the project owner against a fresh clone before V1 ships.
