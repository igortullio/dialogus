# ADR-007: Flat per-stage storage layout

## Status

Accepted

## Date

2026-04-24

## Context

The ingestion pipeline produces two types of file artifacts on local disk: raw downloaded EPUB/TXT (stage 1 output) and cleaned plain text (stage 2 output). Both need a persistent home on the filesystem across `apps/worker` restarts. Three layouts were considered:

- **Flat per-stage**: `./storage/raw/<gutendex_id>.<ext>` and `./storage/clean/<gutendex_id>.txt`. Two top-level directories, one file per book per stage.
- **Nested per-book**: `./storage/<gutendex_id>/raw.epub`, `./storage/<gutendex_id>/clean.txt`, `./storage/<gutendex_id>/meta.json`. Each book gets its own directory.
- **Content-addressable**: `./storage/raw/<sha256>.<ext>`. Book rows reference via `raw_hash`; natural deduplication across books.

## Decision

Use **flat per-stage layout**:

```
./storage/
├── raw/
│   ├── 996.epub         Don Quixote (EPUB)
│   ├── 15.txt           Moby Dick (TXT fallback)
│   └── 2000.epub        ...
└── clean/
    ├── 996.txt          Don Quixote cleaned plain text
    ├── 15.txt           Moby Dick cleaned
    └── 2000.txt         ...
```

Filename convention: `<gutendex_id>.<extension>`. Two top-level directories (`raw/`, `clean/`), flat inside.

`./storage/` is listed in `.gitignore`; it is a local-dev persistence surface, not a build artifact. In docker-compose, the directory is mounted as a volume for persistence across container restarts (though V1 is local-node-only; compose volume is for Phase 2 containerized dev).

## Alternatives Considered

### Alternative 1: Nested per-book

- **Description**: `./storage/<gutendex_id>/raw.epub` + `./storage/<gutendex_id>/clean.txt` + optional `meta.json` per book.
- **Pros**: `rm -rf ./storage/<id>/` removes everything for a book cleanly; soft-delete cleanup job (Phase 2) trivially implements.
- **Cons**: one extra directory level to navigate; `ls` no longer gives a quick bird's-eye view ("what stages are present for which books"); tooling friction for a single-user developer.
- **Why rejected**: cleanup is not a V1 concern (books are not hard-deleted), so the primary benefit doesn't apply; flat layout is easier to reason about day-to-day.

### Alternative 2: Content-addressable by SHA-256

- **Description**: `./storage/raw/<sha256>.<ext>`; `books.raw_hash` references. If two Gutendex IDs somehow yield byte-identical files, they share storage.
- **Pros**: natural dedup; cryptographic integrity check built-in.
- **Cons**: filenames opaque to humans (`3a7f8b...epub` vs. `996.epub`); cleanup requires walking the `books` table to find unreferenced hashes; complexity for a benefit (dedup) that realistically never triggers on Gutendex (every book has a unique ID).
- **Why rejected**: complexity without practical payoff.

## Consequences

### Positive

- `ls ./storage/raw/` gives an immediate overview of downloaded books.
- `rm ./storage/raw/996.epub` is obvious; developer doesn't need to check a reference count.
- Filesystem tooling (find, grep, rsync, etc.) works naturally.
- Path construction in code is a single template: `\`./storage/${stage}/${gutendexId}.${ext}\``.
- `docker-compose` volume mount is straightforward: `./storage:/app/storage`.

### Negative

- Per-book cleanup requires two deletes (one per directory). Trivial in V1; if Phase 2 adds cleanup automation, the helper is 2 lines of code.
- No natural dedup: two Gutendex entries with identical content (theoretical; unlikely) would store twice.

### Risks

- **Risk**: `./storage/` grows unboundedly during dogfooding.
  **Mitigation**: not a V1 concern (estimated 500 MB after 10 books). Phase 2 can add a cleanup job that deletes files for soft-deleted books older than N days.
- **Risk**: filesystem path conflicts if `gutendex_id` format changes upstream (unlikely; Gutendex IDs are stable positive integers).
  **Mitigation**: Gutendex has not changed ID format since inception; if a theoretical change happens, it's a Phase 2 migration.
- **Risk**: permissions issues when docker-compose mounts the directory.
  **Mitigation**: README documents `chmod` or uid/gid alignment if needed; V1 runs locally as the dev user, so permissions are trivially correct.

## Implementation Notes

- `GutendexDownloader.download(gutendexId, ext)` returns the path `./storage/raw/${gutendexId}.${ext}` after writing the file.
- `GutenbergCleaner.clean(rawPath, gutendexId)` reads from `rawPath`, writes to `./storage/clean/${gutendexId}.txt`, returns the cleaned path.
- Both stages create their target directory if missing (`mkdir -p` at the start of the handler).
- Unit tests use a temporary directory via `fs.mkdtemp()` per test to avoid pollution.
- Integration tests mount a temporary path into `apps/worker` via env var override of the storage root (`STORAGE_ROOT` env var, default `./storage`).
