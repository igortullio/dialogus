# ADR-006: Mastra Memory owns conversation persistence

## Status

Accepted

## Date

2026-04-24

## Context

Conversation threads, messages, tool calls, and tool outputs (which carry citations) need a persistence model. Three approaches were considered: (A) **Mastra Memory** via `@mastra/pg` — Mastra owns a `mastra_*` schema and evolves it automatically; (B) **Custom Drizzle tables** in `@dialogus/db` for `threads`, `messages`, `citations`, `thread_book_state`; (C) **Sidecar** — Mastra Memory owns threads/messages, Drizzle owns a `citations` table keyed to chunks.

Given ADR-005 picked Mastra Dev Server as the agent runtime, and given Mastra Studio natively visualizes threads stored in Mastra Memory, option A is the idiomatic path and yields the least code.

## Decision

Use **Mastra Memory with `@mastra/pg` as the storage adapter**:

- `mastra.config.ts` configures `@mastra/pg` against `DATABASE_URL` (same DSN as `@dialogus/db`).
- Mastra owns and migrates the `mastra_*` schema (threads, messages, tool_calls, tool_outputs). dIAlogus never runs Drizzle migrations on these tables.
- Citations are emitted by the `semantic_search` tool as `{ book_id, chapter_id, chapter_ordinal, chunk_id, excerpt_preview }` on `tool_output`; Mastra persists them as-is.
- **Spoiler cap is NOT stored in Mastra Memory**: it's passed per-request as a tool parameter (or request body field), with the UI holding the source of truth per thread/book.
- If persistent cross-session spoiler-cap memory becomes desirable, a thin `thread_book_preferences(thread_id, book_id, current_chapter)` table owned by `@dialogus/db` can be introduced in the Chat UI TechSpec without disturbing this ADR.

## Alternatives Considered

### Alternative 1: Custom Drizzle schema in `@dialogus/db`

- **Description**: `threads`, `messages`, `citations` (FK to chunks), `thread_book_state` all defined by dIAlogus; Mastra Memory configured with a custom storage adapter or fed messages as per-turn context.
- **Pros**: Full control over the schema; foreign-key enforced integrity for citations; direct SQL for analytics (citation frequency, spoiler-cap behavior).
- **Cons**: Reimplements what Mastra provides; custom memory adapters for Mastra are more surface area to maintain; Mastra Studio may not visualize threads stored outside `@mastra/pg` natively.
- **Why rejected**: with Mastra Dev Server as the runtime (ADR-005), forgoing Mastra Memory also forgoes the main reason that runtime was chosen. Net negative.

### Alternative 2: Sidecar — Mastra Memory + dedicated `citations` table

- **Description**: Mastra owns threads/messages; Drizzle owns a `citations` table with FK to `chunks`. The agent writes citations after each retrieval tool-call.
- **Pros**: Keeps tool-output citations referentially sound; enables analytical queries on citations without parsing Mastra's JSON.
- **Cons**: Dual writes per message (Mastra + Drizzle); citations can become orphans if Mastra deletes a message; two systems of record to reconcile.
- **Why rejected**: complexity outweighs benefit for V1. If citation analytics become important, Phase 2 can project Mastra's tool outputs into a materialized view.

## Consequences

### Positive

- Minimal code in `@dialogus/rag` for persistence concerns — Mastra handles it.
- Mastra Studio shows threads, messages, tool calls, and tool outputs natively — huge productivity win during prompt tuning.
- Schema evolution is Mastra's responsibility; dIAlogus is insulated from breaking changes via Mastra version pinning.

### Negative

- Citations live inside Mastra's JSON-shaped tool output, not in a first-class normalized table. Analytics queries ("how many citations cite chapter 3 of Moby Dick") require JSON path operators, not simple joins.
- Spoiler cap state has nowhere native to live; UI must remember it. If the user switches devices, the cap resets unless synced server-side (deferred).
- Coupling: replacing Mastra Memory later means migrating conversation history out of `mastra_*` — non-trivial but bounded.

### Risks

- **Risk**: `@mastra/pg` schema evolves in a breaking way between minor versions, invalidating local DBs.
  **Mitigation**: pin `@mastra/*` to exact versions; review changelog before upgrading; keep Mastra DSN pointed at a distinct logical DB or schema so wiping it is cheap during dev.
- **Risk**: Citation references become stale when a chunk is deleted (book removed from library).
  **Mitigation**: soft-delete books in V1 (already planned in PRD); chunks survive. Phase 2 can add a cleanup policy.

## Implementation Notes

- `mastra.config.ts` example shape:
  ```typescript
  import { Mastra } from '@mastra/core'
  import { PgStorage } from '@mastra/pg'
  import { loadConfig } from '@dialogus/shared/config'
  import { createDialogusAgent } from '@dialogus/rag'

  const { DATABASE_URL } = loadConfig()

  export const mastra = new Mastra({
    storage: new PgStorage({ connectionString: DATABASE_URL }),
    agents: { dialogusAgent: createDialogusAgent(/* deps */) },
  })
  ```
- Citations emitted by `semantic_search` must include both `chunk_id` (for server-side excerpt lookup) and an `excerpt_preview` (short text for instant UI display). UI resolves full excerpt via `GET /api/library/chunks/:id`.
- Spoiler cap: `/api/qa/threads/:id/messages` request body carries `{ message, spoiler_caps: Record<book_id, chapter_ordinal> }`; Mastra passes it to `semantic_search` tool.
