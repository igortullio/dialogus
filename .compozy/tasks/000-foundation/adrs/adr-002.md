# ADR-002: Generate-only Drizzle migrations

## Status

Accepted

## Date

2026-04-24

## Context

Drizzle ORM ships three commands: `push` (sync schema from TypeScript definitions directly into the DB without a SQL file), `generate` (emit an SQL migration file from schema diffs), and `migrate` (apply pending SQL migrations in order). Three policies were on the table for dIAlogus:

- **Dual-mode (m5nita pattern)**: dev freely uses `push`; prod uses `generate` + `migrate`.
- **Generate-only strict**: all schema changes, including dev iteration, go through `generate` + `migrate`; `push` is disallowed.
- **Push-only in Foundation**: Foundation uses `push` (no domain tables yet); Feature 001 switches to `generate` + `migrate`.

The old `dialogus-2` attempt had `generate`-style migrations committed but did not enforce a single mode, leading to schema drift between the persisted SQL files and what was actually in the dev DB.

## Decision

**Generate-only strict.** Every schema change, including Foundation's first migration, flows through `drizzle-kit generate` (produces an SQL file committed to `packages/db/drizzle/`) followed by `drizzle-kit migrate` (applied via `pnpm db:migrate`). `drizzle-kit push` is not installed as a root script; if developers invoke it directly in a package, they are explicitly going off-policy.

Root scripts:

- `pnpm db:generate` — run `drizzle-kit generate` to produce the next SQL migration file.
- `pnpm db:migrate` — apply pending SQL migrations (then run `pgboss.start()` per ADR-003).
- `pnpm db:studio` — launch Drizzle Studio.
- `pnpm db:reset` — drop + recreate the database, then run `db:migrate`.

No `db:push`.

## Alternatives Considered

### Alternative 1: Dual-mode (m5nita pattern)

- **Description**: dev uses `push` for fast schema iteration; commits happen via `generate` + `migrate`; prod/CI uses `migrate` only.
- **Pros**: fastest dev loop — schema changes reflected in the DB without a generate round-trip; matches the m5nita reference verbatim.
- **Cons**: two mental models (push in dev, generate in prod); easy for devs to forget to `generate` before committing, leading to schema drift between TypeScript and SQL files; code review of schema changes is only possible when the migration is committed, not when `push` runs ad hoc.
- **Why rejected**: the exact drift failure mode that hurt `dialogus-2`; for a solo-dev portfolio project, the cost of strict discipline is small, and the benefit of "the SQL file is always the truth" is large.

### Alternative 2: Push-only in Foundation, switch later

- **Description**: Foundation doesn't have domain tables, so use `push` and skip SQL committal; Feature 001 introduces the `generate` workflow.
- **Pros**: slightly less Foundation ceremony.
- **Cons**: a convention change mid-project (drift risk); Foundation's first real SQL migration (the one creating `pgvector` + `uuid-ossp` extensions + `system_health`) is the easiest one to get right and also the one that will be inspected most by reviewers — worth committing.
- **Why rejected**: consistency from day 1 costs nothing more and yields a reviewable first migration that demonstrates the discipline.

## Consequences

### Positive

- SQL file is always the source of truth. Any state difference between filesystem and DB is observable via `drizzle-kit generate` diff.
- Code review gets a readable SQL diff on every schema change.
- Foundation's first migration (extensions + canary) is an artifact that signals discipline to reviewers.
- Zero risk of dev-prod drift; CI and local setup apply identical migration sequences.

### Negative

- Schema iteration in dev requires a `generate` round-trip instead of an instant `push`. On a single-developer project with short migration files, this is a minor friction.
- Reverting a mid-development migration requires either dropping + re-migrating (`pnpm db:reset`) or hand-editing the SQL file — both acceptable.

### Risks

- **Risk**: developer forgets to run `pnpm db:generate` after editing a schema file and pushes TS-only changes.
  **Mitigation**: typecheck will not catch this, but the next `pnpm db:migrate` on a fresh DB will produce a shape that doesn't match the TS types at runtime, surfacing the error. Feature 002 integration tests additionally catch this since they run migrations against a fresh Testcontainers DB.

## Implementation Notes

- `drizzle.config.ts` lives at `packages/db/drizzle.config.ts` and points at `./src/schema/` for definitions and `./drizzle/` for SQL output.
- Initial migration (`drizzle/0000_init.sql`) is generated, then hand-edited to add raw SQL for extensions (`CREATE EXTENSION IF NOT EXISTS vector;`, `CREATE EXTENSION IF NOT EXISTS "uuid-ossp";`) at the top. This is the one legitimate reason to hand-edit a generated file.
- Seed row insertion (`INSERT INTO system_health (status) VALUES ('ok');`) appended to the migration.
- The `db:reset` script drops the database and recreates it via `psql` rather than dropping individual tables — simpler and safe for Foundation.
