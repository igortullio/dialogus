# ADR-005: Thread book scope is fixed at thread creation

## Status

Accepted

## Date

2026-04-24

## Context

A thread is scoped to one or more books (set via the composer's book picker on first message). The question is whether the user can change that scope mid-conversation: add more books, remove existing ones, or both. Each option carries different agent semantics (already-cited messages reference books that may now be removed) and different UI surface (sidebar header chips, popover for editing, etc.).

Three meaningful options exist: (A) lock at creation, (B) allow add but not remove, (C) allow add + remove.

## Decision

**The book scope of a thread is fixed at thread creation. To change scope, the user starts a new thread.**

- The composer's book picker is active only when creating a new thread (when there are zero messages).
- Once the first message is sent, the book scope becomes a read-only chip array in the thread header.
- The chip array shows each book's title + language flag + spoiler-cap chip when active. Clicking a chip opens the spoiler-slider popover for that book; it does NOT offer to remove the book.
- Adding more books to an active thread is disabled. The empty-state next to the composer suggests "Comece uma nova conversa para incluir outros livros."

## Alternatives Considered

### Alternative B: Add books mid-thread, never remove

- **Description**: A `+` button in the header chip array opens a Gutendex-search popover that adds a book (must be `ready`) to the thread's `book_ids`. Removal is disabled.
- **Pros**: enables expansive discussions ("agora compare com Crime e Castigo"); preserves citation history (no ghosts).
- **Cons**: implicit retrieval semantics shift mid-thread — questions before the add only had book A in scope; questions after include both. The user could be confused about why the agent cites book B for an earlier-feeling question.
- **Why rejected**: the cognitive cost of "what was in scope at this point" outweighs the convenience. Starting a new thread is a clean reset.

### Alternative C: Free add + remove

- **Description**: Both `+` and `×` chip controls.
- **Pros**: maximum flexibility.
- **Cons**: removing a book makes prior citations point to "removed" books; UI has to either gray them out (looks broken) or hide them (rewrites history); retrieval semantics on subsequent turns become unpredictable.
- **Why rejected**: citation history is the trust contract; rewriting it visually is worse than disallowing the action.

### Alternative D: Per-message override

- **Description**: Each message has an optional book_ids override; default is thread scope.
- **Pros**: surgical retrieval control per question.
- **Cons**: enormous cognitive load for the user; obscures the "this thread is about these books" mental model; chat-first UX forbids this much per-message configuration.
- **Why rejected**: violates chat-first product shape (product ADR-001).

## Consequences

### Positive

- Clear mental model: one thread = one fixed set of books. New scope = new thread.
- Citations always trace back to a book that's still in the active scope — no UI ghosts, no "removed book" rendering.
- The composer's book picker is single-purpose: a thread-creation widget. No conditional logic for "first turn vs. subsequent."
- The thread sidebar can show the book set as a stable secondary identifier (e.g., chip showing "MD + DC" for Moby Dick + Dom Casmurro) — discriminates threads that share titles.

### Negative

- The user who realizes mid-conversation "I should also have included book X" must start over. Workaround: the new thread inherits localStorage spoiler caps if the same `book_id` was capped in a prior thread? No — caps are per-thread per ADR-002. So the user re-sets the cap.
- "Discuss-then-pivot" workflow is awkward — a thread about Moby Dick can't extend into a comparison with War and Peace. The user creates a fresh thread for the comparison.
- Phase 2 may relax this if dogfooding reveals strong demand; the relaxation is forward-compatible (allow add-only first, remove-only second).

### Risks

- **Risk**: the user attempts to use the picker mid-thread and sees no affordance, gets confused.
  **Mitigation**: header chips have a clear "Trocar livros = nova conversa" tooltip. Empty composer states reinforce the message.
- **Risk**: a thread becomes "stale" because the user wants to add a book but doesn't want to lose history.
  **Mitigation**: thread rename + pin (ADR-004) lets the user keep meaningful threads visible; the new comparative thread can be pinned alongside.

## Implementation Notes

- Thread creation flow: the user clicks "Nova conversa" → composer opens with the book picker active → user selects 1-3 `ready` books → user types message + sends → thread is created via Mastra at first send, with `book_ids` baked into the thread's metadata.
- Header chip array: read-only after thread has ≥ 1 message. Renders from `thread.metadata.book_ids` or equivalent Mastra field; falls back to deriving from the first message's payload if metadata is unavailable.
- The book picker component itself is reused on the `/library` page's "Adicionar do Gutendex" modal — UI consistency across surfaces.
- The thread chips show language flag emoji (🇬🇧 / 🇧🇷) for instant scope recognition.
