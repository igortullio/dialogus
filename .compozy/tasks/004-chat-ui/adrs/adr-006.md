# ADR-006: assistant-ui owns the chat shell; shadcn/ui owns everything else

## Status

Accepted

## Date

2026-04-24

## Context

The product TechSpec lists both `assistant-ui` and `shadcn/ui` as locked dependencies for `apps/web`. By 2026 these libraries have meaningful overlap: shadcn/ui shipped AI-domain primitives (`<Conversation>`, `<Message>`, `<InlineCitation>`, `<Composer>`) that compete with assistant-ui's conversation primitives. Either could plausibly host the chat experience. The Chat UI also needs ~30 non-chat UI primitives (buttons, dialogs, popovers, sheets, tooltips, dropdowns, alerts, sliders, tabs, etc.).

A clean responsibility split reduces decision fatigue during implementation and prevents two libraries from solving the same problem inconsistently.

## Decision

**`assistant-ui` owns the chat shell only.** Specifically: `<Thread>`, `<ThreadList>` (composer-style), `<Composer>`, `<Message>`, and message-render primitives. Streaming, auto-scroll, keyboard navigation, Markdown rendering inside messages, and code highlighting come for free from assistant-ui at the pinned version.

**`shadcn/ui` owns everything else.** Buttons, dialogs, sheets, popovers, tooltips, dropdowns, alerts, sliders, tabs, badges, cards, inputs, separators, navigation menus, and any primitive that is not specifically a chat conversation atom. shadcn/ui's AI-domain primitives (e.g., `<InlineCitation>`) are NOT used — the citation badge is a custom dIAlogus component built on shadcn `<Tooltip>` + shadcn `<Sheet>`, so the citation contract from Feature 003 ADR-007 stays under our control.

**Glue components** are dIAlogus-owned thin wrappers around assistant-ui primitives (`<DialogusThread>`, `<DialogusComposer>`) so an upstream API change in assistant-ui is one component's worth of edits, not a feature-wide migration.

## Alternatives Considered

### Alternative 1: shadcn/ui AI primitives for everything (replace assistant-ui)

- **Description**: Drop assistant-ui; use shadcn/ui's `<Conversation>`, `<Message>`, `<InlineCitation>`, etc.
- **Pros**: one library; shadcn/ui has design consistency by construction; `<InlineCitation>` is pre-built.
- **Cons**: shadcn/ui's AI primitives are newer than assistant-ui (less battle-tested for edge cases like SSE reconnect, mid-stream cancel, tool-use rendering); product TechSpec explicitly committed to assistant-ui (`<Thread>`, `<Composer>`); the breaking-change risk window is 2026-Q2.
- **Why rejected**: product TechSpec is the contract; defection requires a stronger argument than "newer alternative exists."

### Alternative 2: assistant-ui for everything, including library and sidebar

- **Description**: Stretch assistant-ui to cover non-chat surfaces.
- **Pros**: single library; no integration tax.
- **Cons**: assistant-ui is chat-domain by design; using it for a library grid or a thread sidebar is non-idiomatic; fewer primitives for those surfaces.
- **Why rejected**: forces a wrong-domain abstraction; library ergonomics suffer.

### Alternative 3: Custom on Radix/Headless UI

- **Description**: Skip both libraries; build chat primitives directly on Radix Primitives or Headless UI.
- **Pros**: maximum control.
- **Cons**: re-implements months of assistant-ui work (auto-scroll, streaming, retry, keyboard, voice-disabled state); ships bugs the libraries already fixed.
- **Why rejected**: not justified for a single-user portfolio project; assistant-ui exists exactly for this use case.

## Consequences

### Positive

- Clean responsibility line: chat-domain → assistant-ui; UI-domain → shadcn/ui.
- Glue layer (`<DialogusThread>`, `<DialogusComposer>`) absorbs upstream churn cheaply.
- shadcn/ui's `init` config + component-by-component generation pattern keeps the Tailwind-v4 + design-token discipline working uniformly.
- Custom citation badge stays small and ours: hover preview tooltip + side-panel sheet are shadcn primitives composed by us, not adopted from a black-box `<InlineCitation>` whose API we didn't design.

### Negative

- Two component conventions in the same app: assistant-ui's bring-your-own-styles pattern vs. shadcn/ui's class-based component pattern. The glue components mediate.
- assistant-ui at pre-1.x maturity may ship breaking minors; pinned versions help, but periodic upgrades are still required.
- Voice input (assistant-ui supports it; PRD non-goals disable it) requires explicit prop/config to stay disabled — risk of accidental enabling in a future upgrade.

### Risks

- **Risk**: assistant-ui's `<Thread>` API changes between minor versions and breaks the streaming or scroll behavior.
  **Mitigation**: pinned exact version in `package.json`; glue layer absorbs the surface; upgrade is a deliberate task.
- **Risk**: shadcn `<Tooltip>` + `<Sheet>` together create a focus-trap edge case (tooltip dismissed by sheet open, etc.).
  **Mitigation**: integration tests in task_07 of 004 exercise the badge → preview → click flow with assertions on focus order.

## Implementation Notes

- `apps/web/package.json` deps: `@assistant-ui/react@<pinned>`, `@assistant-ui/react-ai-sdk@<pinned>` (Vercel AI SDK adapter), shadcn `init` config + components on demand, `@radix-ui/react-*` peers via shadcn.
- Glue layer in `apps/web/src/components/chat/`: `DialogusThread.tsx`, `DialogusComposer.tsx`, `DialogusMessage.tsx`. Each is a thin wrap that re-exports assistant-ui primitives with project-specific styling and event hooks.
- Citation badge in `apps/web/src/components/citation/`: `CitationBadge.tsx`, `CitationTooltip.tsx`, `CitationSidePanel.tsx`. Built on shadcn `<Tooltip>` + shadcn `<Sheet>`.
- Chat-shell-only assistant-ui wraps: never use assistant-ui's primitives outside `apps/web/src/components/chat/`.
