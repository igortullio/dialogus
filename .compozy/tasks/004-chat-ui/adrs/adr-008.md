# ADR-008: Streaming-aware citation marker parser

## Status

Accepted

## Date

2026-04-24

## Context

Feature 003 emits `{{cite:<chunk_id>}}` markers inline in the streaming response. SSE chunks arrive token-by-token; a marker can straddle chunk boundaries (`{{cite:abc-` then `123}}`). The Chat UI must transform markers into rendered badges (per 004 ADR-003: superscript badge + hover preview + click-for-side-panel). Four parsing strategies were considered: (1) streaming-aware buffer; (2) render markers as text during stream, replace at done event; (3) hide markers during stream, reveal badges at done; (4) placeholder during stream, swap to badge when marker fully arrived.

The streaming-aware option is the only one that delivers badges in real time as text streams. The trade-off is parser complexity: managing a buffer that holds partial markers across chunks.

## Decision

**Implement a streaming-aware parser that emits badges in real time as their markers complete.**

The parser is a stateful function consuming SSE token events from `useChat`. State machine:

- **State `text`** (default): forwarding text to the renderer.
- On encountering `{`: peek next char.
- On encountering `{{`: enter state `marker_pending`. Buffer subsequent characters.
- **State `marker_pending`**: accumulate characters into a buffer; check for `}}` terminator on every new char.
- On `}}` arrival: extract the buffered content; if it matches `cite:<UUID-v4>`, emit a `<CitationBadge>` token with the chunk_id; otherwise emit the literal characters as text. Return to state `text`.
- If the buffer exceeds a sanity limit (e.g., 60 chars without `}}`) without finding the terminator: bail out — emit the buffered chars as literal text. Return to state `text`. Prevents an unbounded buffer if the LLM goes off-format.

The parser sits between the SSE stream and the message renderer. Its output is an alternating sequence of `text` and `citation` tokens that the renderer converts into React elements.

## Alternatives Considered

### Alternative 2: Render raw markers during stream, swap at done event

- **Description**: During stream, the user sees `{{cite:abc-123}}` as plain text inside the prose. When Mastra emits the stream-done event, a single pass over the final message text replaces all markers with badges.
- **Pros**: trivially simple parser (one regex replace at the end); no state machine; no buffer.
- **Cons**: user sees `{{cite:...}}` raw text during the entire stream — for a 6-second response, that is 6 seconds of "broken-looking" output. Visual jank. Also, the swap happens all at once at the end, which feels less smooth than progressive badge appearance.
- **Why rejected**: visible raw markers during stream is a poor UX signal; users interpret "machine code in the prose" as either a bug or unfinished output.

### Alternative 3: Hide markers entirely during stream, reveal at done

- **Description**: During stream, the parser strips markers; user sees clean prose with no citations. At done event, badges are inserted at the recorded positions.
- **Pros**: stream is visually clean.
- **Cons**: user perceives the response as "uncited" until the last moment, then citations appear all at once — felt like a popup; harder to match badges to claims because they appear after the user already read the claim.
- **Why rejected**: undermines the trust signal citations are meant to provide; "all citations appear at the end" reads as a separate event from the response.

### Alternative 4: Placeholder during stream, swap when marker complete

- **Description**: When `{{` opens, immediately render a `…` placeholder; when `}}` closes, swap the placeholder for the badge.
- **Pros**: visual feedback that "a citation is coming"; stream stays clean of raw braces.
- **Cons**: the placeholder is a third visual state competing with text and badge; users may interpret `…` as part of the agent's prose; complexity is similar to streaming-aware without the cleaner result.
- **Why rejected**: streaming-aware solves the same problem with less visual noise; the badge appears at the same moment a placeholder would have been swapped.

## Consequences

### Positive

- Badges appear progressively as markers complete — same temporal pattern as the prose around them. UX feels "continuous."
- No visible raw `{{cite:...}}` artifacts during stream.
- Parser is testable in isolation: feed a series of text chunks, assert the emitted token sequence.
- Phase 2 extensions (e.g., quote-level citations `{{cite:<chunk_id>:<start>:<end>}}` per Feature 003 ADR-007's extension path) are additive — the regex matched inside `marker_pending` grows; everything else stays the same.

### Negative

- The parser maintains state across `useChat` token events; bugs in state transitions can swallow text or emit duplicate badges.
- Test surface: the parser needs targeted unit tests covering all state transitions (open marker, partial marker, complete marker, malformed marker, marker with garbage inside).
- The 60-char buffer-bailout heuristic is a magic number; documented but tuned during dogfooding if the LLM produces anomalous output.

### Risks

- **Risk**: Mastra's streaming response shape (deltas, tool-use blocks, full message events) differs from a flat token stream — the parser must handle the actual event types from `@assistant-ui/react-ai-sdk` adapter.
  **Mitigation**: parser wraps the adapter's text-only event stream; tool-use events and metadata pass through unchanged. TechSpec's task_03 verifies the actual event types at the pinned version.
- **Risk**: a marker spans more SSE chunks than expected (e.g., LLM tokens break the UUID into 4-char fragments).
  **Mitigation**: the buffer accommodates this naturally; the 60-char bailout is comfortably above any reasonable marker length (`{{cite:<36-char-uuid>}}` = 46 chars).
- **Risk**: cancellation mid-marker (user clicks "cancel" while parser is in `marker_pending`).
  **Mitigation**: on cancel, parser flushes its buffer as literal text (the partial marker shows as raw text in the cancelled message — acceptable since the cancelled state is non-final).

## Implementation Notes

- Parser lives in `apps/web/src/lib/citation-parser.ts` as a pure function `parseStream(deltaText: string, state: ParserState): { tokens: Token[]; nextState: ParserState }`.
- Token types: `{ kind: 'text', text: string }` or `{ kind: 'citation', chunkId: string }`.
- Renderer maps `text` → React text node, `citation` → `<CitationBadge>`.
- Citation regex inside `marker_pending` reuses `CITATION_MARKER_REGEX` from `@dialogus/rag` (Feature 003 task_01 export).
- Unit tests in `apps/web/__tests__/lib/citation-parser.test.ts` cover: split-marker across two deltas, multiple markers in one delta, malformed marker, marker with leading whitespace inside braces (rejected), 60-char bailout.
