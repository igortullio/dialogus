# ADR-008: Chapter-summary generation as a seventh ingestion stage

## Status

Accepted

## Date

2026-04-24

## Context

Feature 003 (RAG Agent) needs a `get_chapter_summary` tool that returns a bounded, LLM-crafted summary per chapter. Three implementation paths for the summary existed: (A) precompute during ingestion; (B) on-demand at tool-call time; (C) lazy precompute on first tool call, cached thereafter. Feature 003 chose (A) after analyzing cost, latency, and truncation edge cases on long chapters (Tolstoy, Hugo, Melville produce chapters easily over 15k tokens).

Feature 003 ADR-001 then assigned *ownership* of summary generation to Feature 002, keeping `@dialogus/rag` read-only. Feature 003 ADR-005 specified a dedicated `chapter_summaries` table (1:1 with chapters).

This ADR captures Feature 002's side of the contract: how the new summary-generation step slots into the existing six-stage pipeline (download â†’ clean â†’ parse â†’ chunk â†’ embed â†’ index) without disturbing its serial, streaming, resumable architecture.

## Decision

**Add a seventh pipeline stage â€” `summarize` â€” positioned between `chunk` and `embed`.**

New pipeline order:

```
download â†’ clean â†’ parse â†’ chunk â†’ summarize â†’ embed â†’ index
```

Position rationale:

- `summarize` depends on `chapters.plain_text` being persisted (produced by `parse`); not on `chunks`. Placing it after `chunk` is a convenience â€” keeps the existing serial chain readable.
- Placing it **before** `embed` means a failing Anthropic call (the new, less-tested integration) never wastes OpenAI embedding tokens. Resume from `summarize` is cheaper than resume from `embed`.
- Placing it before `index` is non-negotiable: `books.ingestion_status = 'ready'` must imply that `chapter_summaries` rows exist for every chapter, so the agent's `get_chapter_summary` tool never hits `null`.

New stage contract (matching the existing `StageHandler` shape from feature TechSpec):

- **Input**: `{ bookId }` via pg-boss job `ingestion.summarize`.
- **Output**: one `chapter_summaries` row per `chapters` row for the book; enqueues `ingestion.embed` on completion.
- **Idempotency**: re-runnable. Handler queries `chapters WHERE book_id = $1 AND id NOT IN (SELECT chapter_id FROM chapter_summaries WHERE book_id = $1)` â€” processes only chapters without existing summaries.
- **Failure semantics**: if Anthropic fails mid-book, written summaries persist; `books.ingestion_status` flips to `failed` with `ingestion_last_stage = 'summarize'`; `/retry` resumes from `summarize` and completes only the missing chapters.

Status enum extension: `books.ingestion_status` gains a new value `summarizing` between `chunking` and `embedding`. This keeps the UI-facing progress string honest and aligns with the per-stage pattern established by feature ADR-001.

## Alternatives Considered

### Alternative 1: Summarize after `embed`, before `index`

- **Description**: `download â†’ clean â†’ parse â†’ chunk â†’ embed â†’ summarize â†’ index`.
- **Pros**: `chunks` with embeddings are ready first; if we ever want to relax the "ready means summarized" invariant (Phase 2), the agent could theoretically answer without summaries while still retrieving.
- **Cons**: Anthropic failures now waste OpenAI spend on a book that never reaches `ready`; more total work redone on retry when summarize fails; no product upside in V1 because feature 003 ADR-005 makes summaries a hard prerequisite for `ready`.
- **Why rejected**: cost asymmetry wins â€” Anthropic is the less reliable dependency in this pipeline (rate limits, 429s during dogfood), so surface its failures before the pricier embed step.

### Alternative 2: Summarize as a parallel branch (fan out after `chunk`)

- **Description**: After `chunk`, enqueue `ingestion.summarize` AND `ingestion.embed` in parallel; join before `index` via a barrier job.
- **Pros**: total wall-clock minimized for a book; stages independent.
- **Cons**: breaks the serial-chain invariant (feature ADR-002 "serial ingestion"); barrier coordination in pg-boss is verbose; failure-mode matrix grows; no meaningful latency gain in single-user V1 dogfooding (one book at a time, stages take seconds each).
- **Why rejected**: feature ADR-002 commits to serial. Parallelism contradicts a load-bearing architectural decision for V1's sanity.

### Alternative 3: Summaries computed on-demand at query time (no pipeline stage)

- **Description**: `get_chapter_summary` in feature 003 calls Haiku at tool-call time, caches result in `chapter_summaries` after first call.
- **Pros**: no ingestion coupling; feature 002 stays sealed.
- **Cons**: first user question about a chapter eats ~5-8 second Haiku latency; misses feature 003's "3-second first token" goal on summary questions; adds retry/caching complexity inside the agent package; reverses the explicit decision made during feature 003 PRD clarification round.
- **Why rejected**: already rejected by feature 003 PRD (option C was offered and declined). Recorded here only for completeness.

### Alternative 4: Store summaries as columns on `chapters` (no new table)

- **Description**: Add `summary text`, `summary_generated_at timestamptz` to `chapters`.
- **Pros**: no new table; one fewer join.
- **Cons**: mixes an immutable parse artifact (`plain_text`) with a regenerable LLM artifact (`summary`) on the same row; row-churn on summary regeneration touches the hot `chapters` table.
- **Why rejected**: feature 003 ADR-005 covers this comparison in depth and chose the dedicated table. Listed here so that feature 002's ADR trail makes the schema choice traceable inside 002 too.

## Consequences

### Positive

- New stage is a mechanical addition to the existing pipeline â€” same `StageHandler` shape, same idempotency discipline, same resume semantics.
- Resume from `summarize` is cheap; failure cost is bounded to Anthropic spend for the chapters already summarized in the failed run.
- `ready` stays the single authoritative signal that a book is agent-consumable (no "ready but partially summarized" edge state).
- `chapter_summaries` rows can be regenerated independently via a `pnpm books:resummarize <book_id>` script (deferred to Phase 2) without re-running the full pipeline.

### Negative

- Ingestion now requires `ANTHROPIC_API_KEY` at runtime. Feature 002 originally needed only `OPENAI_API_KEY`; the scope now includes a second LLM vendor dependency.
- Total ingestion wall-clock increases by the summarize stage duration (estimated 5-15 seconds per chapter via Haiku; e.g., War and Peace at 365 chapters â‰ˆ 30-90 minutes of summarize work alone). Serial execution makes the cost visible.
- `books.ingestion_status` enum churns (adds `summarizing`), which affects catalog/library endpoints that surface the status to the UI. Change is additive; no existing consumers break, but Zod enums need updating.
- Feature 002 integration test suite grows by one stage-level test; full-pipeline integration test now includes a summarize leg.

### Risks

- **Risk**: Anthropic rate limits during ingestion of a large book with many chapters (e.g., *War and Peace*, 365 chapters). Back-to-back Haiku calls could hit tier-1 RPM limits.
  **Mitigation**: `bottleneck` rate limiter (already in ingestion for Gutendex) extended to Anthropic calls in the summarize stage; default 30 RPM with jitter.
- **Risk**: bilingual summaries â€” a PT book summarized in EN (or vice versa) would look wrong in the agent's response.
  **Mitigation**: summarize-stage prompt reads `books.languages[0]` and instructs Haiku to summarize in that language; validated during integration smoke with one EN + one PT fixture.
- **Risk**: summary quality drift if the summarize prompt is poorly tuned.
  **Mitigation**: prompt lives as a committed Markdown asset in `@dialogus/ingestion/src/prompts/summarize.md`; snapshot-tested for size; the owner's dogfooding round exercises summary-dependent journeys and can iterate the prompt as a single-file change.
- **Risk**: partial failures leave orphan `chapter_summaries` rows if a book is soft-deleted mid-ingestion.
  **Mitigation**: `on delete cascade` on `chapter_summaries.book_id` handles both soft-delete (if Phase 2 turns it into hard-delete) and direct deletion.

## Implementation Notes

- New Drizzle table `chapter_summaries` per feature 003 ADR-005. Migration `drizzle/0004_chapter_summaries.sql`, generated by `drizzle-kit` (no hand-editing).
- New ports in `@dialogus/ingestion`:
  - `ChapterSummaryRepository` (write-side, symmetric to `ChapterRepository` / `ChunkRepository`).
  - `ChapterSummaryGenerator` (LLM-agnostic port wrapping `generate(chapter: ParsedChapter, language: 'en' | 'pt'): Promise<{ summary: string; tokenCount: number; model: string }>`).
- New adapters:
  - `DrizzleChapterSummaryRepository` in `infrastructure/persistence/`.
  - `AnthropicChapterSummaryGenerator` in `infrastructure/external/` â€” wraps `@ai-sdk/anthropic` with `claude-haiku-4-5`; rate-limited via `bottleneck`.
  - `MockChapterSummaryGenerator` for tests â€” deterministic short summary derived from chapter title.
- New stage handler `application/stages/summarize.ts`. Reads chapters without summaries; generates via `ChapterSummaryGenerator`; writes via `ChapterSummaryRepository`; enqueues `ingestion.embed` on completion.
- Summary prompt asset `infrastructure/prompts/summarize.md` â€” Markdown-authored; covers: scholarly academic tone, preserve source language, 150-300 words per chapter, no spoilers of later chapters (self-contained to the chapter being summarized).
- `.env.example` for `apps/worker` adds `ANTHROPIC_API_KEY` reference (already planned globally; ensures worker boot validates presence).
- Existing chunk-stage handler (task_12) updated: enqueue `ingestion.summarize` (not `ingestion.embed`) as the next step.
- Existing embed-stage handler (task_13) is reached from summarize's completion; no semantic change to embed itself.
- Extended `IngestionStatus` enum in `@dialogus/shared/schemas/ingestion` adds `'summarizing'` between `'chunking'` and `'embedding'`.
- New problem slug `ingestion-summarize-failed` (503, retryable) added to `problem.ts` middleware.
