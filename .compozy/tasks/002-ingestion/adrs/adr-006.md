# ADR-006: Chapter heuristics in a YAML data file

## Status

Accepted

## Date

2026-04-24

## Context

The parse stage detects chapters in plain-text Gutenberg files via language-specific regex patterns. English chapters use patterns like `/^CHAPTER [IVX]+/i`; Portuguese use `/^Cap[íi]tulo \d+/` and `/^PARTE \w+/`. The PRD flagged this as an open question because dogfooding will surface new patterns — a reader who finds Pessoa poetry or an unusual translation will hit heuristics the V1 registry didn't cover.

Three storage options were considered:

- **YAML data file** (`chapter-heuristics.yaml`) Zod-validated at boot.
- **TypeScript constant** (`chapter-heuristics.ts`) with `as const` typing.
- **DB table** (`chapter_heuristics`) mutable at runtime.

## Decision

Chapter heuristics live in a **YAML data file** at `packages/ingestion/src/infrastructure/parsing/chapter-heuristics.yaml`. Schema:

```yaml
en:
  patterns:
    - '^CHAPTER\s+[IVXLCDM]+\.?\s*$'
    - '^Chapter\s+\d+\.?\s*$'
    - '^CHAPTER\s+\d+'
  fallback_title: 'Full text'
pt:
  patterns:
    - '^CAP[ÍI]TULO\s+\d+\.?\s*$'
    - '^Cap[íi]tulo\s+\d+\.?\s*$'
    - '^CAP[ÍI]TULO\s+[IVXLCDM]+\.?\s*$'
    - '^PARTE\s+\w+\.?\s*$'
  fallback_title: 'Texto completo'
```

Loader:

```typescript
const schema = z.object({
  en: z.object({
    patterns: z.array(z.string()),
    fallback_title: z.string(),
  }),
  pt: z.object({
    patterns: z.array(z.string()),
    fallback_title: z.string(),
  }),
})

const raw = yaml.parse(readFileSync(resolve(__dirname, 'chapter-heuristics.yaml'), 'utf8'))
const config = schema.parse(raw)
const compiled = {
  en: { patterns: config.en.patterns.map(p => new RegExp(p)), fallbackTitle: config.en.fallback_title },
  pt: { patterns: config.pt.patterns.map(p => new RegExp(p)), fallbackTitle: config.pt.fallback_title },
}
```

Loaded once at process boot (inside `TxtChapterParser` constructor or module-level). Patterns are evaluated in order; first match wins.

## Alternatives Considered

### Alternative 1: TypeScript constant

- **Description**: `chapter-heuristics.ts` exports `CHAPTER_HEURISTICS = { en: { patterns: [/.../g], ... }, pt: {...} } as const`.
- **Pros**: type-safe at import; no parser; RegExp objects directly.
- **Cons**: adding a pattern requires editing TypeScript + recompile. Dogfooding friction: "I just saw a new chapter pattern — let me restart the whole rebuild pipeline". For a heuristics registry that is expected to grow organically, a data file is more inviting than source code.
- **Why rejected**: data-as-data beats data-as-code for a registry; YAML is easier to audit in a diff review.

### Alternative 2: DB table `chapter_heuristics`

- **Description**: runtime-mutable table; admin endpoint or seed script populates; parser reads on each book.
- **Pros**: true runtime mutability; could be multi-tenant Phase 3.
- **Cons**: adds a table, a seeding ceremony, and an admin surface for what is fundamentally a static config. Tests need DB setup to exercise the parser.
- **Why rejected**: over-engineering; YAML + restart is indistinguishable from DB-backed for a single-user project where restart happens every dev iteration anyway.

## Consequences

### Positive

- A new chapter pattern discovered during dogfooding is a one-line YAML edit + restart — no recompile, no migration, no seed script.
- YAML diffs are legible in PRs ("Added `'^Volume\s+\w+'` to PT patterns for Tolstoy translation" reads cleanly).
- The file is committed + code-reviewable + discoverable (no hunt through code to find the registry).
- Zod validation catches typos at boot with grouped error messages.

### Negative

- YAML parser + Zod validation add ~20 lines of boilerplate vs. a static TS constant.
- RegExp compiled from string loses line-number context on error (less of a concern for typical patterns).
- YAML "gotchas" (special characters, escaping) can produce confusing errors; documentation in the file header mitigates.

### Risks

- **Risk**: YAML parse error at boot crashes `apps/worker` until fixed.
  **Mitigation**: Zod validation runs at boot with descriptive grouped errors; CI enforces that the YAML parses successfully in every pipeline.
- **Risk**: a bad regex (e.g., catastrophic backtracking) is added and slows parse stage dramatically.
  **Mitigation**: unit test exercises all patterns against the 6 reference books (3 EN + 3 PT); regression test catches pathological patterns before merge.

## Implementation Notes

- File location: `packages/ingestion/src/infrastructure/parsing/chapter-heuristics.yaml`.
- File header includes a comment block documenting pattern conventions: `i` flag added automatically by loader; anchoring uses `^` + `\s*$` by convention.
- Loader caches the compiled config in a module-level variable; called once per process lifetime.
- When a new book fails parsing, the recommended workflow is: (1) inspect the raw text near the missing chapter marker, (2) craft a new pattern, (3) add to YAML, (4) run unit tests against the fixture book, (5) commit.
