# ADR-004: In-memory LRU cache for Gutendex responses

## Status

Accepted

## Date

2026-04-24

## Context

The product TechSpec mandates a 60-second cache on Gutendex search responses to soften upstream flakiness and reduce latency on repeated queries during dogfooding. Three implementations were considered:

- **In-memory `lru-cache`** living in the `apps/api` process.
- **Postgres-backed cache table** — `gutendex_cache(key TEXT PK, body JSONB, expires_at TIMESTAMP)`.
- **Redis** — new container in `docker-compose.yml`, dedicated cache layer.

For single-user local dev + dogfooding + portfolio-grade demo, the cache's purpose is strictly latency / availability smoothing, not global consistency or cross-process sharing.

## Decision

Use a **single in-memory `lru-cache` instance** per `apps/api` process:

- Library: `lru-cache@^11` (current major; API-compatible with v10+).
- Options: `{ max: 500, ttl: 60_000 }` — 500 entries, 60-second TTL.
- Cache key: normalized URL path + query-string (sort keys alphabetically, strip irrelevant headers).
- Cache value: the already-mapped `Book[]` DTOs or single `Book`, not the raw Gutendex response (avoids re-mapping on hit).
- Exposed via `GutendexHttpClient` inside `@dialogus/catalog/infrastructure/external/`; not directly accessible from routes.
- Zero external infrastructure (no Redis container, no Postgres table).

## Alternatives Considered

### Alternative 1: Postgres-backed cache table

- **Description**: a `gutendex_cache(key, body, expires_at)` table; handler writes on miss, reads on hit.
- **Pros**: survives restart; visible via psql for debugging; shared across hypothetical multi-instance deploy.
- **Cons**: adds a migration, a cleanup concern, and two DB round-trips per search request. For a 60-second TTL that would otherwise cost a single Gutendex call, this is over-engineered. Gutendex is not critically slow (~300ms); caching it to Postgres trades one I/O for another.
- **Why rejected**: cost outweighs benefit for the use case.

### Alternative 2: Redis

- **Description**: docker-compose adds a `redis` service; cache client connects to it.
- **Pros**: production-grade; shared across instances; rich TTL primitives.
- **Cons**: introduces new infra to manage (image, port, volume); violates "Postgres-only for infra" convention from product TechSpec; single-user local dev doesn't benefit.
- **Why rejected**: over-engineering; brings complexity with no user-visible payoff.

## Consequences

### Positive

- Zero infrastructure additions — one npm package, one file.
- Cold cache for 60 seconds after restart is acceptable given Gutendex's public rate and owner's typical session behavior.
- Cache lives inside the adapter (`GutendexHttpClient`), preserving hexagonal boundaries; use cases don't know a cache exists.
- Simple to unit-test: inject a mocked cache-like object in tests.

### Negative

- Loses all entries on restart. In a restart-heavy dev loop, the first call after restart pays the upstream cost. At Gutendex's typical 300 ms response time, this is a non-issue.
- In a future multi-instance deploy (Phase 3), cache is per-instance — multiple hits may still cross the Gutendex boundary. Acceptable; Phase 3 revisits if warranted.

### Risks

- **Risk**: memory pressure if max size is set too high. `500 entries × ~5 KB average` = 2.5 MB RSS overhead; benign.
- **Risk**: stale Gutendex data cached during a 60-second window right after Gutendex updates. Negligible — book catalogs don't change hour-by-hour.
- **Risk**: cache key collision if normalization is wrong (e.g., query-order inconsistency). **Mitigation**: canonicalize key via helper `cacheKey(path, params)` that sorts params alphabetically.

## Implementation Notes

- `GutendexHttpClient` constructs its own `LRUCache` instance in its constructor; factory function wires it up during dependency injection.
- Cache invalidation is TTL-only; no manual `cache.clear()` exposed in V1 (dev can restart the process).
- Log line emitted on each call: `{ gutendex_cache: 'hit' | 'miss', key }`.
