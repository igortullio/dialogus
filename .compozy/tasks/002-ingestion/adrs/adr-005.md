# ADR-005: apps/worker as sole pg-boss worker

## Status

Accepted

## Date

2026-04-24

## Context

In Feature 001 (Catalog), `apps/api` became the only process with a long-running pg-boss instance, hosting the `catalog.cleanup-idempotency-keys` scheduled job. That was a pragmatic choice because `apps/worker` did not yet exist. Feature 002 introduces `apps/worker` as a dedicated background process, creating the question of which process owns pg-boss workers going forward.

Three options were considered:

- **`apps/worker` as the sole worker** — all long-running pg-boss `work()` and `schedule()` calls live in `apps/worker`. `apps/api` uses transient pg-boss instances for `send()` only (create, send, stop). The catalog cleanup job handler migrates from `apps/api` to `apps/worker`.
- **Split** — `apps/api` keeps catalog cleanup; `apps/worker` handles only ingestion queues. Two processes host long-running pg-boss instances.
- **Both host all workers** — redundant registration; pg-boss load-balances.

## Decision

**`apps/worker` is the sole long-running pg-boss worker process.** All consumer registrations (`boss.work(...)` and `boss.schedule(...)`) move to or live in `apps/worker`. This includes:

- All six `ingestion.<stage>` queues.
- The `catalog.cleanup-idempotency-keys` hourly job (migrated from `apps/api`).
- Any future queues added by Features 003 or 004.

`apps/api` becomes purely HTTP request-handling. When it needs to enqueue a job, it creates a transient pg-boss instance, calls `send()`, and stops:

```typescript
// apps/api route handler
const boss = createPgBoss(DATABASE_URL)
await boss.start()
await boss.send('ingestion.download', { bookId })
await boss.stop()
```

`apps/api/src/jobs/cleanup-idempotency-keys.ts` (from Feature 001 task_15) is **deleted**; the handler code moves to `apps/worker/src/handlers/catalog-cleanup-idempotency-keys.ts`. Feature 001 task_15 reference in this TechSpec is noted as retroactively modified by Feature 002.

## Alternatives Considered

### Alternative 1: Split ownership

- **Description**: `apps/api` keeps the catalog cleanup worker (as-is from Feature 001); `apps/worker` adds only ingestion queues. Two long-running pg-boss instances, one per app.
- **Pros**: no retrofit of Feature 001; status-quo preservation.
- **Cons**: two processes each hold a live pg-boss client; mental model split ("which app runs this job?"); per-process resource footprint doubled; in a multi-instance deploy (Phase 3), coordination becomes awkward.
- **Why rejected**: now that `apps/worker` exists, splitting responsibilities by process type (HTTP vs. background) is cleaner than splitting by feature accident.

### Alternative 2: Both host all workers

- **Description**: `apps/api` AND `apps/worker` both register `boss.work(...)` for the same queues. pg-boss naturally distributes jobs via its FIFO + leader election.
- **Pros**: fault tolerance (if one process crashes, the other picks up).
- **Cons**: single-user local project doesn't need HA; two processes competing for the same job creates debugging complexity; ADR-002 (serial ingestion) assumes single-worker concurrency — two processes would require additional cross-process coordination.
- **Why rejected**: overkill for V1; violates ADR-002 implicitly.

## Consequences

### Positive

- Clear single-responsibility: `apps/api` = HTTP, `apps/worker` = jobs.
- `apps/api` boot path simplifies (no `boss.start()` at boot; no graceful-shutdown coordination for long-running workers).
- Feature 003 and 004 inherit the convention automatically — any new job queue lands in `apps/worker`.
- ADR-002 serial concurrency is trivially enforced (one worker process, `teamConcurrency: 1` per queue).
- Deployment separation possible in Phase 3 (different container, different scaling).

### Negative

- Retrofit to Feature 001 task_15: code change required. Documented here; the code change happens as part of Feature 002 Build Order step 2 (not a separate task).
- `apps/api` enqueue path creates a transient pg-boss client per request (small per-call cost; pg-boss connection pool handles it; measured at < 10ms in testing).
- Developer running `apps/api` alone without `apps/worker` sees enqueued jobs stuck in "created" state indefinitely — potentially confusing.

### Risks

- **Risk**: developer forgets to start `apps/worker` during local dev; ingestion looks broken.
  **Mitigation**: root `pnpm dev` starts both in parallel; README explicit about the two-process requirement; `apps/api /health` could optionally report a warning if no worker has processed any job recently (Phase 2 observability).
- **Risk**: transient pg-boss client overhead adds latency to `/ingest`.
  **Mitigation**: typical `start()+send()+stop()` cycle is 10-20ms against a hot Postgres; for a 202-Accepted endpoint, invisible to users.

## Implementation Notes

- `apps/worker/src/index.ts` pseudocode:
  ```typescript
  import { loadConfig } from '@dialogus/shared/config'
  import { createPgBoss } from '@dialogus/db'
  // import all handlers
  const cfg = loadConfig()
  const boss = createPgBoss(cfg.DATABASE_URL)
  await boss.start()
  await boss.work('ingestion.download', { teamConcurrency: 1 }, downloadHandler)
  // ...repeat for each ingestion stage
  await boss.work('catalog.cleanup-idempotency-keys', cleanupHandler)
  await boss.schedule('catalog.cleanup-idempotency-keys', '0 * * * *', {})
  process.on('SIGTERM', async () => { await boss.stop(); process.exit(0) })
  ```
- `apps/api` enqueue helper (shared across routes):
  ```typescript
  export async function enqueue<T>(queue: string, data: T): Promise<string> {
    const boss = createPgBoss(cfg.DATABASE_URL)
    await boss.start()
    const jobId = await boss.send(queue, data)
    await boss.stop()
    return jobId
  }
  ```
- Feature 001 task_15 is marked "superseded in step 2 of Feature 002 Build Order" in its header. The catalog TechSpec may later add a cross-reference for clarity.
