# ADR-004: Thread management — full CRUD + pin in V1

## Status

Accepted

## Date

2026-04-24

## Context

The chat-first landing page lists threads in a sidebar (product PRD § User Experience). What operations the UI exposes on those threads — create, delete, rename, pin, reorder — is a product-shape decision: the minimum is "create + switch", and each addition is one more UI affordance to design and one more API call to wire.

Mastra Memory (per product ADR-006) owns thread persistence; Mastra exposes thread list + thread create + thread delete via its native API. Rename and pin are Mastra-Memory features at the pinned version (or implementable as Mastra metadata fields) — verified during PRD clarification.

## Decision

**V1 ships full CRUD on threads plus pin-to-top.** Specifically:

- **Create**: "Nova conversa" button in the sidebar header; opens a fresh composer with empty book picker. Thread is created lazily on first message send (no empty threads) — Mastra Memory creates the thread when the first message arrives.
- **Switch**: clicking a thread in the sidebar opens it in the main view.
- **Delete**: hover-action on a thread row (or three-dot menu); shows a `<AlertDialog>` confirmation with thread title; on confirm, calls Mastra's delete-thread API + clears localStorage spoiler-cap entries (per ADR-002 implementation notes).
- **Rename**: thread title defaults to the first user message truncated to ~40 chars; the user can edit via the same three-dot menu → "Renomear"; opens an inline `<Input>` overlaying the row title; saves on Enter / blur.
- **Pin**: pinned threads sort to the top of the sidebar in their own group ("Fixadas"); unpinned threads below in "Recentes" (most-recent-first by default); toggle via three-dot menu → "Fixar" / "Desafixar". Pin state is a thread metadata field on Mastra Memory.

## Alternatives Considered

### Alternative: Minimal — create + switch only

- **Description**: No delete, no rename, no pin. Threads accumulate; sort by recency.
- **Pros**: smallest UI surface; one less menu component; covers the absolute basic flow.
- **Cons**: dogfooding accumulates garbage threads (test runs, exploratory prompts); without delete, the sidebar becomes a wall of "perguntinha de teste" within a week; rename and pin are ergonomic upgrades the owner will resent missing within a week of dogfooding.
- **Why rejected**: dogfooding sustainability (a Primary Success Metric in product PRD) is the gate this UI must clear; sidebar pollution is a real friction.

### Alternative: Medium — create + switch + delete (no rename, no pin)

- **Description**: Adds delete with confirmation; defers rename/pin to Phase 2.
- **Pros**: covers the worst pollution case (delete); ~1 task less than full CRUD.
- **Cons**: no rename means the sidebar is a list of "first-question fragments" — useful but ugly when conversations evolve; no pin means "what I'm currently reading" sinks below test threads as soon as I run a smoke test.
- **Why rejected**: the additional surface for rename + pin is small (one shared three-dot menu), and the dogfooding ergonomic gain is concrete.

### Alternative: Implicit creation — no explicit "Nova conversa" button

- **Description**: ChatGPT-style: any composer interaction with a different book scope auto-creates a new thread; sidebar shows the auto-created threads.
- **Pros**: minimalist; one fewer button.
- **Cons**: the user can't intentionally start a new thread without first interacting with a book picker; the sidebar has less affordance for "I want to start fresh"; the `book_ids` change-detection logic gets murky.
- **Why rejected**: the explicit button is a small cost with clearer mental model.

## Consequences

### Positive

- Sidebar stays curated through dogfooding — pin "current reading", delete test runs, rename meaningful threads.
- Three-dot menu is shadcn `<DropdownMenu>` — one component shared across rename/delete/pin.
- "Fixadas" + "Recentes" grouping is a visible structural cue; the owner can scan the sidebar at a glance.
- Mastra Memory's native thread metadata supports rename + pin without a custom backend table.

### Negative

- More UI surface to design + test (rename overlay, three-dot menu, pin sort logic, confirmation dialog).
- Mastra Memory's pinning + rename APIs may differ between minor versions — TechSpec pins exact versions, but a future Mastra upgrade may need adapter shims.
- Empty-state cleanup logic (delete a thread → also remove its localStorage spoiler caps) introduces cross-system coordination.

### Risks

- **Risk**: Mastra at the pinned version doesn't support per-thread metadata (rename + pin) natively.
  **Mitigation**: TechSpec verifies before authoring this functionality. If unavailable, the fallback is a small `thread_metadata(thread_id, title, pinned, updated_at)` table in `@dialogus/db` and two endpoints in `apps/api`. Documented as a contingency, not a hard plan.
- **Risk**: rename overlay competes with the active thread highlight; styling regression.
  **Mitigation**: TechSpec component design covers focus state.
- **Risk**: a thread with hundreds of messages takes long to delete (Mastra cascade).
  **Mitigation**: optimistic UI — remove from sidebar immediately, show a "desfazendo..." toast if the API errors; in practice deletes are fast.

## Implementation Notes

- The three-dot menu lives on hover (desktop) or always-visible on touch (mobile). Use `<DropdownMenu>` with three items: "Renomear", "Fixar"/"Desafixar", "Excluir".
- Delete confirmation: `<AlertDialog>` with thread title in the description. Destructive action color (red). No "are you sure" if pinned (still confirm — pin is not protection).
- Rename: clicking "Renomear" replaces the title text with an `<input type="text">` styled to match; auto-focus + select-all; Enter confirms, Esc cancels, blur confirms.
- Pin: toggling pin re-sorts the sidebar via TanStack Query optimistic update + `revalidate` on success.
- Spoiler-cap cleanup on delete: enumerate localStorage keys matching `dialogus:spoiler_cap:<thread_id>:*` and remove. Done synchronously before the API call so a failed delete doesn't leave orphan keys.
- Empty sidebar (no threads): show "Primeiros passos" card per ADR-001 (PRD-level decision).
