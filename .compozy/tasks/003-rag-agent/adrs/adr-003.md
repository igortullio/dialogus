# ADR-003: Refusal with reformulation hints on empty retrieval

## Status

Accepted

## Date

2026-04-24

## Context

The product PRD commits the agent to "refuse rather than hallucinate when retrieval returns nothing usable." This is a trust contract with the user — the agent's single most important behavioral promise. The PRD does not specify the *shape* of the refusal: bare "not found" vs. apologetic + helpful vs. retry-and-recover.

Two failure modes exist simultaneously. Hallucination — answering without retrieval support — breaks the grounding promise. Dead-end refusal — saying only "não encontrei" — kills the conversation and trains the user to stop asking hard questions.

## Decision

**On empty or irrelevant retrieval, the agent abstains from answering the substantive question and offers 2–3 concrete reformulation hints.**

- The agent calls `semantic_search` first. If the tool returns zero chunks, or if all returned chunks score below a relevance floor (threshold resolved in 003 TechSpec), the agent does not synthesize an answer.
- The agent produces a short refusal message in the user's message language (per ADR-002), explicitly naming that no relevant passage was found.
- The agent then offers 2–3 reformulation suggestions grounded in what *is* available: chapter titles from `list_chapters`, character names visible in top-scoring-but-rejected chunks, alternative phrasings of the user's intent.
- The agent does NOT automatically retry with a rewritten query. A second retrieval pass is a V2 decision (see ADR-004 on reranking and retrieval strategy).

Example reply (PT, Moby Dick thread, user asks about Pequod's carpenter):

> Não encontrei passagens relevantes sobre esse tema em *Moby Dick*. Você poderia tentar:
> - Perguntar pelo personagem usando o nome em inglês (p. ex. "carpenter of the Pequod").
> - Explorar o capítulo 107 ("The Carpenter"), que aparece na lista de capítulos.
> - Reformular a pergunta focando em uma cena específica.

## Alternatives Considered

### Alternative: Strict refusal, no hints

- **Description**: "Não encontrei passagens relevantes." Period. No suggestions, no guidance.
- **Pros**: unambiguous; no risk of the refusal itself looking like a hedged answer; minimal prompt complexity.
- **Cons**: dead-end UX; trains the user to stop at first "no"; wastes the context already loaded (chapter list, rejected chunks' metadata).
- **Why rejected**: too abrupt; erodes dogfooding stamina without a corresponding trust benefit.

### Alternative: Best-effort + disclaimer

- **Description**: Agent attempts an answer from the model's pre-training knowledge, clearly labeled "(sem citação disponível)" or "(resposta sem passagem encontrada)".
- **Pros**: higher apparent utility; user gets something.
- **Cons**: directly weakens the grounding promise; even with a disclaimer, the answer still enters the user's mind; confuses the "grounded" framing; one of the exact failure modes the product PRD calls out.
- **Why rejected**: breaks the trust contract the product is built on.

### Alternative: Silent retry with query rewrite

- **Description**: Agent rewrites the user's query (e.g., synonyms, alternative phrasings) and calls `semantic_search` a second time. Only abstains if the second pass is also empty.
- **Pros**: recovers marginal cases without bothering the user; may feel "smarter."
- **Cons**: opaque — the user cannot tell why the second attempt worked or what the rewritten query was; doubles retrieval cost per turn; delays the abstention; risks the agent rewriting itself into a relevant-but-unrelated topic (false positive).
- **Why rejected**: adds complexity with ambiguous user-visible benefit; reformulation hints keep the user in control of the rewrite.

## Consequences

### Positive

- Honors the product's grounding contract without being abrasive.
- Turns a failed question into a continued conversation, preserving dogfooding momentum.
- Reformulation hints are concrete and grounded in available data (chapter titles, character mentions) — not hallucinated.

### Negative

- System prompt is longer (must teach the refusal-plus-hint shape); small prompt-complexity and prompt-caching tax.
- The hint-generation step runs on the turn that is *already* failing to retrieve — a second LLM pass on the same turn. In most cases this is the same generation that produces the refusal prose, so no extra API call.
- Defining "empty or irrelevant retrieval" requires a threshold (either zero results or a score floor). That detail is a TechSpec decision, not a PRD decision.

### Risks

- **Risk**: the agent produces reformulation hints that reference characters or themes not actually in the book, falling back into hallucination.
  **Mitigation**: hints are generated from the output of `list_chapters` and the rejected-chunks metadata — both are grounded in real persisted data. System prompt explicitly says "suggest reformulations using only the chapters and names that appeared in tool results."
- **Risk**: threshold tuning — a too-strict threshold makes the agent refuse too often; too loose and bad chunks drive bad answers.
  **Mitigation**: start permissive (any non-empty tool result = attempt answer), tighten only if dogfooding reveals hallucinations from weak chunks.

## Implementation Notes

- System prompt structure (in `@dialogus/rag/src/prompts/system.md`):
  1. Identity + scholarly posture (ADR-002 inheritance from product).
  2. Grounding contract: always call `semantic_search` before substantive answers.
  3. Citation format (excerpt markers inline).
  4. Refusal + reformulation template — this ADR.
  5. Language-match rule (ADR-002).
  6. Spoiler-cap reinforcement.
- Agent logs (thread_id, tool_name, chunks_returned, action) surface in Mastra Studio; the owner can inspect when refusals fire.
- The "relevance floor" threshold (if any) is a TechSpec decision: zero-result refusal vs. similarity-score-below-X refusal. V1 defaults to zero-result refusal; tighter thresholds deferred.
