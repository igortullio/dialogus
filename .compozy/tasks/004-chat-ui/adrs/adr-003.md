# ADR-003: Citation UX — superscript badge + hover preview + click-for-side-panel

## Status

Accepted

## Date

2026-04-24

## Context

Feature 003 ADR-007 fixed the inline citation marker emitted by the agent: `{{cite:<chunk_id>}}` where `<chunk_id>` is a UUID v4. The Chat UI must transform that marker into something visually integrated with the agent's prose. Industry conventions in 2026 give four well-known patterns: (1) superscript badge with hover preview + click-to-expand, in the NotebookLM / Perplexity / scholarly papers style; (2) bold inline badge that opens a side panel on click only, no hover; (3) numbered footnote-style with a sources block at the end of the response; (4) inline expandable excerpt blocks where each citation expands in place.

Citation UX is the most user-visible expression of the "grounded" promise. The pattern affects readability, trust signal, and discoverability of source verification.

## Decision

**Citation markers render as small superscript badges + hover preview tooltip + click-for-side-panel.**

Behavior:

1. **Marker substitution**: every `{{cite:<chunk_id>}}` in the streaming response is replaced (during render, not in the source text) with a `<sup><CitationBadge id={chunk_id} /></sup>` component. Badges sit immediately after the claim they support, separated by zero spacing — tight visual coupling to the claim.
2. **Visual style**: the badge is a small numbered indicator (numbers re-counted per response, so first citation is `¹`, second `²`, etc., regardless of the chunk_id under the hood). Subtle background, neutral color, ≤ 16px tall. Accessible with `aria-label="Citação 1: capítulo 23 de Moby Dick"` populated from `tool_outputs[*].chunks[*]`.
3. **Hover preview** (300ms delay): displays a compact tooltip with `chapter_title` (book name in italic; chapter number + title) and `excerpt_preview` (≤ 200 chars). The preview is read-only; no interactive elements.
4. **Click**: opens a side panel anchored to the right edge of the screen (or below on narrow viewports). The panel fetches the full excerpt via `GET /api/library/chunks/<chunk_id>` (Feature 002 endpoint), shows full text + chapter context (1-2 surrounding chunks if available), with a close button; remains open until manually closed (does not close on outside click, to allow scrolling the agent's response while reading the source).
5. **Multi-citation per claim**: stacks badges with hairspace separator: `claim¹·²` reads naturally.
6. **Unknown chunk_id**: if the marker references a chunk_id not in the current message's `tool_outputs`, render a warning glyph (⚠) instead of a number; tooltip explains "citação não-resolvida" in user's message language; click opens an explanatory panel.

## Alternatives Considered

### Alternative 2: Click-only badge → side panel; no hover preview

- **Description**: Same badge visual; click always opens side panel; no tooltip layer.
- **Pros**: simpler — one interaction model; better mobile (no hover semantics on touch); fewer state machines.
- **Cons**: loses the "glance verification" pattern that makes citations feel low-friction; every check requires a click + close cycle; users in dogfooding will likely click 5+ times per response just to skim sources, which gets tiring.
- **Why rejected**: hover preview is the affordance that turns citations from a "verifiable but heavy" feature into a "ambient trust signal." The cost (one tooltip component) is small.

### Alternative 3: Footnote-style numbered citations + sources block

- **Description**: Inline `¹` `²` `³` markers with a "Fontes" block below the response listing each cited chunk with title + excerpt.
- **Pros**: scholarly aesthetic matches the product's "scholarly companion" positioning; works without JavaScript (plain Markdown rendering); good for printing/sharing transcripts.
- **Cons**: forces vertical scrolling for every source check (response → footnotes → back); numbered markers can drift if the agent emits them inconsistently (Feature 003 ADR-007 picked UUID-based markers exactly to avoid this drift); doubled real estate (inline marker + bottom block).
- **Why rejected**: real estate cost is concrete; the scholarly aesthetic is achievable inside the superscript-badge variant by using small serif numerals.

### Alternative 4: Inline expandable excerpt blocks

- **Description**: Each citation is a clickable inline block that expands the excerpt in place.
- **Pros**: zero side panel, zero tooltip — everything happens in the response stream; copy-paste of the response includes citations naturally.
- **Cons**: massive vertical real estate per response (5 citations × 200 chars expanded = ~1500 extra chars); breaks the prose flow; impossible to cross-reference two citations side-by-side; citation text invades the agent's own writing.
- **Why rejected**: harms readability; defeats the product's "grounded but readable" positioning.

## Consequences

### Positive

- The badge UX is the same pattern users learned in NotebookLM, Perplexity, and academic PDF readers — zero learning curve.
- Hover-then-click is the right depth-on-demand affordance: skim → preview → full context.
- Side panel keeps the agent's prose unbroken; citations live in their own column when needed.
- Numbering per-response (1, 2, 3 within each agent message) is more readable than UUID fragments and matches scholarly convention.
- The marker regex from Feature 003 ADR-007 is the only contract; the visual transformation is a Feature 004 internal concern. Future Feature 5 could change it without changing the agent.

### Negative

- The badge component is one of the most-touched UI atoms; styling regressions are easy to ship.
- Side panel adds layout complexity (responsive: side panel becomes a bottom sheet on narrow viewports?); resolved in TechSpec.
- Hover semantics on touch devices fall back to "tap once to preview, tap again to open panel" — the standard pattern but adds a state.
- Rendering performance: a long response with 20+ citations creates 20+ React components. Negligible at typical message size; flagged here only to track in Phase 2 if perf becomes an issue.

### Risks

- **Risk**: streaming responses produce citations one chunk at a time; the badge component must hydrate as `tool_outputs` arrive, which may lag the prose.
  **Mitigation**: while a citation chunk_id has not yet been resolved, render a placeholder badge (greyed-out number, `aria-busy`); replace with the live badge as soon as `tool_outputs` populate. The placeholder is a 1-line CSS state.
- **Risk**: the side panel competes with the thread sidebar for screen real estate on narrow viewports.
  **Mitigation**: TechSpec defines breakpoints; below 1024 px width, side panel becomes a full-width bottom sheet that pushes the keyboard.
- **Risk**: the `GET /api/library/chunks/:id` round-trip on every panel-open feels slow.
  **Mitigation**: per-response prefetch — when a response finishes streaming, preemptively fetch all referenced chunks and cache via TanStack Query; panel open is a cache hit.

## Implementation Notes

- Component: `<CitationBadge chunkId={...} index={...} thread={...}>` — wraps a `<sup>` and registers click + hover handlers.
- Hover delay: 300 ms, matches OS conventions.
- Side panel: use shadcn `<Sheet>` primitive (slide from right on desktop; bottom on mobile per `useMediaQuery`).
- Tooltip: shadcn `<Tooltip>` primitive (built on Radix).
- Numbering: a per-message context provider counts citations as they're encountered during render (post-stream-finish or via streaming-aware reducer); the ordinal is local to one message.
- Unknown chunk_id: a separate component `<UnresolvedCitationBadge>` with a distinct visual (⚠ glyph) keeps the resolution-status visible without breaking layout.
- Excerpt prefetch: `usePrefetchCitations(messageId)` reads all `tool_outputs[*].chunks[*].chunk_id` in a useEffect and `queryClient.prefetchQuery` each one.
