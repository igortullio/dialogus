# ADR-005: Tuple cursor `{created_at, id}` base64 JSON

## Status

Accepted

## Date

2026-04-24

## Context

Feature ADR-002 mandates cursor pagination on list endpoints. Three encodings were considered:

- **Tuple cursor** `base64(JSON.stringify({ createdAt, id }))` — carries the (created_at, id) of the last returned row; the next page filters `WHERE (created_at, id) < (decoded.created_at, decoded.id)`.
- **UUID-only cursor** — carries only `id`; next page filters `WHERE id < decoded_id`.
- **Self-contained cursor** — carries `{ createdAt, id, limit, direction }` so clients pass only an opaque token.

The `books` table is sorted `created_at DESC, id DESC`. `created_at` alone is not unique — two books inserted in the same ms would collide, causing dropped rows or duplicates. `id` alone has no temporal order (UUIDv4). The tuple (created_at, id) is both temporally meaningful and guarantees uniqueness.

## Decision

Cursors are **base64-encoded JSON of `{ createdAt, id }` tuples** taken from the last row of the current page.

- **Encoding**: `encodeCursor({ createdAt: Date, id: string }) → string` = `Buffer.from(JSON.stringify({ createdAt: createdAt.toISOString(), id })).toString('base64url')`.
- **Decoding**: `decodeCursor(cursor: string) → { createdAt: Date, id: string }` parses the inverse; throws `InvalidCursorError` → 400 Problem Details `invalid-cursor` on malformed input.
- **Filter**: `WHERE (created_at, id) < ($1, $2) ORDER BY created_at DESC, id DESC LIMIT $3`. The `(a, b) < (x, y)` tuple-compare is PostgreSQL standard and uses the partial index `books(created_at desc, id desc) where deleted_at is null`.
- **Opaque to clients**: clients store and resend cursors without inspecting the structure.
- **Shared utility**: codec lives in `@dialogus/shared/http/cursor.ts`; reused by Feature 002 (ingestion job list) and Feature 003 (thread list) without re-implementing.

**Lifetime**: no expiry in V1. A cursor is valid until the row it references is soft-deleted or the database is reset. Phase 3 (public deploy) revisits with HMAC signing + explicit expiry.

## Alternatives Considered

### Alternative 1: UUID-only cursor

- **Description**: `cursor = last_row.id`; query `WHERE id < decoded_id ORDER BY id DESC LIMIT N`.
- **Pros**: simplest possible encoding; no base64, no JSON.
- **Cons**: UUIDv4 has no temporal ordering — books paginated by `id DESC` would arrive in random order, not most-recent-first as the PRD specifies. Breaking the user-visible sort contract.
- **Why rejected**: violates PRD "sorted by `created_at DESC`" explicit requirement.

### Alternative 2: Self-contained cursor `{ createdAt, id, limit, direction }`

- **Description**: cursor carries pagination config (limit, direction) in addition to position; client only needs to pass the opaque token.
- **Pros**: cleaner client API — one opaque string replaces multiple query params.
- **Cons**: mixes configuration with navigation state. If a client's `limit=10` cursor is replayed in a context that expects `limit=20`, behavior is ambiguous. Standard practice is to keep limit/direction as explicit query params and cursor as pure position.
- **Why rejected**: adds ambiguity without meaningful gain for single-user dogfooding or portfolio demo.

## Consequences

### Positive

- Stable cursor under concurrent inserts — inserting a new book at `created_at = now()` does not invalidate an in-flight cursor pointing at earlier rows.
- Deterministic tie-break via `id` prevents dropped or duplicated rows when `created_at` collides.
- Partial index `books(created_at desc, id desc) where deleted_at is null` directly supports the tuple-compare filter — efficient at > 10k rows.
- Codec reusability: Features 002+ list endpoints pick up the same helper without reinventing pagination.

### Negative

- Cursor tokens are visible to anyone who base64-decodes them; they leak `created_at` timestamps and row IDs. For single-user local scale, not a concern; Phase 3 revisits with HMAC signing.
- Client must not modify cursor contents; violations produce undefined behavior. **Mitigation**: documentation, and `decodeCursor` validates the JSON shape — bad cursors 400 with `invalid-cursor`.

### Risks

- **Risk**: future feature changes the default sort, breaking old cursors.
  **Mitigation**: cursors documented as opaque; clients re-paginate from scratch on sort-change. No migration pain because cursors aren't persisted long-term.
- **Risk**: `base64url` not URL-safe in an odd edge case (trailing `=` handling).
  **Mitigation**: use `base64url` with padding-stripped output; decoder is tolerant.

## Implementation Notes

- `encodeCursor({ createdAt, id })`:
  ```typescript
  const payload = JSON.stringify({ createdAt: createdAt.toISOString(), id })
  return Buffer.from(payload, 'utf8').toString('base64url')
  ```
- `decodeCursor(cursor)`:
  ```typescript
  try {
    const raw = Buffer.from(cursor, 'base64url').toString('utf8')
    const parsed = cursorPayloadSchema.parse(JSON.parse(raw))
    return { createdAt: new Date(parsed.createdAt), id: parsed.id }
  } catch (e) {
    throw new InvalidCursorError(cursor, e)
  }
  ```
- `cursorPayloadSchema = z.object({ createdAt: z.string().datetime(), id: z.string().uuid() })`.
- Round-trip tests live in `@dialogus/shared/__tests__/cursor.test.ts`.
