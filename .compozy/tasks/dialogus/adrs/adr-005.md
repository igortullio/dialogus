# ADR-005: Mastra Dev Server as a separate process

## Status

Accepted

## Date

2026-04-24

## Context

The RAG agent needs an HTTP runtime that exposes chat/thread endpoints to `apps/web` and integrates with `@mastra/pg` memory. Three composition shapes were on the table: (A) Hono monolithic with Mastra as an embedded library; (B) Hono + Mastra Dev Server as two separate processes; (C) Hono mounting Mastra's Hono adapter at a sub-path.

The owner values Mastra Studio's built-in observability (thread history, tool-invocation traces, token accounting) for the solo-developer workflow, especially during prompt tuning for the scholarly posture (see ADR-002). Studio is first-class with Mastra Dev Server; getting equivalent visibility in an embedded setup requires extra plumbing.

## Decision

Run **Mastra Dev Server as a separate process** hosted in `apps/mastra`:

- `apps/mastra/mastra.config.ts` declares the agent (`dialogusAgent`) built via `createDialogusAgent()` from `@dialogus/rag`, and configures `@mastra/pg` memory against the same Postgres DSN shared with `@dialogus/db`.
- `apps/mastra/index.ts` is the thin entry: reads env via `@dialogus/shared/config`, boots Mastra Dev Server on port 3002, enables Studio on port 4111 (dev only).
- Dev topology: `apps/api` (3001), `apps/mastra` (3002 + 4111), `apps/worker`, `apps/web` (3000), Postgres (5432). Root `pnpm dev` uses `pnpm --parallel -r dev` (or equivalent) to orchestrate.
- Cross-process communication is HTTP: `apps/web` calls `apps/api` for catalog/library and `apps/mastra` for chat; `apps/mastra` reads `chunks`/`chapters` tables directly (read-only), and Mastra Memory writes `mastra_*` tables.

## Alternatives Considered

### Alternative 1: Hono monolithic with Mastra embedded as a library

- **Description**: One `apps/api` Hono process hosts every route; `createDialogusAgent` is imported and called from a custom handler that serves `/api/qa/*`.
- **Pros**: Single process to run and monitor; simplest ops; one CORS policy; single log stream.
- **Cons**: Loses Mastra Studio native integration — Studio expects Mastra Dev Server at a known address; replicating its UX in-app is a non-trivial dev-tool build. Also ties HTTP concerns of the agent (streaming SSE, token accounting) to the Hono middleware stack.
- **Why rejected**: Studio observability is a primary tool for prompt tuning; losing it costs far more time than the operational overhead of a second process saves.

### Alternative 2: Hono mounts Mastra's Hono adapter at `/api/qa/*`

- **Description**: Single Hono app, but Mastra's Hono adapter (if published) is mounted at a sub-path, giving Mastra-conventional routes inside the shared server.
- **Pros**: One process; Mastra routing conventions preserved; less orchestration complexity.
- **Cons**: Middleware composition is fragile — CORS, CSRF, rate-limit, and Mastra's own streaming machinery must coexist cleanly, which is undocumented territory. Studio still expects a dedicated Dev Server endpoint. Adapter maturity is unknown.
- **Why rejected**: middle-ground option with the worst of both — no Studio win, still couples Hono middleware to Mastra internals.

## Consequences

### Positive

- Mastra Studio works out of the box — visible thread graphs, tool call inspection, token usage per turn during local dev.
- Clean boundary between product HTTP (Hono) and AI HTTP (Mastra) — easier to reason about latency budgets, middleware, and failure domains.
- Mastra upgrades are contained to `apps/mastra` and `@dialogus/rag`; nothing else in the monorepo pins Mastra.

### Negative

- Four Node processes in dev (api, mastra, worker, web) plus Postgres. `pnpm dev` orchestration, port allocation, and health-visibility must be right from day one.
- Web app must hold two base URLs (`NEXT_PUBLIC_API_URL`, `NEXT_PUBLIC_MASTRA_URL`) — minor complexity in env handling.
- CORS policy has to be configured on both servers; two places for a future token-auth layer (Phase 2) to cover.

### Risks

- **Risk**: one of the four processes crashes silently during dogfooding and the owner doesn't notice until a chat fails.
  **Mitigation**: `apps/api /health` aggregates pg-boss + db ping; Mastra Studio's own visibility surfaces thread failures; Phase 2 can add a dev dashboard if needed.
- **Risk**: Mastra Dev Server behavior in prod is underspecified (it's positioned as a dev tool).
  **Mitigation**: V1 is local-first; public deploy is Phase 2 and will revisit whether to keep Mastra Dev Server or switch to an embedded adapter at that time.

## Implementation Notes

- `apps/mastra/package.json` `"type": "module"`, `"scripts": { "dev": "mastra dev", "start": "mastra start" }` (or equivalent per Mastra CLI).
- Keep `apps/mastra` thin — all agent + tool logic lives in `@dialogus/rag`, imported by `mastra.config.ts`.
- Env vars: `DATABASE_URL` shared; `ANTHROPIC_API_KEY` read via `@dialogus/shared/config`; `MASTRA_PORT=3002`.
- `apps/web` reads `NEXT_PUBLIC_MASTRA_URL` (default `http://localhost:3002`) for `useChat` base.
