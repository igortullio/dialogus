# ADR-004: Infrastructure-first layout for `apps/api`

## Status

Accepted

## Date

2026-04-24

## Context

`apps/api` uses a hexagonal / DDD layout in mature form per the m5nita reference: `src/domain/`, `src/application/`, `src/infrastructure/`. Three options exist for how much of that structure ships in Foundation, which has a single endpoint (`/health`) and no real domain logic yet:

- **Flat**: `src/index.ts` + `src/routes/health.ts`. DDD layout introduced in Feature 001 when catalog brings real entities and use cases.
- **Infrastructure-first**: `src/index.ts` + `src/infrastructure/http/routes/health.ts`. Routes live in `infrastructure/` from day 1; `domain/` and `application/` are absent until Feature 001 needs them.
- **Full hexagon ready**: all three folders (`domain/`, `application/`, `infrastructure/`) present from Foundation with `.gitkeep` or empty barrels.

The layout chosen here is what subsequent features extend, so it matters that the pattern is predictable without being performative.

## Decision

**Infrastructure-first.** Foundation's `apps/api/src/` has:

- `src/index.ts` — the boot module.
- `src/infrastructure/http/routes/health.ts` — the `/health` handler.
- No `domain/` folder.
- No `application/` folder.

When Feature 001 arrives, it introduces `domain/`, `application/`, and additional `infrastructure/` subfolders (`persistence/`, `external/`) as catalog's domain logic requires. No empty placeholder folders exist until domain logic is real.

Packages (`@dialogus/shared`, `@dialogus/db`) stay flat with module-level grouping (`config/`, `errors/`, `types/`, `schema/`) — they are utilities, not bounded contexts, and don't use DDD layering.

## Alternatives Considered

### Alternative 1: Flat `src/routes/health.ts`

- **Description**: routes live directly under `src/`; Feature 001 introduces DDD when it arrives.
- **Pros**: minimum structure for a minimum feature; no naming ceremony.
- **Cons**: Feature 001 must then move routes from `src/routes/` to `src/infrastructure/http/routes/` — a churny refactor that touches every existing route; loses the m5nita convention at the critical starting moment.
- **Why rejected**: the move is avoidable; starting with the final layout for routes costs nothing in Foundation.

### Alternative 2: Full hexagon ready (all three layers)

- **Description**: `src/domain/`, `src/application/`, `src/infrastructure/` all exist from day 1 with `.gitkeep` files or empty `index.ts` barrels.
- **Pros**: maximally predictable structure; onboarding is unambiguous ("put use cases here").
- **Cons**: empty folders feel performative; reviewers scanning the repo see scaffolding without substance and may question seriousness; `.gitkeep` files are clutter.
- **Why rejected**: the YAGNI principle applies — folders without content are noise; Feature 001 introduces them when they hold real code.

## Consequences

### Positive

- Routes live in the final location from day 1 — Feature 001 doesn't move existing files.
- No empty folders; every directory in the repo has meaningful content.
- `domain/` and `application/` appear organically when they house real domain code, signaling actual DDD rather than boilerplate.
- Package structure stays flat where DDD isn't applicable, avoiding unnecessary ceremony in utility packages.

### Negative

- The DDD convention isn't fully inaugurated until Feature 001; a reviewer scanning only Foundation might not see the DDD intent.
- Feature 001 must be disciplined about introducing `domain/` + `application/` correctly on its first file, because there's no prior example to follow.

### Risks

- **Risk**: Feature 001 introduces `domain/` + `application/` inconsistently with the m5nita convention.
  **Mitigation**: Feature 001 PRD + TechSpec explicitly reference m5nita's `apps/api/src/domain/pool/` and `apps/api/src/application/` as templates to mirror.

## Implementation Notes

- `apps/api/src/infrastructure/http/routes/health.ts` exports a `createHealthRoute(deps)` function that returns a Hono sub-app or a handler registered on the main app.
- Route registration happens in `src/index.ts`: `app.route('/health', createHealthRoute({ db }))`.
- No `src/infrastructure/http/middleware/` in Foundation — middleware appears when Feature 001 adds CORS, rate limiting, etc.
- Root-level imports (`@dialogus/shared`, `@dialogus/db`) always use package paths, never relative `../../` from `src/infrastructure/` — keeps routing independent of depth.
