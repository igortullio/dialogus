# ADR-001: Full chat-first V1 with polished library page

## Status

Accepted

## Date

2026-04-24

## Context

Feature 004 ships the user-facing surface for everything the prior features built: catalog (001), ingestion (002), and the RAG agent (003). The product PRD's Core Features list five surfaces — book catalog, ingestion, agent, spoiler boundary, and chat-first interface. The first four are non-UI or backend; the chat-first interface is the one piece a user actually clicks. Cutting depth here would directly weaken the dogfooding sustainability metric (PRD Primary Success Metrics) and the screencast portfolio artifact (PRD Goal #3).

Three V1 shapes were considered: (A) Full chat + polished library page; (B) Chat polished + library minimal; (C) Chat only, library deferred to Phase 1.5.

## Decision

**Ship the full chat-first V1 with a polished library page in Feature 004.** Concretely:

- Landing `/` — chat view: sidebar with thread list + "Nova conversa" CTA + "Gerenciar acervo" footer link. Empty state shows the "Primeiros passos" card pre-filled with three recommended titles (resolved in PRD).
- Main composer — book picker (multi-select from `ready` library) + spoiler slider (per-book chapter cap, localStorage-persisted) + message textarea with streaming send.
- Thread view — message list with token streaming, citation badges per claim (badge + hover preview + click-for-side-panel), refusal-with-hints rendering, language-matched UI affordances.
- `/library` (label "Gerenciar acervo") — grid of book cards (cover + title + authors + language flag + status badge), inline search of Gutendex via modal, retry button on `failed` rows, visible per-stage ingestion progress.
- Side panel — opens on citation click; shows full chunk excerpt + chapter context + book metadata.
- Phase 2 polish (regenerate, copy, share, evals UI) explicitly deferred.

## Alternatives Considered

### Alternative B: Lean V1 — chat polished, library minimal

- **Description**: Chat identical to A. `/library` is a flat list (table or simple cards) without grid/cover, without inline Gutendex search modal — add-book lives in a separate `/library/add` page.
- **Pros**: ~3 fewer tasks; faster path to dogfooding; allows real-usage feedback to shape library polish in Phase 1.5.
- **Cons**: portfolio screencast loses one of its three "wow" moments (the library grid is the visual proof that classics are first-class objects in dIAlogus); the gap between chat polish and library polish is glaring; the "Primeiros passos" onboarding card alone cannot carry the visual weight; Phase 1.5 work is harder to schedule than V1 work because dogfooding pressure shifts attention.
- **Why rejected**: the cost of polishing the library now is ~3 tasks; the cost of doing it later, once the owner is dogfooding daily, is higher because every library annoyance becomes a context-switch from reading to UX work. Pay the polish cost while UI mental model is fresh.

### Alternative C: Chat only, library deferred to Phase 1.5

- **Description**: Ship just chat + onboarding card; `/library` is a 404 or "em breve"; book add/manage via cURL during dogfooding.
- **Pros**: smallest V1 surface; fastest to dogfooding.
- **Cons**: directly contradicts product PRD Core Feature #5 (chat-first interface lists library as a secondary surface, not absent); no portfolio screencast can credibly show "the library grid" if there is none; cURL-driven book management is engineer-only — not a usable product.
- **Why rejected**: rompe um contrato fechado do product PRD em troca de economia de ~5 tasks. Net negative for portfolio + product coherence.

## Consequences

### Positive

- All user journeys promised by the product PRD work end-to-end after Feature 004 closes — no "almost-V1" state.
- Screencast captures four moments: search/add → ingest progress → ask + cite → spoiler-safe read. All four feel polished.
- Dogfooding starts with a tool that respects its own promises; the owner spends time reading, not patching UX.
- Phase 1.5 is freed for actual depth (Ragas evals, multi-translation, structure-aware chunking) instead of consuming itself with V1 cleanup.

### Negative

- Feature 004 has the largest task surface of the five features (~14-16 tasks).
- Library polish work competes for attention with chat polish work — both are visible.
- "Polished" is not free: time spent on grid layouts, cover-image fallbacks, and progress-bar animations is time not spent on prompt tuning or ingestion robustness.

### Risks

- **Risk**: Tailwind v4 + shadcn migrations between minor versions disrupt mid-feature.
  **Mitigation**: pin exact versions per product TechSpec § Known Risks; upgrades are deliberate.
- **Risk**: assistant-ui's primitives churn (still pre-1.x maturity at the time of PRD).
  **Mitigation**: pin exact `@assistant-ui/*` versions; treat all primitives as "wrapped" — encapsulate in dIAlogus-owned components so an upstream API break is one component's worth of edits.
- **Risk**: the "Primeiros passos" card looks like a feature ad rather than an empty state, and users dismiss it.
  **Mitigation**: copy is conversational ("comece com:"), not promotional; one-click ingestion is genuinely faster than typing the title in Gutendex search.

## Implementation Notes

- The `/library` page becomes a fully-fledged Next 16 App Router route; sub-route for `/library/add` (or modal — Feature 004 TechSpec resolves) handles Gutendex search.
- Cover-image fallbacks must be considered: not every Gutendex book has `cover_url`; use a generated SVG placeholder with the title in monospace as a fallback (PRD-level is silent on this; TechSpec resolves).
- Onboarding card titles per PRD Open Questions resolution (in PRD body): The Count of Monte Cristo (EN), Memórias Póstumas de Brás Cubas (PT), Crime and Punishment (EN). UUIDs for Gutendex IDs frozen in a shared constant.
- Phase 2 (Phase 1.5 effectively) explicitly absorbs: thread regeneration, copy-message-to-clipboard, share-thread-link, settings pane (model toggle, temperature), Ragas-style evals UI.
