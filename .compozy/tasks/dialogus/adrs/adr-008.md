# ADR-008: Fresh start from dialogus-2

## Status

Accepted

## Date

2026-04-24

## Context

`/Users/igortullio/Developer/igortullio/dialogus-2/` contains real code from the previous attempt: 6 packages (`catalog`, `config`, `core`, `db`, `ingestion`, `rag`), 3 apps (`api`, `web`, `worker`), and 8 test files in `packages/ingestion/__tests__/` covering chapter-cleaner, parser-epub, run-stage, enqueue-service, index-writer (integration), chapter-splitter, status-service, and a smoke test. There are also committed fixtures (`__fixtures__/minimal.epub`) and verification scripts (`verify-monte-cristo.ts`).

Three options were considered for the new project: (A) **Fresh start** — nothing copied, old project is reference-only; (B) **Port testable modules** — copy parsers, chunker, cleaner, fixtures along with their tests, rewrite orchestration; (C) **Port everything and refactor** — copy the whole `ingestion` package structure and adapt incrementally.

Key context: the owner described the previous project as having had "problems" and lost code; architecture decisions also shifted (Mastra Dev Server topology, Mastra Memory ownership, spoiler boundary feature, Postgres 18, TypeScript 6, simpler package structure with 5 instead of 6 packages).

## Decision

**Fresh start.** No code is copied from `dialogus-2` to the new project. `dialogus-2/` remains as read-only reference material for:

- Scope boundaries (what was in versus out of V1).
- Algorithm defaults (chunk size 600, overlap 100, paragraph-aligned — inherited as values, not code).
- Fixture content (if a new EPUB fixture is needed, choose a different book; don't copy the old `minimal.epub`).
- Lessons learned (pre-commit + Testcontainers was too heavy → resolved in ADR-007; spec drift → resolved in ADR-003).

## Alternatives Considered

### Alternative 1: Port tested modules (parsers, chunker, cleaner, fixtures)

- **Description**: Copy `chapter-cleaner.ts`, `parser-epub.ts`, `chapter-splitter.ts`, `chunker.ts` with their test files and the `__fixtures__/minimal.epub`; reimplement pg-boss orchestration and status service from scratch.
- **Pros**: Saves ~2 weeks of parser/chunker work; inherits a validated test surface for domain-pure code.
- **Cons**: Hidden assumptions bleed in — modules were written against the previous schema (`chunks` shape, chapter ordinal conventions), the previous `EmbeddingProvider` shape, and previous Zod schemas. Adapting them is subtle work that can revive bugs which happened to be latent in `dialogus-2`.
- **Why rejected**: the owner explicitly chose fresh start to prevent inheriting latent bugs. The algorithm knowledge is transferable; the specific code is not worth the integration risk.

### Alternative 2: Port everything and refactor incrementally

- **Description**: Copy the entire `packages/ingestion/` structure, wire it up, then refactor to match the new architecture.
- **Pros**: Maximum code reuse.
- **Cons**: High risk of ressurecting the original problems; hard to distinguish "code that was validated" from "code that was copied and never retested". Couples architectural choices to past decisions that are already being changed (schema, Memory ownership, process topology).
- **Why rejected**: defeats the purpose of restarting.

## Consequences

### Positive

- New code is written against the new architecture from day one — no impedance mismatch with Mastra Dev Server, Mastra Memory, or the simpler 5-package layout.
- Every line of code has a known author and context; nothing is "inherited" with unclear provenance.
- Clean test coverage baseline — every file has a test written with the current tooling (Vitest 4, Testcontainers, MSW 2).
- Owner's mental model stays current; there's no lingering "why is this here?" code that traces back to a deleted project.

### Negative

- Re-implementing the EPUB parser, TXT cleaner, chapter splitter, and chunker costs real engineering time (estimated 1-2 weeks inside Feature 002).
- Writing new fixtures — test EPUB files, reference PT books for heuristic validation — is work that `dialogus-2` already did.
- Some algorithm insights (specific regex patterns for Gutenberg boilerplate, chapter-header variants) are only discoverable by re-reading `dialogus-2` code; the owner or a future implementer may not do that exercise thoroughly and miss known edge cases.

### Risks

- **Risk**: re-implementation misses edge cases that the previous version handled, wasting time re-discovering them.
  **Mitigation**: Feature 002 TechSpec explicitly names `dialogus-2/packages/ingestion/src/` as a required reading step before chunker/parser design; extract learned edge cases (but not code) into the Feature 002 spec.
- **Risk**: owner sees the old `__fixtures__/minimal.epub` as a tempting shortcut and copies it.
  **Mitigation**: this ADR documents the explicit ban; choose a different public-domain EPUB for the new fixture (e.g., Edgar Allan Poe short stories, Machado's *Memórias Póstumas de Brás Cubas* short form).

## Implementation Notes

- The new `dialogus/` directory is already partially scaffolded (has `apps/web/.next` from an earlier exploration). Step 1 of the TechSpec Build Order will clean this and start from a bare `pnpm init`.
- Feature 002 TechSpec must include a section "Lessons inherited from dialogus-2" that captures:
  - Chapter-detection regex patterns that were validated.
  - Gutenberg boilerplate markers (START/END/copyright blocks) that were observed.
  - EPUB library choice that worked (and any that didn't).
  - Any subtle edge cases from the `verify-monte-cristo.ts` script.
- Do not link/import/symlink anything from `dialogus-2/` into `dialogus/`.
