# ADR-007: Inline citation markers use `{{cite:<chunk_id>}}` double-brace syntax

## Status

Accepted

## Date

2026-04-24

## Context

The PRD commits the agent to emitting inline citations for every non-trivial claim (PRD Goal 1 + ADR-003). Mastra Memory stores the structured citation data as tool_output; the agent separately weaves a citation marker into the prose. The Chat UI (Feature 004) parses the prose, matches markers to tool_output entries, and renders a hoverable badge per citation.

Four marker shapes were evaluated against three criteria: parse robustness (does the pattern survive LLM output variations?), collision safety (can it appear verbatim in a book's text and cause false positives?), and LLM compliance (does the model reliably emit the format?).

| Option | Example | Parse | Collision | LLM |
|---|---|---|---|---|
| XML-like | `<cite id="abc-123"/>` | easy regex | very rare | good |
| Footnote | `claim[^1]` + `[^1]: ...` | two-pass | common in books | OK |
| Bracket | `[MD:10]` | regex | possible in footnotes | OK |
| Double-brace | `{{cite:abc-123}}` | easy regex | very rare | good |

## Decision

**The agent emits inline citations as `{{cite:<chunk_id>}}`** — double-brace, literal prefix `cite:`, followed by the UUID of the chunk that supports the adjacent claim.

- `chunk_id` is the primary key from the `chunks` table; the UI resolves full excerpt + chapter metadata server-side via `GET /api/library/chunks/:id` (already defined in Feature 002).
- The citation appears immediately after the claim it supports, inside the sentence: *"Ishmael conhece Queequeg no capítulo 10 {{cite:abc-123}}."*
- Multiple citations per claim stack: `"...{{cite:abc-123}} {{cite:def-456}}."`
- Mastra Memory's tool_output from `semantic_search` holds the structured metadata `{ chunk_id, book_id, chapter_id, chapter_ordinal, chapter_title, excerpt_preview, score }` for every chunk returned. The UI cross-references marker IDs against tool_output for badge rendering; the HTTP endpoint is the authoritative excerpt source.

## Alternatives Considered

### Alternative 1: XML-like `<cite id="abc-123"/>`

- **Description**: self-closing XML-style tag with `id` attribute.
- **Pros**: parse regex is trivial (`/<cite id="([^"]+)"\/>/g`); unambiguous; LLMs emit it well.
- **Cons**: looks like HTML/XML and the user's prose sits inside a Markdown renderer; some Markdown parsers attempt to sanitize unknown tags and may strip them; LLMs sometimes drop the self-closing slash producing `<cite id="...">` which is malformed.
- **Why rejected**: sanitization risk + malformed-tag risk. Double-brace is Markdown-safe by default.

### Alternative 2: Markdown footnote `[^1]` + `[^1]: ...`

- **Description**: Markdown-native footnote syntax. Each citation gets a number; a table at the end maps numbers to chunk IDs.
- **Pros**: Markdown-native; renderers support out-of-the-box; numbered citations look scholarly.
- **Cons**: two-pass parsing (extract numbers, extract map, join); LLM must maintain consistent numbering across a long response, which is a known failure mode (`[^1]` then `[^1]` again for a different chunk, or `[^2]` before `[^1]` exists); numbering-hostile when the UI wants to resolve a badge independently of position; collision with book text that legitimately uses `[^1]` (rare but non-zero in academic PG books).
- **Why rejected**: LLM numbering discipline is unreliable; parsing is more complex than necessary; collision possible.

### Alternative 3: Bracket style `[MD:10]`

- **Description**: short bracket tokens with book abbreviation and chapter ordinal.
- **Pros**: compact; legible without a renderer.
- **Cons**: requires maintaining a stable book-abbreviation map (Moby Dick → MD, Dom Casmurro → DC?); loses chunk granularity (chapter 10 ≠ specific chunk); `[MD:10]` collides with editorial footnotes in PG texts; bilingual titles produce abbreviation ambiguity.
- **Why rejected**: loses chunk-level resolution, which is the entire point of citations grounded in embeddings.

### Alternative 4: Structured JSON blocks

- **Description**: agent emits a full JSON object inline: `{"cite":"abc-123"}`.
- **Pros**: parseable as JSON.
- **Cons**: LLMs emit JSON reliably in structured-output mode but unreliably inside free prose; valid JSON inside prose is awkward and hard to visually ignore when reading the raw text; collision with book text is near zero but parsing is more expensive than regex.
- **Why rejected**: overkill.

## Consequences

### Positive

- Parsing is a single regex: `/\{\{cite:([0-9a-f-]{36})\}\}/g`.
- Collision with real book text is effectively zero — double-brace `{{` does not appear in any public-domain classic we're likely to ingest.
- Markdown renderers pass the marker through verbatim; neither escaped nor stripped.
- LLMs handle template-like patterns (`{{var}}`) reliably — this is the same pattern used in thousands of template engines the model has seen.
- Chunk ID (UUID v4) is the primary key; no abbreviation map, no numbering discipline.
- The format survives copy-paste: a user pasting an agent response into a note-taking app keeps the citation markers intact for later resolution.

### Negative

- The raw marker is ugly — `{{cite:abc-123}}` in an un-rendered view is distracting. Only relevant in the degenerate case of reading the agent's raw output outside the Chat UI (debugging, logs).
- UUID inside prose inflates message tokens (~36 chars per citation). On a response with 5 citations, ~180 extra characters. Acceptable given the response-length budget.
- The UI must strip the markers before rendering the prose (substituting inline badges in their place). Feature 004 task.

### Risks

- **Risk**: LLM produces truncated or malformed UUIDs inside the marker (`{{cite:abc-12}}`).
  **Mitigation**: UI parser validates UUID regex; malformed markers are surfaced as a warning icon in place of a badge, logged for system-prompt tuning. The dogfooding validation gate (≥10 questions) surfaces systematic formatting errors early.
- **Risk**: LLM emits markers for chunk IDs that weren't in tool_output (hallucinated IDs).
  **Mitigation**: UI parser cross-references marker chunk_id against tool_output. Unknown IDs are flagged. The `GET /api/library/chunks/:id` endpoint returns 404 for non-existent IDs — the user sees a "citação inválida" badge and the dogfooding log records it.
- **Risk**: the system prompt instruction for citation format gets diluted in a long conversation.
  **Mitigation**: prompt caching keeps the system prompt authoritative every turn; the instruction is a single clear sentence.

## Implementation Notes

- System prompt section (in `@dialogus/rag/src/prompts/system.md`):
  > For every non-trivial claim, append a citation marker of the form `{{cite:<chunk_id>}}` where `<chunk_id>` is the `chunk_id` from the `semantic_search` tool output that supports the claim. Do not omit the double braces. Do not add spaces inside the braces. Multiple citations may follow the same claim.
- Regex for UI parser (Feature 004, documented here for contract): `/\{\{cite:([0-9a-f-]{36})\}\}/g`.
- Tool output from `semantic_search` is the authoritative source of chunk IDs; UI should never accept a citation for a chunk_id that does not appear in the current thread's tool_outputs for this message.
- Extension path: if Phase 2 wants quote-level citations (character offsets within a chunk), the marker grows to `{{cite:<chunk_id>:<start>:<end>}}` — backward compatible because the regex can branch on optional colons.
