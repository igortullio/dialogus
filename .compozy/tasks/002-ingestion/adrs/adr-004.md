# ADR-004: Streaming discipline from day 1

## Status

Accepted

## Date

2026-04-24

## Context

Public-domain classic books vary wildly in size: Moby Dick is ~200k words, War and Peace is ~500k, Les Misérables is ~530k, Don Quixote full is ~380k. A naive ingestion implementation that "load the whole book into memory, process, write" runs into memory ceilings on the larger ones and produces unstable behavior on `apps/worker`. Three strategies were considered:

- **Streaming always** — no stage holds the entire book in memory; chapters / chunks / embeddings are written to DB incrementally.
- **Accept with warning** — estimate tokens upfront; warn for large books; attempt anyway until OOM.
- **Reject > 300k words** — fail with clear error; defer large books to a future Phase 2 "streaming ingestion" feature.

The original `dialogus-2` project acknowledged this as a safety-valve concern ("if War and Peace doesn't fit, create feature 002.5"). Baking streaming into V1 avoids that branch entirely.

## Decision

**Streaming is the default discipline** across all stages of Feature 002:

- **Download stage**: raw file streamed to disk via `fetch().body.pipe()` or equivalent; no buffering the whole response in memory.
- **Clean stage**: processes the raw text via a streaming line iterator; writes cleaned output to `./storage/clean/<gutendex_id>.txt` as it progresses.
- **Parse stage**: detects chapters on-the-fly (EPUB spine iteration or TXT scan); writes each chapter row to DB as it's discovered. Never accumulates "all chapters" in memory.
- **Chunk stage**: iterates chapters one-by-one from DB; within each chapter, splits into chunks and INSERTs each chunk (without embedding) as produced. Releases chapter text from memory after the chapter is chunked.
- **Embed stage**: queries chunks with `embedding IS NULL` in batches of 100; sends each batch to OpenAI; writes embeddings back to DB in a single UPDATE per batch; releases batch memory before loading the next. Never materializes "all chunks" in memory.
- **Index stage**: runs `VACUUM ANALYZE` on the `chunks` table scoped to the book if possible; HNSW index maintenance is Postgres-internal.

Peak per-book memory during ingestion target: ≤ 150 MB regardless of book size.

## Alternatives Considered

### Alternative 1: Accept with warning

- **Description**: estimate upfront, log warning, attempt in-memory; fail with OOM if exceeded.
- **Pros**: simpler initial implementation.
- **Cons**: silent failures for the owner (worker restart mid-ingestion, no clear error); eventual re-implementation required when the owner adds a large PT classic (Os Lusíadas, multi-volume Machado collections) — at which point streaming becomes a migration instead of a design.
- **Why rejected**: defers an inevitable refactor; the "is this book too big?" question becomes a permanent friction point.

### Alternative 2: Reject > 300k words

- **Description**: API returns 413 Payload Too Large with slug `book-too-large` when size estimate exceeds threshold.
- **Pros**: smallest initial scope.
- **Cons**: excludes major classics (Don Quixote full, War and Peace, Les Misérables, Don Casmurro PT-edition with extensive notes). Violates the product promise — a reader's study companion that refuses the very books that most benefit from RAG is a weak product.
- **Why rejected**: too narrow a V1 scope.

## Consequences

### Positive

- Any Gutenberg book fits the pipeline — no "is this too big?" gate.
- `apps/worker` memory footprint is bounded and predictable.
- Phase 2's "streaming ingestion" feature disappears — already done.
- Incremental writes mean an OOM-kill mid-ingestion leaves partial but resumable state (consistent with ADR-003).

### Negative

- Every stage handler is ~20 % more code than a naive "load + process + write" implementation (iterators + batch management + explicit memory release).
- Testing streaming behavior requires a large fixture book; small fixtures don't exercise the pressure.
- Some operations that are simpler in-memory (e.g., computing per-book statistics during chunking) become more verbose.

### Risks

- **Risk**: a developer forgets the discipline and writes `const allChunks = await getAllChunks(bookId)` in embed stage.
  **Mitigation**: PR review checklist explicitly calls out streaming audit; large-book integration test in CI catches regressions by running Les Misérables or similar through the pipeline and asserting peak memory.
- **Risk**: streaming libraries (`stream.pipeline`, async iterators) have subtle error-propagation gotchas.
  **Mitigation**: every stream-handling code path has a dedicated unit test exercising an error injected mid-stream.

## Implementation Notes

- Node's `stream.pipeline` + `async/await` + `for await (const row of queryIterator)` is the standard pattern.
- Drizzle supports streaming queries via `postgres.js` cursor mode for SELECT; use it for iterating chapters/chunks.
- Batch INSERT is preferred over row-by-row for chunk / embedding writes (100 rows per batch); still compatible with streaming if the batch buffer is bounded.
- `large-book.integration.test.ts` uses a 400k-token fixture (or generates one synthetically) to enforce the discipline automatically.
- Peak memory measurement: integration test runs with `--max-old-space-size=200`; if the pipeline fits, the test passes.
