# ADR-010: "Adicionar do Gutendex" flow uses a left-side drawer (shadcn `<Sheet>`)

## Status

Accepted

## Date

2026-04-24

## Context

The user adds books from Gutendex via two entry points: a button on `/library` and a link inside the chat composer's book picker dropdown. Both entry points open the same flow: search Gutendex by query + language → see results → click "Adicionar" on a result → book queues for ingestion. UX shape options: modal dialog, dedicated route, intercepting routes, side drawer.

Feature 004 ADR-003 already reserves the **right-side** sheet for citation panels in the chat view. Re-using the right side for "Adicionar do Gutendex" risks confusion (same affordance, different purpose).

## Decision

**The Gutendex add flow opens as a `<Sheet>` from the LEFT side of the screen.**

- Trigger 1: "Adicionar do Gutendex" button on `/library` opens the drawer. Behind it, the library grid stays visible (slightly faded but readable).
- Trigger 2: "Adicionar do Gutendex" link inside the chat composer's book picker dropdown opens the same drawer.
- Drawer content: search input (auto-focused) + language filter chips (EN, PT, both) + paginated results (cover, title, authors) + per-result "Adicionar" button. On click, the API queues the book; the result row updates to show "Adicionado — ingestindo..." with a spinner; the drawer stays open so the user can add more.
- Close: button at top-right of the drawer, click-outside, or Esc.
- Width: 480 px on desktop; full-screen overlay on viewports < 768 px wide (mobile fallback).

The chat view's citation side panel (ADR-003) uses the right side. The library add drawer uses the left side. Visual differentiation is structural, not just stylistic — the same affordance shape (sheet) appears on opposite sides for opposite purposes.

## Alternatives Considered

### Alternative 1: Modal dialog (shadcn `<Dialog>`)

- **Description**: Centered modal with backdrop dim. Same content as the drawer.
- **Pros**: classic centered modal; clear "this is overlay UI"; symmetric with confirm dialogs (rename, delete) used elsewhere.
- **Cons**: occupies center screen — full library grid is dimmed and hard to glance at while searching ("did I already add this?"); modal width competes with content width (search results need vertical scrolling); pattern feels modal-heavy for a workflow that may take 30+ seconds (browsing search results).
- **Why rejected**: workflow length and "browse-while-searching" use case favor a side panel that preserves library context.

### Alternative 2: Dedicated route `/library/add`

- **Description**: Click navigates to a separate page; back button returns.
- **Pros**: deep-linkable; URL reflects state.
- **Cons**: full page navigation loses the library context (can't see "is this book already in my library?"); back-button behavior on chat composer trigger is confusing (returns to chat or library?); slower than a drawer for "add and continue browsing."
- **Why rejected**: workflow context matters more than URL shareability for a single-user local-first app.

### Alternative 3: Parallel route + intercepting routes (Next 16)

- **Description**: Click opens drawer (modal-like) but URL changes to `/library/add`; refresh shows the dedicated page version; deep-link works.
- **Pros**: best of both — modal UX + URL updates.
- **Cons**: advanced Next 16 pattern; more code; more bugs; overkill for a single-user app where deep-linking the add flow has near-zero practical value.
- **Why rejected**: complexity outweighs benefit for this specific workflow.

### Alternative 4: Right-side sheet (same side as citation panel)

- **Description**: Sheet from right; same shape as citation panel.
- **Pros**: visual consistency with one sheet pattern.
- **Cons**: same shape, different purpose creates confusion — clicking a citation in the chat opens a right sheet; clicking "Adicionar do Gutendex" in the chat composer opens a right sheet. Two right-side sheets for unrelated purposes.
- **Why rejected**: visual ambiguity is a UX cost.

## Consequences

### Positive

- Library context stays visible behind the drawer — the user can verify "is this book already in my library?" by glancing past the drawer.
- Chat composer's "Adicionar do Gutendex" link reuses the same component and side; the user learns one drawer for one purpose.
- Right side stays exclusively for citation panels — clean cognitive split.
- Closing the drawer is the same gesture as closing a citation panel (Esc / click-outside / close button) — consistency where it matters (interaction), differentiation where it matters (location + content).
- Mobile fallback is straightforward: full-screen overlay covers everything.

### Negative

- Two sides of the screen now host sheets — keyboard users may mix them up. Mitigation: `aria-label` on each sheet describes the purpose distinctly.
- Left-side sheet on desktop slightly off-center vs. the right-side citation panel; designers may want symmetry. Acceptable trade-off: function over symmetry.
- The library grid is partially obscured (left ~480 px); the user can scroll the grid vertically while the drawer is open.

### Risks

- **Risk**: a screen reader user opens the drawer from the chat composer, then hits Tab — focus may jump to the chat behind, disorienting them.
  **Mitigation**: shadcn `<Sheet>` traps focus by default; a11y-friendly out of the box.
- **Risk**: the drawer being on the left feels backwards if the user's visual habit is right-side panels.
  **Mitigation**: the cognitive cue "left = library, right = citations" is consistent within dIAlogus; the user adapts within minutes.

## Implementation Notes

- Component: `apps/web/src/components/library/AddGutendexSheet.tsx` (Client Component).
- Uses shadcn `<Sheet side="left" />` with width `w-[480px]` on `md:` and up.
- Triggers: `<SheetTrigger>` on the library page button; programmatic open from the composer dropdown via shared state (URL hash `#add-gutendex` or a context provider; resolved during implementation).
- Search input: debounced `useQuery({ queryKey: ['gutendex-search', q, lang], ... })` with `staleTime: 60_000` matching the API's LRU cache TTL.
- Pagination: cursor-based per Feature 001 catalog endpoints; "Carregar mais" button at the bottom.
- Result row mutation: `useMutation` calling `POST /api/library/books`; on success, optimistic update inserts the new book into `['library']` cache.
- Cover-image fallback (PRD Open Question): generated SVG with the title rendered in monospace over a hash-based color block.
